## Topic
API 설계에서 DTO, Entity, 계층 분리 기준

### 1. Why
- **실무**: API 응답에 `Entity`를 그대로 노출하면 필드가 과하게 열리거나, 지연 로딩과 양방향 연관관계 때문에 예기치 않은 쿼리와 직렬화 문제가 생긴다. DTO 분리와 계층 책임 분리는 이런 리스크를 줄이는 기본 장치다.
- **구조적 의미**: `Controller`, `Service`, `Repository`를 나누는 목적은 코드 수를 늘리기 위해서가 아니라 HTTP 처리, 비즈니스 로직, 영속성 로직의 변경 이유를 분리하기 위해서다.
- **면접 의도**: 면접에서는 DTO와 Entity를 왜 나누는지, 각 계층이 무엇을 책임져야 하는지, 계층 간 데이터 전달을 어떤 기준으로 설계하는지 확인한다.

### 2. Core Concept
- **개념 정의**: DTO는 요청과 응답을 전달하기 위한 객체이고, Entity는 DB 테이블과 영속성 컨텍스트에 연결된 객체다. `Controller`는 입출력 경계를 다루고, `Service`는 유스케이스와 트랜잭션을 처리하며, `Repository`는 DB 접근을 담당한다.
- **동작 방식**: 클라이언트 요청은 `Request DTO`로 들어오고 `Controller`가 검증 후 `Service`로 넘긴다. `Service`는 DTO를 기반으로 도메인 로직을 수행하고 `Repository`를 통해 `Entity`를 조회 또는 저장한 뒤, 최종 결과를 `Response DTO`로 변환해 반환한다.
- **장점 / 단점**: DTO와 Entity를 분리하면 API 계약이 안정적이고 보안상 안전하며 계층별 테스트가 쉬워진다. 반면 매핑 코드가 추가되고, 작은 프로젝트에서는 다소 번거롭게 느껴질 수 있다.
- **비교 포인트**: `Entity` 직접 반환은 빠르게 개발할 수 있지만 API와 DB 모델이 강하게 결합된다. DTO 분리는 초기 비용이 있지만 스펙 변경, 필드 숨김, 응답 최적화, 버전 관리에 훨씬 유리하다.
- **사용 조건 / 주의점**: 외부 API 경계에서는 DTO 사용을 기본값으로 두는 것이 안전하다. `Controller`에는 비즈니스 로직을 넣지 말고, `Repository`가 DTO 검증까지 책임지지 않도록 계층 역할을 분명히 해야 한다.

### 3. Interview Answer Version
- 30~40초 답변: DTO와 Entity는 역할이 다르기 때문에 분리하는 게 맞다고 생각합니다. Entity는 영속성 모델이라 DB 구조와 연관관계, 변경 감지를 포함하지만, DTO는 API 요청과 응답 계약에 맞춘 전달 객체입니다. 그래서 `Controller`는 DTO를 받아 검증하고, `Service`는 비즈니스 로직과 트랜잭션을 처리하고, `Repository`는 영속성 처리를 맡는 구조가 가장 안정적입니다. 이렇게 해야 API 스펙 변경이 DB 모델에 직접 전파되지 않고, 직렬화 문제나 보안 노출도 줄일 수 있습니다.

### 4. Practical Tip
- 실제 적용 사례: 회원 가입 API에서는 `SignupRequest`를 받아 검증한 뒤 `Service`에서 `Member` Entity를 만들고 저장하고, 응답은 `SignupResponse`로 필요한 정보만 돌려주는 방식이 가장 안전하다.
- 잘못 쓰면 생기는 문제: `Entity`를 그대로 반환하면 비밀번호 같은 민감 정보 노출, `LazyInitializationException`, 양방향 참조 무한 직렬화, 스펙 변경 시 연쇄 수정이 발생할 수 있다.
- 프로젝트 연결: Spring MVC + JPA 프로젝트에서는 요청 DTO에 validation 어노테이션을 두고, `Service`에서 Entity를 다룬 뒤 응답 DTO로 변환하는 흐름이 가장 일반적이고 유지보수에 유리하다.

### 5. 예상 꼬리 질문
1. Service 계층은 DTO를 반환해야 하나요, 도메인 객체를 반환해야 하나요?
- 답변: 정답이 하나로 고정되지는 않지만, 외부 API 경계와 느슨하게 연결하려면 컨트롤러 근처에서 DTO로 변환하는 쪽이 보통 더 안전합니다. 다만 내부 유스케이스 조합이 많으면 Service는 도메인 객체나 별도 결과 객체를 반환하고, 최종 응답 DTO는 컨트롤러에서 만드는 방식도 충분히 합리적입니다.
2. DTO 매핑을 수동으로 할지 MapStruct 같은 도구를 쓸지 기준은 무엇인가요?
- 답변: 매핑 수가 적고 로직이 단순하면 수동 매핑이 가장 읽기 쉽습니다. 반대로 DTO 종류가 많고 반복 매핑이 계속 늘어나면 MapStruct 같은 도구를 써서 보일러플레이트를 줄이는 편이 유지보수에 유리합니다.
3. 단순 CRUD 프로젝트에서도 DTO 분리를 꼭 해야 하나요?
- 답변: 학습용이나 아주 작은 내부 도구라면 완화할 수 있지만, 외부 API 계약이 존재한다면 DTO 분리를 기본값으로 두는 것이 좋습니다. 나중에 필드 노출 통제나 응답 구조 변경이 생길 때 비용 차이가 크게 납니다.
