## Topic
Filter, Interceptor, AOP를 언제 어떻게 구분해서 쓰는가

### 1. Why
- **실무**: 인증, 로깅, 공통 응답 처리처럼 여러 요청에 반복되는 로직은 많지만, 어떤 시점에서 적용해야 하는지에 따라 `Filter`, `Interceptor`, `AOP` 중 적절한 선택이 달라진다.
- **구조적 의미**: 세 기술은 모두 공통 관심사를 분리하는 수단이지만 적용 레벨이 다르다. 요청 레벨, Spring MVC 레벨, 메서드 실행 레벨을 구분해야 설계가 깔끔해진다.
- **면접 의도**: 면접에서는 "세 개를 단순히 외운 것인지", 아니면 각각의 역할과 적용 시점, 인증/로깅/공통처리 사례를 기준으로 구분해서 사용할 수 있는지 본다.

### 2. Core Concept
- **개념 정의**: `Filter`는 Servlet 스펙 기반으로 `DispatcherServlet` 앞단에서 동작한다. `Interceptor`는 Spring MVC에서 Handler 실행 전후를 가로챈다. `AOP`는 Spring Bean 메서드 호출을 프록시로 감싸 공통 로직을 적용한다.
- **동작 방식**: 요청은 보통 `Filter -> DispatcherServlet -> Interceptor -> Controller -> Service` 순서로 흐르고, `AOP`는 이 과정 중 Bean 메서드 실행 지점에서 동작한다. 그래서 적용 시점과 접근 가능한 정보가 서로 다르다.
- **장점 / 단점**: `Filter`는 모든 HTTP 요청에 일괄 적용하기 좋지만 컨트롤러 정보는 알기 어렵다. `Interceptor`는 컨트롤러 정보와 요청 핸들러를 알 수 있어 웹 계층 제어에 좋지만 MVC 외 영역에는 적용되지 않는다. `AOP`는 서비스 메서드 단위로 재사용성이 높지만 HTTP 요청/응답 자체를 직접 다루는 용도로는 부적절하다.
- **비교 포인트**: 인증 토큰 파싱이나 CORS, 인코딩 같은 저수준 요청 처리는 `Filter`가 적합하다. 로그인 여부 검사, 요청 시간 측정, 관리자 페이지 접근 기록처럼 MVC 요청 문맥이 필요한 경우는 `Interceptor`가 적합하다. 트랜잭션, 메서드 실행 시간, 공통 로깅처럼 비즈니스 메서드 공통 처리는 `AOP`가 적합하다.
- **사용 조건 / 주의점**: 인증/인가 전체를 직접 구현하기보다 Spring Security의 `Security Filter Chain`을 우선 활용하는 것이 좋다. `AOP`에 HTTP 응답 수정 로직을 넣거나, `Filter`에 비즈니스 로직을 넣으면 책임이 꼬인다. 적용 순서와 예외 전파도 함께 설계해야 한다.

### 3. Interview Answer Version
- 30~40초 답변: 세 기술은 모두 공통 관심사를 분리하지만 적용 레벨이 다릅니다. `Filter`는 `DispatcherServlet` 앞단에서 모든 HTTP 요청을 처리할 수 있어서 인코딩, CORS, JWT 파싱 같은 요청 전처리에 적합합니다. `Interceptor`는 Spring MVC 핸들러 전후에서 동작하므로 로그인 체크나 요청 추적처럼 컨트롤러 문맥이 필요한 경우에 좋습니다. `AOP`는 Bean 메서드 실행을 감싸는 방식이라 트랜잭션이나 실행 시간 로깅처럼 서비스 계층 공통 처리에 적합합니다. 핵심은 기술 이름이 아니라 어느 시점의 관심사인지 먼저 판단하는 것입니다.

### 4. Practical Tip
- 실제 적용 사례: JWT 토큰 추출과 검증은 `Filter`, 관리자 페이지 접근 로깅은 `Interceptor`, 서비스 메서드 성능 측정이나 `@Transactional` 같은 횡단 관심사는 `AOP`로 분리하면 책임이 자연스럽다.
- 잘못 쓰면 생기는 문제: `Filter`에서 도메인 로직까지 처리하거나 `AOP`로 HTTP 예외 응답을 직접 조작하면 디버깅이 어려워지고, 요청 흐름이 프레임워크 레벨과 비즈니스 레벨에서 뒤섞인다.
- 프로젝트 연결: Spring Boot 프로젝트에서는 `OncePerRequestFilter`, `HandlerInterceptor`, `@Aspect`가 각각 다른 위치의 공통 처리 수단이며, 인증은 보통 Spring Security 필터 체인 안에서 다루는 것이 정석이다.

### 5. 예상 꼬리 질문
1. JWT 인증은 왜 `Interceptor`보다 `Filter` 쪽이 더 자주 쓰이나요?
- 답변: JWT 검증은 컨트롤러 이전, 즉 가능한 가장 앞단에서 모든 요청에 일관되게 적용하는 것이 중요하기 때문입니다. 또 Spring Security도 필터 체인 기반이라 JWT 인증을 그 흐름 안에 넣는 것이 구조적으로 자연스럽습니다.
2. `@Transactional`이 AOP 기반이라는 말은 정확히 무슨 뜻인가요?
- 답변: Spring이 대상 메서드를 직접 바꾸는 것이 아니라 프록시 객체로 감싸서 메서드 호출 전후에 트랜잭션 시작, 커밋, 롤백 로직을 끼워 넣는다는 뜻입니다. 그래서 자기 자신 메서드 호출에는 프록시가 끼지 못해 `@Transactional`이 기대대로 동작하지 않을 수 있습니다.
3. 공통 예외 처리는 `Filter`, `Interceptor`, `AOP` 중 어디서 처리하는 것이 좋나요?
- 답변: 기본적인 API 예외 응답은 보통 `ControllerAdvice` 같은 전역 예외 처리기로 두는 것이 가장 명확합니다. 다만 보안 필터에서 발생한 인증 예외처럼 MVC에 오기 전 실패한 경우는 필터나 Security 예외 처리 흐름에서 따로 다뤄야 합니다.
