## Topic
HTTP와 REST API 설계 원칙

### 1. Why
- **실무**: API는 프론트엔드, 모바일, 외부 시스템과의 계약이다. HTTP 메서드, 상태 코드, URI 설계, 요청/응답 구조를 일관되게 잡아야 협업 비용과 장애 대응 비용이 줄어든다.
- **구조적 의미**: REST API 설계는 단순 URL 작명이 아니라 자원을 어떤 방식으로 표현하고, 상태 변화와 오류를 어떤 HTTP 의미로 전달할지 정하는 계약 설계다.
- **면접 의도**: 면접에서는 HTTP 메서드 의미, 상태 코드 사용 기준, URI 설계 방식, 멱등성, RESTful 원칙을 실제 설계 판단으로 설명할 수 있는지 확인한다.

### 2. Core Concept
- **개념 정의**: HTTP는 웹에서 데이터를 주고받는 프로토콜이고, REST는 자원을 URI로 표현하고 HTTP 메서드와 상태 코드로 행위를 전달하는 설계 스타일이다. 요청/응답 구조는 클라이언트와 서버가 데이터를 주고받는 계약 형식이다.
- **동작 방식**: 보통 조회는 `GET`, 생성은 `POST`, 전체 수정은 `PUT`, 부분 수정은 `PATCH`, 삭제는 `DELETE`를 사용한다. 성공과 실패는 적절한 상태 코드로 표현하고, URI는 동사가 아니라 자원 중심 명사로 설계한다. 응답은 데이터와 메타데이터, 에러 정보 구조를 일관되게 유지하는 것이 중요하다.
- **장점 / 단점**: HTTP 의미에 맞춘 RESTful 설계는 API 예측 가능성과 협업 효율을 높인다. 반면 모든 상황을 REST 원칙에 기계적으로 맞추려 하면 복잡한 비즈니스 액션 표현이 어려워질 수 있다.
- **비교 포인트**: `PUT`은 멱등성을 가진 전체 수정에 가깝고, `PATCH`는 부분 수정에 적합하다. `GET`과 `DELETE`는 멱등성이 있지만 `POST`는 일반적으로 멱등하지 않다. URI에서는 `/users/{id}`처럼 자원을 표현하고, 검색이나 정렬 조건은 보통 query parameter로 분리한다.
- **사용 조건 / 주의점**: 상태 코드를 모두 `200`으로 통일하면 클라이언트가 실패 원인을 해석하기 어렵다. URI에 동사를 남발하거나 요청/응답 포맷이 엔드포인트마다 제각각이면 API 일관성이 무너진다. 멱등성은 재시도와 중복 요청 방지 전략과도 연결되므로 의미를 정확히 지켜야 한다.

### 3. Interview Answer Version
- 30~40초 답변: HTTP와 REST API 설계의 핵심은 자원, 행위, 상태를 일관되게 표현하는 것이라고 생각합니다. URI는 `/orders/{id}`처럼 자원 중심으로 설계하고, 조회는 `GET`, 생성은 `POST`, 수정은 `PUT` 또는 `PATCH`, 삭제는 `DELETE`처럼 HTTP 메서드 의미에 맞게 사용해야 합니다. 또 성공과 실패를 상태 코드로 명확히 구분하고, 요청과 응답 구조를 통일해야 협업이 쉬워집니다. 특히 멱등성은 재시도나 중복 요청 방지와 연결되기 때문에 `PUT`, `DELETE`, `GET`의 의미를 정확히 지키는 게 중요합니다.

### 4. Practical Tip
- 실제 적용 사례: 목록 조회 API는 `/records?page=0&size=20&sort=createdAt,desc`처럼 자원 URI와 query parameter를 분리하고, 성공 시 `200`, 생성 시 `201`, 잘못된 요청은 `400`, 권한 문제는 `401/403`으로 명확히 나누는 편이 좋다.
- 잘못 쓰면 생기는 문제: 모든 응답을 `200`으로 보내거나 `/getRecords`, `/deleteUser`처럼 URI를 동사 위주로 설계하면 API 의미가 불명확해지고 클라이언트 협업이 어려워진다.
- 프로젝트 연결: Spring MVC에서는 `@GetMapping`, `@PostMapping`, `@PatchMapping` 등으로 HTTP 메서드 의미를 코드에 직접 드러낼 수 있고, 전역 예외 처리와 공통 응답 포맷을 함께 설계하면 API 일관성이 좋아진다.

### 5. 예상 꼬리 질문
1. `PUT`과 `PATCH`는 실무에서 어떤 기준으로 구분하나요?
- 답변: 전체 리소스 표현을 교체하는 의미가 분명하면 `PUT`, 일부 필드만 선택적으로 수정하면 `PATCH`를 사용하는 편이 자연스럽습니다. 특히 누락 필드까지 새 상태로 간주할지 여부가 구분 기준이 됩니다.
2. RESTful 원칙을 지키기 어려운 액션성 API는 어떻게 설계하나요?
- 답변: 취소, 승인처럼 자원 상태 변경으로 모델링이 가능하면 `/orders/{id}/cancel` 같은 하위 리소스 또는 액션 endpoint를 두고 `POST`로 표현하는 식이 많이 쓰입니다. 핵심은 완벽한 이론보다 자원 맥락과 의도가 읽히는 계약을 만드는 것입니다.
3. 멱등성과 중복 요청 방지는 어떤 관계가 있나요?
- 답변: 멱등성은 같은 요청을 여러 번 보내도 결과가 같아야 한다는 의미적 성질이고, 중복 요청 방지는 그 성질을 실제 시스템에서 보장하기 위한 구현 전략입니다. 즉 멱등성이 설계 원칙이라면, 중복 요청 방지는 이를 operational하게 구현하는 방식이라고 볼 수 있습니다.
