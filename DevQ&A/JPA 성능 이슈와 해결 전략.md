## Topic
JPA 성능 이슈와 해결 전략

### 1. Why
- **실무**: JPA는 편리하지만 성능 문제를 숨기기 쉽다. 특히 조회 API에서는 지연 로딩, N+1, 불필요한 Entity 로딩, OSIV 설정 때문에 예상보다 많은 쿼리가 발생한다.
- **구조적 의미**: JPA 성능 이슈는 단순한 튜닝 문제가 아니라, 어떤 시점에 무엇을 로딩하고 어떤 형태로 조회할지 결정하는 설계 문제다.
- **면접 의도**: 면접에서는 지연 로딩 / 즉시 로딩, N+1, `fetch join`, `EntityGraph`, DTO 조회, OSIV를 서로 연결해 설명할 수 있는지 자주 본다.

### 2. Core Concept
- **개념 정의**: 지연 로딩은 실제 연관 객체가 필요할 때 조회하는 방식이고, 즉시 로딩은 조회 시점에 함께 가져오는 방식이다. N+1 문제는 1번의 조회 후 연관 데이터를 읽는 과정에서 추가 쿼리가 N번 더 발생하는 문제다.
- **동작 방식**: 기본적으로 연관관계는 지연 로딩을 선호하는 편이 안전하다. 하지만 목록 조회 후 연관 데이터를 반복 접근하면 N+1이 생길 수 있고, 이때 `fetch join`, `EntityGraph`, DTO 조회 같은 전략으로 필요한 데이터만 한 번에 가져온다. OSIV는 영속성 컨텍스트를 뷰까지 열어 두는 방식인데, 편의성은 있지만 계층 경계를 흐리고 예기치 않은 쿼리를 유발할 수 있다.
- **장점 / 단점**: 지연 로딩은 불필요한 데이터 로딩을 줄이지만 접근 시점 제어를 못 하면 N+1이 생긴다. `fetch join`은 효과적이지만 컬렉션 조인과 페이징을 함께 쓸 때 주의가 필요하다. `EntityGraph`는 선언적으로 연관 조회를 제어하기 좋고, DTO 조회는 필요한 컬럼만 가져와 성능상 유리하지만 Entity 변경 추적은 할 수 없다.
- **비교 포인트**: 즉시 로딩이 항상 빠른 것은 아니고, 기본값으로 두면 예상 못 한 조인이 붙어 더 느려질 수 있다. 읽기 전용 API나 복잡한 목록 조회는 DTO 조회가 유리하고, 연관 Entity를 실제 도메인 로직에서 써야 할 때는 `fetch join`이나 `EntityGraph`가 적합하다. OSIV는 개발 편의성은 높지만 운영 관점에선 주의가 필요하다.
- **사용 조건 / 주의점**: 연관관계는 우선 지연 로딩으로 두고, 실제 조회 시점에 필요한 연관만 명시적으로 가져오는 습관이 좋다. 컬렉션 `fetch join`과 페이징은 조심해야 하고, API 계층 밖에서 무심코 연관 객체를 접근하지 않도록 OSIV 설정과 DTO 변환 위치를 함께 봐야 한다.

### 3. Interview Answer Version
- 30~40초 답변: JPA 성능 이슈에서 가장 대표적인 것은 N+1 문제라고 생각합니다. 보통 연관관계를 지연 로딩으로 두는 것이 안전하지만, 목록 조회 후 연관 데이터를 반복 접근하면 추가 쿼리가 N번 더 발생할 수 있습니다. 이럴 때는 상황에 따라 `fetch join`, `EntityGraph`, DTO 조회를 사용해 필요한 데이터만 한 번에 가져오는 것이 중요합니다. 또 OSIV를 켜 두면 개발은 편하지만 뷰나 컨트롤러에서 예기치 않은 쿼리가 나갈 수 있어서, 운영 환경에서는 계층 경계를 의식해서 사용하는 편이 좋습니다.

### 4. Practical Tip
- 실제 적용 사례: 주문 목록 화면에서 주문과 회원 이름이 함께 필요하면 `fetch join`이나 DTO projection으로 한 번에 가져오고, 단순 목록에 불필요한 연관 객체까지 전부 로딩하지 않도록 설계한다.
- 잘못 쓰면 생기는 문제: 즉시 로딩을 남발하면 예상 못 한 조인이 늘어나고, 지연 로딩을 이해 없이 쓰면 컨트롤러나 직렬화 시점에서 N+1이나 `LazyInitializationException`이 발생한다.
- 프로젝트 연결: Spring Data JPA에서는 Repository 메서드마다 조회 목적을 분리하고, 읽기 전용 API는 DTO 조회, 연관 객체 활용이 필요한 경우는 `fetch join` 또는 `EntityGraph`를 선택하는 식으로 기준을 세우는 것이 좋다.

### 5. 예상 꼬리 질문
1. 즉시 로딩보다 지연 로딩을 기본으로 두는 이유는 무엇인가요?
- 답변: 즉시 로딩을 기본으로 두면 필요하지 않은 연관 객체까지 함께 불러오면서 예기치 않은 조인과 성능 저하가 생기기 쉽습니다. 지연 로딩을 기본으로 두고, 필요한 조회 시점에만 명시적으로 가져오는 편이 제어하기 더 쉽습니다.
2. `fetch join`과 `EntityGraph`는 어떤 기준으로 선택하나요?
- 답변: 쿼리 수준에서 조인 구조를 직접 제어하고 싶으면 `fetch join`이 직관적입니다. 반면 Repository 메서드에 선언적으로 연관 로딩 정책을 붙이고 재사용하고 싶다면 `EntityGraph`가 더 깔끔할 수 있습니다.
3. OSIV를 끄면 어떤 점이 좋아지고 무엇을 주의해야 하나요?
- 답변: OSIV를 끄면 웹 계층에서 숨은 쿼리가 나가는 일을 줄이고, 영속성 컨텍스트와 DB 커넥션 점유 구간을 더 명확히 통제할 수 있습니다. 대신 서비스 계층 안에서 필요한 데이터를 DTO로 미리 변환하거나 연관 로딩을 끝내야 합니다.
