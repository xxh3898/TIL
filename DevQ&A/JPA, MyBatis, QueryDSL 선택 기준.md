## Topic
JPA, MyBatis, QueryDSL 선택 기준

### 1. Why
- **실무**: 데이터 접근 기술은 프로젝트 생산성과 유지보수성, 성능 대응 방식에 직접 영향을 준다. 무조건 한 기술만 고집하기보다 요구사항에 맞는 선택 기준이 있어야 한다.
- **구조적 의미**: JPA, MyBatis, QueryDSL은 경쟁 관계이면서도 일부는 보완 관계다. 특히 QueryDSL은 JPA를 대체하는 ORM이라기보다 JPA 위에서 동적 쿼리를 더 안전하게 작성하기 위한 도구라는 점을 구분해야 한다.
- **면접 의도**: 면접에서는 각 기술의 장단점만 나열하는 것이 아니라, 어떤 상황에서 무엇을 선택하고 혼합 사용 시 기준을 어떻게 잡는지 보려는 경우가 많다.

### 2. Core Concept
- **개념 정의**: JPA는 객체와 관계형 DB를 매핑하는 ORM 표준이고, MyBatis는 SQL 중심으로 직접 쿼리를 작성하는 매퍼 프레임워크다. QueryDSL은 타입 세이프하게 동적 쿼리를 작성하도록 돕는 쿼리 빌더다.
- **동작 방식**: JPA는 Entity와 영속성 컨텍스트를 중심으로 동작해 객체 지향적으로 CRUD를 다루기 좋다. MyBatis는 SQL을 직접 제어하므로 복잡한 쿼리 튜닝과 DB 특화 기능 활용이 쉽다. QueryDSL은 보통 JPA와 함께 쓰이며, 조건 조합이 많은 조회 로직을 문자열 없이 코드로 표현한다.
- **장점 / 단점**: JPA는 생산성이 높고 객체 중심 모델링에 유리하지만 내부 동작을 모르면 성능 문제를 숨기기 쉽다. MyBatis는 SQL 제어력이 높고 복잡한 조회에 강하지만 반복 매핑 코드와 SQL 관리 비용이 크다. QueryDSL은 동적 쿼리 유지보수에 강하지만 설정과 학습 비용이 있고, 결국 JPA 이해가 전제되어야 한다.
- **비교 포인트**: 단순 CRUD와 도메인 모델 중심 개발은 JPA가 유리하다. SQL 튜닝이 중요한 복잡한 조회나 DB 특화 기능 활용은 MyBatis가 유리할 수 있다. JPA를 쓰면서 검색 조건 조합이 많아지면 QueryDSL 도입 이유가 분명해진다. 즉 QueryDSL은 JPA와 대체 관계라기보다 보완 관계다.
- **사용 조건 / 주의점**: 기술 선택은 팀 숙련도, 쿼리 복잡도, 도메인 모델의 중요도, 성능 요구사항을 함께 보고 결정해야 한다. 혼합 사용 시에는 생성/수정은 JPA, 복잡한 조회는 QueryDSL 또는 MyBatis처럼 책임을 분리하는 기준을 명확히 해야 한다.

### 3. Interview Answer Version
- 30~40초 답변: 저는 세 기술을 하나만 정답으로 보기보다 역할에 따라 선택합니다. JPA는 객체 중심 모델링과 CRUD 생산성이 좋아서 기본 선택지로 적합하고, QueryDSL은 JPA를 쓸 때 동적 조회 조건이 많아지면 타입 세이프한 쿼리 작성을 위해 함께 도입하는 경우가 많습니다. 반면 SQL 제어력이 아주 중요하거나 DB 특화 쿼리가 많은 경우는 MyBatis가 더 유리할 수 있습니다. 실무에서는 생성과 수정은 JPA, 복잡한 조회는 QueryDSL이나 MyBatis로 분리하는 혼합 전략도 충분히 현실적입니다.

### 4. Practical Tip
- 실제 적용 사례: 회원 가입, 주문 생성, 단건 조회 같은 기본 도메인 로직은 JPA로 처리하고, 관리자 검색 화면처럼 조건 조합이 많은 목록 조회는 QueryDSL로 분리하면 균형이 좋다.
- 잘못 쓰면 생기는 문제: JPA 내부 동작을 이해하지 못한 채 모든 조회를 JPA 기본 메서드로만 밀어붙이면 N+1과 불필요한 Entity 로딩이 생기고, 반대로 모든 로직을 MyBatis SQL로만 작성하면 객체 모델의 장점과 생산성을 잃기 쉽다.
- 프로젝트 연결: Spring 프로젝트에서는 도메인 중심 CRUD는 JPA, 검색 API는 QueryDSL, 극단적으로 복잡한 통계성 조회는 MyBatis를 선택하는 혼합 구성이 자주 사용된다.

### 5. 예상 꼬리 질문
1. QueryDSL 없이도 JPA로 동적 쿼리를 만들 수 있는데 왜 굳이 도입하나요?
- 답변: 가능은 하지만 조건이 많아질수록 `@Query` 문자열이나 Criteria API는 가독성과 유지보수성이 급격히 떨어질 수 있습니다. QueryDSL은 타입 안정성과 IDE 지원이 좋아서 동적 조회가 많은 프로젝트에서 차이가 크게 납니다.
2. MyBatis를 쓰면 JPA보다 성능이 무조건 더 좋은가요?
- 답변: 무조건 그렇지는 않습니다. MyBatis는 SQL 제어력이 높아 튜닝에 유리할 수 있지만, 최종 성능은 쿼리 자체와 인덱스, 조회 전략에 더 크게 좌우됩니다.
3. 한 프로젝트에서 JPA와 MyBatis를 같이 써도 괜찮은가요?
- 답변: 충분히 가능합니다. 다만 생성/수정은 JPA, 복잡한 통계성 조회는 MyBatis처럼 책임 분리 기준과 트랜잭션 경계를 명확히 두는 것이 중요합니다.
