## Topic
SQL 성능과 조회 설계 기초

### 1. Why
- **실무**: 애플리케이션 병목의 상당수는 조회 쿼리에서 발생한다. 특히 join, 서브쿼리, 정렬과 페이징, 집계 쿼리를 어떻게 설계하느냐에 따라 API 성능이 크게 달라진다.
- **구조적 의미**: 조회 설계는 단순 SQL 문법 문제가 아니라, 필요한 데이터를 어떤 조합과 순서로 가져올지 결정하는 데이터 접근 설계다.
- **면접 의도**: 면접에서는 join, 서브쿼리, `group by / having`, 정렬과 페이징, 느린 쿼리 원인을 이해하고 조회 API 설계 시 어떤 점을 주의하는지 설명할 수 있는지 본다.

### 2. Core Concept
- **개념 정의**: join은 여러 테이블을 연결해 조회하는 방식이고, 서브쿼리는 쿼리 안에 포함된 내부 쿼리다. `group by`는 집계 단위를 묶고, `having`은 집계 결과에 조건을 거는 절이다.
- **동작 방식**: 조회 API에서는 보통 필요한 컬럼만 선택하고, 필요한 테이블만 join하며, 정렬과 페이징을 적용해 응답 범위를 제한한다. 서브쿼리는 유용하지만 경우에 따라 join으로 풀 수 있고, 집계가 많아질수록 인덱스와 정렬 비용까지 함께 고려해야 한다.
- **장점 / 단점**: join은 한 번에 데이터를 가져오기에 좋지만 과도한 다중 join은 실행 계획을 무겁게 만들 수 있다. 서브쿼리는 표현력이 좋지만 DB가 비효율적으로 실행할 수 있다. 정렬과 페이징은 사용자 경험에 필수지만 인덱스와 맞지 않으면 느려지기 쉽다.
- **비교 포인트**: `WHERE`는 행 필터링, `HAVING`은 집계 결과 필터링이다. 단순 조건은 서브쿼리보다 join이 더 읽기 쉽고 최적화되기 좋은 경우가 많다. 또 목록 API에서 전체 컬럼을 가져오는 것보다 필요한 응답 컬럼만 조회하는 것이 보통 유리하다.
- **사용 조건 / 주의점**: 느린 쿼리 원인은 보통 불필요한 풀스캔, 과도한 정렬, 잘못된 join 순서, 인덱스 미스, 비효율적인 count 쿼리에서 나온다. 조회 API를 설계할 때는 응답 스펙을 먼저 정하고, 그에 필요한 최소 컬럼과 조건만 조회하도록 맞추는 것이 중요하다.

### 3. Interview Answer Version
- 30~40초 답변: SQL 성능은 결국 필요한 데이터를 얼마나 적은 비용으로 가져오느냐의 문제라고 생각합니다. 그래서 조회 설계에서는 먼저 필요한 컬럼만 조회하고, join은 꼭 필요한 관계만 연결하며, 정렬과 페이징이 인덱스와 맞는지 확인하는 것이 중요합니다. `group by / having`은 집계 쿼리에서 자주 쓰이고, 서브쿼리는 표현력이 좋지만 경우에 따라 join으로 더 효율적으로 풀 수 있습니다. 느린 쿼리는 보통 풀스캔, 정렬 비용, 잘못된 인덱스 사용에서 나오기 때문에 실행 계획까지 함께 봐야 합니다.

### 4. Practical Tip
- 실제 적용 사례: 목록 API는 `select *` 대신 필요한 응답 컬럼만 조회하고, 정렬 기준 컬럼과 검색 조건 컬럼에 맞는 인덱스를 함께 검토하는 것이 기본이다.
- 잘못 쓰면 생기는 문제: 화면에 5개 필드만 필요한데 연관 테이블까지 모두 join하고 전체 컬럼을 읽으면 네트워크, 메모리, DB 비용이 모두 증가한다.
- 프로젝트 연결: JPA나 QueryDSL을 쓰더라도 결국 DB는 SQL을 실행하므로, 실행 계획을 보고 join 수, 정렬 기준, count 쿼리 비용을 확인하는 습관이 필요하다.

### 5. 예상 꼬리 질문
1. 서브쿼리보다 join이 항상 더 좋은가요?
- 답변: 항상 그렇지는 않습니다. DB 옵티마이저와 쿼리 목적에 따라 서브쿼리가 더 자연스럽거나 충분히 효율적인 경우도 있으므로, 실행 계획과 가독성을 함께 보고 판단해야 합니다.
2. `group by`와 `having`은 어떤 상황에서 자주 쓰이나요?
- 답변: 카테고리별 건수, 회원별 주문 합계처럼 집계 결과를 그룹 단위로 보고 싶을 때 자주 씁니다. `having`은 그 집계 결과를 다시 필터링할 때 사용합니다.
3. 조회 API에서 `select *`를 피해야 하는 이유는 무엇인가요?
- 답변: 실제로 필요 없는 컬럼까지 읽어 오면 I/O와 메모리 사용량이 늘고, 스키마 변경에도 더 민감해집니다. 응답에 필요한 컬럼만 가져오는 것이 성능과 계약 안정성 모두에 유리합니다.
