# API 문서화 도구 비교: Spring Rest Docs vs Swagger

## 1. Spring Rest Docs
**"테스트가 곧 문서다"**라는 철학을 가진 도구입니다.

###  작동 방식
1.  **테스트 코드 작성**: 컨트롤러에 대한 슬라이스 테스트(`MockMvc` 등)를 작성합니다.
2.  **스니펫(Snippet) 생성**: 테스트가 성공하면 해당 요청/응답 정보가 `.adoc` 파일들로 추출됩니다.
3.  **문서 조립**: 미리 작성한 템플릿(`index.adoc`)에 스니펫들을 포함(include)시킵니다.
4.  **HTML 변환**: Asciidoctor가 최종적으로 가독성 좋은 HTML 문서를 빌드합니다.

### 장점
-   **100% 신뢰도**: 테스트가 통과해야만 문서가 만들어지므로, 실제 코드와 문서가 다를(Drift) 확률이 거의 없습니다.
-   **코드 정결성**: 비즈니스 코드(Controller)에 문서용 어노테이션을 도배하지 않아도 됩니다. (순수 Java 코드 유지)
-   **커스터마이징**: Asciidoc의 기능을 활용해 가독성 높고 정교한 레이아웃을 구성할 수 있습니다.

### 단점
-   **높은 학습 곡선**: Asciidoc 문법과 설정이 다소 복잡합니다.
-   **번거로움**: 모든 API에 대해 성공하는 테스트 케이스를 반드시 작성해야 합니다.

---

## 2. Swagger (Springdoc-OpenAPI)
**"어노테이션 기반 실시간 문서화"** 도구입니다.

### 작동 방식
1.  **의존성 추가**: 라이브러리를 추가하면 자동으로 `/swagger-ui/index.html`이 생깁니다.
2.  **어노테이션 추가**: `@Operation`, `@Schema` 등 어노테이션으로 API 설명을 보강합니다.
3.  **UI 자동 생성**: 서버 실행 시 코드를 스캔하여 인터랙티브한 웹 페이지를 보여줍니다.

### 장점
-   **인터랙티브 UI**: "Try it out" 버튼으로 웹상에서 즉시 API를 호출하고 결과를 볼 수 있습니다.
-   **간편한 설정**: 의존성만 넣으면 기본 문서는 바로 나오므로 초기 적용이 매우 빠릅니다.
-   **높은 인지도**: 많은 개발자들에게 익숙하며, 프론트엔드 협업 시 편리합니다.

### 단점
-   **코드 오염**: 컨트롤러에 문서용 코드가 실제 로직보다 비대해지는 경우가 많습니다.
-   **신뢰도 하락**: 테스트와 연결되지 않기 때문에, 로직은 바꿨는데 어노테이션 수정을 깜빡하면 문서가 거짓말을 하게 됩니다.

---

## 3. 요약 및 선택 가이드

| 비교 항목 | Spring Rest Docs | Swagger (OpenAPI) |
| :--- | :--- | :--- |
| **문서 품질** | 높음 (정교함) | 보통 (정형화됨) |
| **코드 가독성** | 우수 (오염 없음) | 낮음 (어노테이션 많음) |
| **신뢰도** | 매우 높음 (테스트 기반) | 낮음 (수동 입력 기반) |
| **테스트 필수 여부** | 필수 (테스트 없으면 문서 없음) | 선택 (테스트 없어도 노출 가능) |
| **인터랙티브 기능** | 없음 (정적 HTML) | **있음 (직접 호출 가능)** |

### 어떤 것을 선택할까?
-   **Spring Rest Docs 추천**: "나는 테스트 코드를 철저히 짤 것이며, 비즈니스 코드를 깔끔하게 유지하고 싶다." (주로 **백엔드 중심/안정성 중시** 프로젝트)
-   **Swagger 추천**: "프론트엔드와 실시간으로 소통하며 직접 호출해보는 기능이 필요하고, 빠른 개발이 우선이다." (주로 **속도가 생명인 스타트업/소규모** 프로젝트)
