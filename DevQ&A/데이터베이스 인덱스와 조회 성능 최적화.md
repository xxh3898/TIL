## Topic
데이터베이스 인덱스와 조회 성능 최적화

### 1. Why
- **실무**: 조회 성능 문제를 가장 자주 개선하는 수단 중 하나가 인덱스다. 하지만 인덱스를 무작정 많이 추가하면 쓰기 성능과 저장 공간 비용이 커진다.
- **구조적 의미**: 인덱스는 쿼리 성능을 위한 자료구조 설계다. 어떤 컬럼 조합을 어떤 순서로 인덱싱할지 결정하는 것은 곧 조회 패턴을 구조화하는 일이다.
- **면접 의도**: 면접에서는 인덱스란 무엇인지, `B-Tree`, 복합 인덱스, 실행 계획, `full scan`, 인덱스 설계 주의점을 실제 성능 맥락과 함께 설명할 수 있는지 본다.

### 2. Core Concept
- **개념 정의**: 인덱스는 특정 컬럼 값을 빠르게 찾기 위한 자료구조다. 일반적인 RDBMS 인덱스는 `B-Tree` 기반으로 동작해 정렬된 탐색을 효율적으로 지원한다.
- **동작 방식**: 인덱스가 적절하면 DB는 테이블 전체를 읽는 `full scan` 대신 인덱스를 타고 필요한 범위만 접근할 수 있다. 복합 인덱스는 여러 컬럼을 함께 묶은 인덱스로, 보통 조회 조건과 정렬 순서에 맞춰 왼쪽 컬럼부터 활용된다. 실제 사용 여부는 실행 계획으로 확인한다.
- **장점 / 단점**: 인덱스는 조회 속도를 크게 높일 수 있지만, 추가/수정/삭제 시 인덱스도 함께 갱신돼 쓰기 비용이 증가한다. 저장 공간도 더 필요하고, 잘못 만든 인덱스는 기대만큼 효과가 없다.
- **비교 포인트**: 단일 인덱스는 단순하지만 조건 조합이 많으면 비효율적일 수 있다. 복합 인덱스는 강력하지만 컬럼 순서가 매우 중요하다. 실행 계획을 보지 않고 "인덱스가 있으니 빠르다"고 판단하면 실제 병목을 놓치기 쉽다.
- **사용 조건 / 주의점**: 자주 조회되는 조건, 조인 조건, 정렬 조건을 기준으로 인덱스를 설계해야 한다. 선택도가 낮은 컬럼에만 인덱스를 두면 효과가 약할 수 있고, 복합 인덱스는 조회 패턴에 맞는 컬럼 순서를 신중히 정해야 한다. 항상 실행 계획으로 `full scan`, index range scan 여부를 확인하는 습관이 중요하다.

### 3. Interview Answer Version
- 30~40초 답변: 인덱스는 테이블을 전부 읽지 않고 원하는 데이터를 빠르게 찾기 위한 자료구조이고, 대부분 `B-Tree` 기반으로 동작합니다. 적절한 인덱스가 있으면 `full scan` 대신 인덱스를 타고 필요한 범위만 접근할 수 있어서 조회 성능이 좋아집니다. 다만 인덱스는 쓰기 비용과 저장 공간 비용이 있기 때문에 무조건 많이 두는 게 아니라, 자주 쓰는 검색 조건과 정렬 조건 중심으로 설계해야 합니다. 특히 복합 인덱스는 컬럼 순서가 중요해서 실행 계획을 보고 실제로 타는지 검증하는 것이 핵심입니다.

### 4. Practical Tip
- 실제 적용 사례: `where status = ? and created_at < ? order by created_at desc` 같은 조회가 많다면 `status`, `created_at` 순서의 복합 인덱스를 검토할 수 있다.
- 잘못 쓰면 생기는 문제: 조회 패턴을 보지 않고 인덱스를 많이 추가하면 쓰기 성능이 떨어지고, 컬럼 순서를 잘못 잡은 복합 인덱스는 있어도 못 쓰는 경우가 생긴다.
- 프로젝트 연결: API 성능이 느릴 때는 코드만 보기보다 DB 실행 계획을 먼저 확인하고, 실제 검색 조건과 정렬 패턴 기준으로 인덱스를 조정하는 편이 효과적이다.

### 5. 예상 꼬리 질문
1. 복합 인덱스에서 컬럼 순서는 왜 중요한가요?
- 답변: 복합 인덱스는 보통 왼쪽 컬럼부터 순서대로 활용되기 때문에, 자주 쓰는 검색 조건과 정렬 순서에 맞춰 설계해야 효과가 큽니다. 순서를 잘못 잡으면 인덱스가 있어도 원하는 방식으로 못 타는 경우가 생깁니다.
2. 인덱스가 있는데도 `full scan`이 발생하는 이유는 무엇인가요?
- 답변: 조건 선택도가 낮거나, 컬럼에 함수가 적용되었거나, 통계 정보상 옵티마이저가 풀스캔이 더 낫다고 판단하면 그럴 수 있습니다. 그래서 인덱스 존재 여부보다 실제 실행 계획 확인이 더 중요합니다.
3. 인덱스를 많이 만들면 왜 쓰기 성능이 느려지나요?
- 답변: `INSERT`, `UPDATE`, `DELETE`가 일어날 때 테이블 데이터뿐 아니라 관련 인덱스도 함께 갱신해야 하기 때문입니다. 즉 조회 성능을 얻는 대신 쓰기 비용과 저장 공간 비용을 지불하는 구조입니다.
