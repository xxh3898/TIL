## Topic (자바의 예외 처리) 2025-11-29

- **주제:** 자바의 예외 계층 구조 (Checked vs Unchecked Exception)

### **Why (왜 사용하는가? 왜 중요한가?)**

> 이 개념이 실무, 설계, 면접에서 중요한 이유를 정리한다.

- **실무**: 예외 처리를 제대로 하지 않으면 사소한 오류로 서버 전체가 멈출 수 있다. 또한 트랜잭션 롤백(Rollback) 정책 설정의 기준이 된다.

- **구조적 의미**: 프로그램의 정상적인 흐름을 방해하는 오류를 미리 예측하고 대비(Try-Catch)하여 프로그램의 안정성을 높인다.

- **면접 의도**: 컴파일 시점에 체크하는 예외와 런타임 예외의 차이를 알고 있는지, 올바른 예외 처리 전략(회피, 복구, 전환 등)을 가지고 있는지 확인한다.

### **Core Concept (핵심 개념 정리)**

> 단순한 정의가 아닌, 의미 + 구조 + 비교 + 조건 중심으로 작성한다.

|요소|내용|
|---|---|
|**개념 정의**|프로그램 실행 중 발생할 수 있는 오류(Error)와 예외(Exception)를 객체로 관리하는 메커니즘.|
|**계층 구조**|**Throwable** (최상위)<br><br>  <br><br>├── **Error**: 시스템 레벨의 심각한 오류 (개발자 해결 불가, 예: OOM, StackOverflow)<br><br>  <br><br>└── **Exception**: 개발자가 코드로 수습 가능한 오류|
|**구분 (중요)**|**Checked Exception**: 컴파일 시 확인 (필수 처리). 예: `IOException`, `SQLException`<br><br>  <br><br>**Unchecked Exception**: 실행 시 확인 (처리 선택). 예: `RuntimeException`, `NullPointerException`|
|**키워드**|`try-catch`(처리), `finally`(무조건 실행), `throw`(예외 발생), `throws`(예외 던지기/미루기)|
|**비교**|**Error vs Exception**: 복구 불가능(시스템) vs 복구 가능(코드)|

### **Interview Answer Version (면접 답변식 요약)**

> 30~40초 분량으로, **정의 → 쓰는 이유 → 핵심 효과/특징**순으로 말하듯 요약

"자바의 예외는 크게 복구 불가능한 **Error**와 복구 가능한 **Exception**으로 나뉩니다. Exception은 다시 **Checked Exception**과 **Unchecked Exception**으로 구분되는데, Checked는 컴파일 시점에 확인되어 반드시 예외 처리를 해야 하고, Unchecked는 `RuntimeException`을 상속받아 명시적인 처리가 강제되지 않습니다. 실무에서는 주로 런타임 예외를 활용하여 트랜잭션 롤백 등을 관리합니다."

### **Practical Tip (사용시 주의할 점 or 활용 예)**

> 실제로 어떻게 쓰이며, 잘못 쓰면 어떤 문제가 생기는지를 구체적인 예시로 작성헌다.

- **예외 삼키기 (Swallowing) 금지**:
    
    Java
    
    ```
    try { ... } catch (Exception e) { } // 절대 금지!
    ```
    
    - 에러가 발생해도 로그조차 남지 않아 원인 파악이 불가능해짐. 최소한 `e.printStackTrace()`나 로깅을 해야 함.
    
- **트랜잭션과 예외 (Spring Framework)**:
    
    - 기본적으로 `@Transactional`은 **Unchecked Exception(런타임 예외)** 발생 시에만 롤백함.
    
    - Checked Exception이 발생해도 롤백하고 싶다면 `rollbackFor = Exception.class` 옵션을 줘야 함.
    
- **리소스 정리**:
    
    - `try-catch-finally` 보다는 **`try-with-resources`** (Java 7+)를 사용하여 DB 연결이나 파일 입출력 스트림을 자동으로 닫는 것이 안전함.
