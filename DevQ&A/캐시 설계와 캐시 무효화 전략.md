## Topic
캐시 설계와 캐시 무효화 전략

### 1. Why
- **실무**: 캐시는 응답 속도와 DB 부하를 크게 개선하지만, 잘못 설계하면 오래된 데이터가 내려가거나 무효화가 꼬여 정합성 문제가 생긴다. 그래서 조회 성능만이 아니라 무효화 전략까지 같이 설계해야 한다.
- **구조적 의미**: 캐시는 원본 저장소 앞단에 두는 성능 계층이다. 핵심은 "무엇을 캐시할 것인가"보다 "언제 비우고 언제 다시 채울 것인가"를 결정하는 것이다.
- **면접 의도**: 면접에서는 `cache aside`, `TTL`, `cache miss`, 무효화 시점을 이해하고, 캐시를 단순 성능 도구가 아니라 정합성과 함께 설명할 수 있는지 본다.

### 2. Core Concept
- **개념 정의**: 캐시는 자주 조회되는 데이터를 빠른 저장소에 임시 저장해 재사용하는 구조다. `cache aside`는 애플리케이션이 캐시를 먼저 조회하고 없으면 DB에서 읽어 캐시에 저장하는 가장 흔한 패턴이다.
- **동작 방식**: 요청이 오면 먼저 캐시를 조회한다. 데이터가 있으면 `cache hit`, 없으면 `cache miss`가 발생하고 DB에서 조회한 뒤 캐시에 저장한다. 이때 `TTL`을 설정해 일정 시간이 지나면 자동 만료되도록 하고, 데이터 변경 시점에는 명시적으로 캐시를 삭제하거나 갱신한다.
- **장점 / 단점**: 장점은 조회 성능 향상과 DB 부하 감소다. 단점은 캐시와 원본 데이터 간 불일치 가능성, 무효화 전략 복잡성, 캐시 스탬피드 같은 부하 문제가 있다는 점이다.
- **비교 포인트**: `cache aside`는 구현이 단순하고 유연해 가장 널리 쓰인다. 반면 write-through, write-behind는 쓰기 경로까지 캐시를 강하게 연동하는 방식이라 정합성 전략과 운영 복잡도가 다르다. `TTL` 기반 만료는 단순하지만 즉시 반영이 어렵고, 이벤트 기반 무효화는 정확하지만 설계가 더 복잡하다.
- **사용 조건 / 주의점**: 무효화 시점은 데이터 변경 직후가 기본이다. 단, 모든 데이터를 무조건 캐시하면 안 되고 조회 빈도, 변경 빈도, 허용 가능한 stale 정도를 보고 결정해야 한다. `cache miss`가 한꺼번에 몰릴 수 있으므로 적절한 `TTL` 분산, preload, 락 전략도 함께 고려해야 한다.

### 3. Interview Answer Version
- 30~40초 답변: 캐시 설계에서 가장 많이 쓰는 패턴은 `cache aside`입니다. 애플리케이션이 먼저 캐시를 보고, 없으면 DB에서 조회해서 캐시에 넣는 방식입니다. 이때 중요한 건 단순히 캐시를 두는 게 아니라 `TTL`과 무효화 시점을 함께 설계하는 것입니다. 읽기 성능은 좋아지지만 캐시와 원본 데이터가 어긋날 수 있기 때문에, 데이터 변경 직후 캐시를 비우거나 다시 갱신하는 전략이 필요합니다. 또 `cache miss`가 몰리면 DB 부하가 다시 커질 수 있어서 만료 시간 분산이나 스탬피드 대응도 같이 봐야 합니다.

### 4. Practical Tip
- 실제 적용 사례: 상품 상세나 인기 게시글 목록처럼 조회는 많고 쓰기는 상대적으로 적은 데이터에 Redis 캐시를 두고, 수정 시 해당 key를 삭제하는 방식이 가장 흔하다.
- 잘못 쓰면 생기는 문제: `TTL`만 믿고 무효화를 안 하면 수정 후 오래된 데이터가 계속 내려갈 수 있고, 너무 짧게 잡으면 캐시 효율이 낮아져 DB 부하만 늘어난다.
- 프로젝트 연결: Spring Cache 또는 Redis를 사용할 때는 캐시 key 설계, `TTL`, 삭제 시점, 예외 상황에서 캐시 실패를 어떻게 처리할지까지 같이 정해야 운영이 안정적이다.

### 5. 예상 꼬리 질문
1. `cache aside`와 write-through는 어떤 상황에서 다르게 선택하나요?
- 답변: 읽기 위주 서비스이고 구현 단순성이 중요하면 `cache aside`가 가장 일반적입니다. 반면 쓰기 시점에도 캐시와 저장소를 더 강하게 동기화해야 한다면 write-through를 고려할 수 있지만 운영 복잡도는 더 올라갑니다.
2. 캐시 무효화 시점을 잘못 잡으면 어떤 장애가 생기나요?
- 답변: 너무 늦게 무효화하면 수정된 데이터가 반영되지 않아 stale data가 계속 내려갈 수 있습니다. 반대로 너무 공격적으로 무효화하면 캐시 효율이 떨어지고 `cache miss`가 몰리면서 원본 DB 부하가 급증할 수 있습니다.
3. 캐시 스탬피드는 어떻게 완화할 수 있나요?
- 답변: `TTL`에 랜덤 값을 섞어 만료 시점을 분산시키고, 특정 key 재생성을 락으로 직렬화하거나, 미리 갱신하는 방식이 많이 쓰입니다. 경우에 따라 짧은 시간 동안 stale 데이터를 허용하는 전략도 현실적인 완화책이 됩니다.
