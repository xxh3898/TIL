#### 1\. 프로젝트 개요 및 기획 배경 (Why?)

-   팀원이 실제로 겪었던 일을 토대로, 현재 병원의 자동화되지 않고 사람이 직접 조회하고 예약하는 비효율적이고 복잡했던 예약 시스템을 개선하고자 병원 시설 예약을 중점으로한 실시간 예약 및 통합 관리 병원 ERP 시스템을 기획하였다.
-   나의 역할: DB 관리자로서 초기 테이블 설계와 각 테이블 간의 관계 설정 그리고 컬럼들의 속성을 데이터 무결성을 고려하여 설계했다

#### 2\. 기술적 의사결정 및 구현 (How?)

-   **기술 스택 활용:**
    -   Java Spring Boot와 MyBatis를 사용하여 백엔드 로직을 MVC 패턴에 따라 기능별로 각각 Model, View, Controller로 나누어 코드의 가독성을 높이고 재사용성을 높였다.
    -   Oracle DB를 구상하며 필요한 테이블을 도출하였으며, 직접 시나리오를 만들어 실제로 가상의 테이블에 데이터를 넣어보며 보완 및 수정하며 테이블을 구상해나갔다.  
        특히 정규화를 지키고자 상당히 많은 테이블을 만들며 참조관계에 더욱 신경 썼고, 다수의 테이블 조인으로 인한 쿼리 복잡도 증가와 성능 저하를 방지하기 위해, 자주 조회되는 데이터셋을 Database View로 정의하여 쿼리 가독성과 개발 효율성을 높였다.  
        처음엔 Join을 많이 하며 복잡해지는 쿼리를 보며 이게 맞나 싶었지만 View를 활용해 자주 조인하는 데이터들을 묶어 관리하였다.
    -   JSP와 JSTL/EL을 사용하여 동적 페이지를 구성할 때 데이터를 담는 객체인 VO 실제 DB에서 값을 가져올 Mapper를 구상하여 가독성과 유지보수성을 높였으며, Service 구현체에서 비즈니스 로직을 처리하였고, 컨트롤러에서 필요한 값을 넣고, 요청하며 기능마다 나눴다.
-   **주요 구현 기능:**
    -   근태 관련 페이지를 맡았으며 주요 기능으로는
        -   나의 근태 현황: 이전 출퇴근 기록을 보여준다.
        -   나의 신청 내역: 이전 신청 내역을 보여준다.
        -   승인 관리: 컨트롤러에서 로그인한 멤버 아이디를 비교해 관리자일 경우에만 활성화 되도록 JSP에서 조건문을 넣었다.
        -   전체 근태 현황: 오늘 날짜를 필터링하여 직원들의 간단한 정보와 근무일정, 상태(지각, 정상 등)을 한 눈에 시각화 하였다.
        -   출퇴근 기능: DB SCHEDULE 테이블에 입력되어있는 요일과 현재 요일을 비교하여 출/퇴근 시간을 가져와 출근버튼을 눌렀을 때 ABSENCE 테이블에 출근시간이 기록되며 서비스 구현체에서 SCHEDULE 테이블의 SCHEDULE\_START\_TIME과 비교하여 조건문을 통해 정상 출근과 지각/조퇴 여부를 판별하고, 해당 상태 코드를 ABSENCE 테이블에 저장하여 데이터 정합성을 유지했다.  
            퇴근 버튼을 눌렀을 때도 마찬가지로 SCHEDULE\_END\_TIME과 퇴근 시간을 비교하여 정상/조퇴를 구분하여 ABSENCE 테이블의 ABSENCE\_END에 값을 넣음
        -   신청서 작성: 외출/휴가/외박으로 나눴으며 외출일 경우 시간을 입력하게 구현하였다. 사유와 시간을 입력하여야만 결재요청이 성공적으로 이루어지며 관리자 로그인 시 활성화되는 승인 관리 탭에서 승인 혹은 반려를 할 수 있다.

#### 3\. 트러블 슈팅 (Problem Solving)

-   **문제:** 초기에 ERD를 작성하며 테이블을 만들고 컬럼과 제약 조건 참조 등 완벽하게 만들었다고 생각하였던 것들이 실제로 기능을 설계하고 구현하다보니 바뀌는 부분이 많았고 이를 다른 팀원들과 원활하게 공유가 되지 않아 각자의 비즈니스 로직에 문제가 생김
-   **시도:** 문제를 해결하기 위해 다시 ERD를 검토하며 UI와 비교하여 수정하며 DB관리자뿐만 아닌 모든 팀원들이 머리를 맞대 고민하였다.
-   **해결:** 초기 설계와 실제 요구사항(UI) 간의 괴리를 좁히기 위해, 팀원들과 협의하여 유연하게 스키마를 변경하는 애자일(Agile)한 접근 방식을 취했다.
-   **배운 점:** 생각보다 많은 경우의 수가 존재하며 사소한 차이가 큰 결과를 초래할 수 있다는 것을 배웠다  
    우리 프로젝트에서는 RESERVATION\_STATUS의 들어가는 값이 초기에는 '확정'이라는 단어를 사용하였다가 후반에 '진행중'으로 수정하여 다른 비즈니스 로직에 문제가 생겨 수정하는 경우가 생겼다.  
    이러한 사소한 데이터 불일치가 시스템 전체에 미치는 영향을 깨달았다.

#### 4\. 협업 및 커뮤니케이션 (Teamwork)

-   **Git/GitHub 활용:**
    -   리포지토리를 만듬과 동시에 노션에 팀원간의 깃 커밋 및 푸시 규칙을 세워 충돌을 최소화 하였다.  
        정형화된 문서 덕분에 잘 짜여진 규칙 내에서 기능마다 각자의 브랜치를 생성하여 작업한 후 PR을 올리기 전 메인을 PULL 받아 충돌을 로컬 환경에서 선제적으로 해결하여 메인 브랜치의 안정성을 유지했다.
-   **소통:**
    -   매일 진행 사항을 노션을 통해 확인할 수 있었고  
        해야할 일을 조장의 적절한 분배로 효율적으로 각자 맡은 페이지/기능을 구현할 수 있었다.

#### 5\. 아쉬운 점 및 개선 사항 (Future)

-   승인 관리 페이지를 AJAX를 활용하여 새로고침 없이 비동기로 처리하였으면 더 좋았을 것 같다는 아쉬움이 들었다.
-   각 탭과 전체 근태 현황 페이지에 페이징 처리를 하였으면 더 좋았을 것 같다는 아쉬움이 들었다.
-   만들다보니 컨트롤러에는 맞지 않은 로직들을 추가하였는데 MVC패턴에 맞게 현재 컨트롤러에 포함된 비즈니스 로직을 추후 리팩토링 과정을 통해 Service 레이어로 이관하여, 컨트롤러는 요청/응답 처리만 담당하도록 역할 분리를 명확히 할 계획이다.
