# 1. 2025년 KDT 세미 프로젝트 수행계획서

## I. 프로젝트 개요

| 구분 | 내용 |
| :--- | :--- |
| **프로젝트명** | **CalmDesk** |
| **주제** | 콜센터 직원을 위한 HR SaaS 플랫폼 |
| **기술분야** | SW (웹 애플리케이션) |
| **성과목표** | - 감정 노동자 보호법을 준수하는 실무형 HR SaaS 플랫폼 설계 및 구현<br>- 클라우드 환경(AWS)에서의 배포 및 운영을 통한 풀스택 개발 경험 축적 |
| **수행기간** | 2026. 01. 06. ~ 2026. 02. 27. |

### 프로젝트 소개 및 제안 배경

**1. 프로젝트 소개**
본 프로젝트의 목적은 콜센터 상담 인력이 겪는 고강도의 감정 노동과 정신적 부담을 체계적으로 관리하고 보호할 수 있는 **HR·건강 통합 관리 시스템**을 구축하는 데 있다. 단순히 인력 배치와 근태 관리 기능을 제공하는 기존 HR 시스템의 한계를 넘어, 상담 업무 중 발생하는 스트레스와 악성 민원 노출 빈도를 정량적으로 측정하고, 이를 실질적인 보호 체계로 연결하는 플랫폼을 구현하고자 한다.

**2. 제안 배경**
* **상담 인력의 감정 노동 심화와 번아웃 문제:** 상담원은 욕설, 고성, 성희롱 등 악성 민원에 상시 노출되어 있으며, 이는 서비스 품질 저하와 높은 이직률로 직결되는 악순환의 원인이 되고 있다.
* **기존 HR 시스템의 기능적 한계:** 기존 시스템은 출퇴근 기록, 스케줄링 등 행정적 편의성에만 초점이 맞춰져 있어, 감정 소모나 위험 상황 노출도 등 정서적 리스크를 관리하는 기능은 부재한 실정이다.
* **목표:** 상담사 보호 중심의 데이터 기반 HR 및 멘탈케어 SaaS를 제안하여 콜센터 조직의 생산성 향상과 인권 보호를 동시에 달성하고자 한다.

### 주요 기능

1.  **감정 노동 케어 & 쿨다운(Cool-down)**
    * 악성 민원 발생 시 '긴급 보호 휴식' 신청 및 관리자 즉시 알림
    * 매일의 감정 상태(스트레스 지수) 기록 및 위험군 자동 식별
2.  **유연 근무 & 워케이션**
    * 근태 인정 프로세스
    * 스트레스 지수와 연동된 힐링 프로그램 신청 및 1대1 상담 연계 시스템
3.  **직원 관리 및 실시간 모니터링**
    * 실시간 근무 상태(통화중, 대기중, 자리비움, 퇴근 등) 대시보드 시각화
4.  **스트레스 통계**
    * 매일 출/퇴근 시 입력한 기분 상태와 쿨다운 사용 횟수를 합산해 개인별 스트레스 지수 산출 후 통계화

### 기대효과 및 활용 분야

1.  직원 스트레스와 감정 데이터를 기반으로 과부하를 조기에 발견해 상담 직원 보호 제공.
2.  컨디션과 상담 난이도를 연계 분석해 상담원의 피로도 관리와 휴식을 제공하며 고객 만족도 회복.
3.  관리자 대시보드로 위험 직원을 한눈에 파악해 건강 지표 확보.

## II. 프로젝트 수행계획

### 1. 프로젝트 내용

#### 가. 주요 기능

| 구분 | 기능 | 설명 |
| :--- | :--- | :--- |
| **S/W** | **직원 관리** | 직원의 정보(등록/조회/수정)과 부서 팀 단위 관리, 근태 관리 기능 |
| **S/W** | **스트레스 리스크 분석** | 누적된 쿨다운 로그와 감정 상태 데이터를 정량적으로 분석, 스트레스 지수에 따른 위험군 자동 식별 |
| **S/W** | **업무(스케줄) 분석** | 직원들의 근태에 따른 업무 인원 파악, 직원의 쿨다운이나 자리비움, 근무중인 사람 파악 가능 |
| **S/W** | **맞춤형 치유 솔루션** | 고위험군 대상 강조 표시 및 결재, 근태 반영 |

* **프론트엔드:** React, Axios, Zustand
* **백엔드:** Spring Boot, Spring Web, Spring Data JPA(Hibernate), JWT
* **데이터베이스:** MySQL
* **인프라:** AWS (EC2, S3)
* **협업/형상 관리:** Git, GitHub
* **커뮤니케이션:** Slack, Notion(백업용), 카카오톡

#### 다. 필요 기자재

| 품목 | 활용계획 |
| :--- | :--- |
| **Figma** | UI/UX 디자인 기획을 위한 프로토타입 제작 |
| **Visual Studio Code** | 프론트엔드 페이지 제작 및 CSS 개발 |
| **IntelliJ IDEA** | 기능들을 위한 백엔드 로직 개발 |

### 2. 프로젝트 수행방법

#### 가. 프로젝트 추진일정

| 구분 | 추진내용 | 일정 (1주차 ~ 8주차) |
| :--- | :--- | :--- |
| **계획** | 서비스 주제 선정, 팀 역할 분담, 일정·협업도구 | 1주차 |
| **분석** | 요구사항 분석, 주요 유즈케이스 정의, 페이지 목록 정리 | 2주차 |
| **설계** | 화면 설계(UI/UX 와이어프레임), ERD/DB 설계, API 아키텍처 설계 | 3주차 |
| **개발** | 기본 환경 구축, 직원/근태/건강 기능 구현, 2차 기능 고도화 | 4주차 ~ 6주차 |
| **테스트** | 통합 테스트, 성능·안정성, 버그 수정 및 UI/UX 개선 | 7주차 |
| **종료** | AWS 배포(EC2·RDS·S3), 최종 문서·발표 자료 작성, 최종 발표 | 8주차 |

#### 나. 의사소통방법 및 협업 툴
* **협업툴:** Slack, 카카오톡
* **Slack:** 일정, 회의록, 공지사항, 이슈사항 기록 및 문제 공유.
* **Git:** 코드 실시간 확인, 코드 리뷰를 통한 개선 및 상호 이해.
* **회의:** 데일리 회의, 주간회의, 코드리뷰 진행.

#### 다. 프로젝트 Ground Rule (기본원칙)

**1. 근태 및 연락 (Time & Attendance)**
* **지각 및 결석 공지:** 오전 **08:30 이전**까지 단톡방 공유 필수.
* 단순 통보 지양, 사유와 도착 예정 시간 구체적 명시. (부재로 인한 업무 공백 보완 계획 포함)
    * *예시: "배탈로 인한 병원 진료 후 이동, 10:30 도착 예정."*

**2. 회의 루틴 (Meeting Routine)**
* **데일리 스크럼 (Daily Scrum)**
    * 시간: 매일 오전 **09:10** (수업일은 종료 직후 실시)
    * 방식: 스탠드업 미팅 (15분 이내), 내용 Slack 공유.
    * 공유 항목: 전일 완료 업무 / 금일 예정 업무 / 이슈 사항(Blocker).
* **주간 회의 (Weekly Meeting)**
    * 시간: 매주 월요일 **09:10**
    * 아젠다: 주간 마일스톤 점검, 지난주 회고(KPT), 우선순위 재조정.
* **코드 리뷰 (Code Review)**
    * 시간: 매주 **수요일 / 금요일 17:00**
    * 준비: 회의 시작 1시간 전까지 리뷰 대상 PR 생성 완료.
    * 리뷰 포인트: 설계 의도(Why), 기술적 타당성, 유지보수성, 확장성.

**3. 업무 수행 및 협업 (Work & Collab)**
* **유휴 인력(Idle Time) 최소화:** 할당 업무 조기 종료 시 즉시 후속 작업 수행.
    * 1단계: 이슈 트래커 내 잔여 To Do 확인. -> 2단계: 조장에게 추가 업무 문의.
* **이슈 공유 및 에스컬레이션:** 기술적 문제로 **30분 이상 지체 시** 개인 해결 중단하고 즉시 공유. (P0 등급 이슈로 간주하여 팀 공동 해결)

## III. 기대효과 및 활용분야

**1. 기대효과**
* **차별점:** 사후 처리 중심의 일반 HR과 달리, 스트레스와 감정 데이터를 실시간 분석하여 위험을 예측하고 선제적 보호 조치 제안.
* **활용 가치:** 피로도가 높은 직원에게 적시에 휴식/워케이션 제공으로 번아웃 예방 및 업무 몰입도 회복. 관리자는 데이터 중심의 케어로 전환 가능.

**2. 활용분야**
* **확장성:** 병원 원무과, 금융권 텔러, 공공기관 민원실 등 감정 노동 타 직군으로 확장 가능.
* **데이터 활용:** 감정노동자 보호법 준수 증빙 자료, 기업용 멘탈 헬스케어 모델의 기초 데이터로 활용.
