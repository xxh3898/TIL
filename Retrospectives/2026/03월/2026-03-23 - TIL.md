# TIL - 2026-03-23

## 오늘 배운 것

### 개념
- **IaC (Infrastructure as Code) 정립**
	- 맥, 윈도우, 운영 서버 등 OS 환경과 무관하게, `docker-compose.yml` 리스트 하나만 있으면 코드로 인프라를 100% 동일하게 통일할 수 있다. 도커가 "내 컴퓨터에선 되는데 왜 네 컴퓨터에선 안 돼?"라는 환경의 파편화 문제를 종결시킨다.
- **Docker Image vs Docker Compose의 차이점**
	- **Image:** MySQL, Valkey, Spring Boot 등 **'단일 역할'**을 수행하는 독립적인 패키징(설치 CD).
	- **Compose:** 조각난 여러 이미지들을 한 공간에 띄우고 서로 통신할 수 있게 연결해주는 **'레고 조립 설명서'**.
	- **왜 하나로 크게 안 굽나?** DB와 서버를 하나의 거대 이미지로 합치면, 스케일 아웃(서버만 여러 대 늘리기)이 불가능하고, 자바 코드 한 줄 수정할 때마다 무거운 DB 파일까지 매번 다시 구워야 하므로 실무에선 무조건 용도별로 컨테이너를 잘게 쪼갠다. (마이크로서비스 철학)
- **모던 CI/CD 파이프라인 흐름 (Docker 기반)**
	- **기존:** `.jar` 빌드 -> SCP로 EC2 전송 -> EC2에서 수동으로 `kill` 하고 `java -jar`로 재실행 (롤백 번거로움, 환경 종속적)
	- **Docker 기반:** `.jar` 빌드 -> `docker build` (이미지 굽기) -> `docker push` (Docker Hub에 업로드) -> EC2에서는 단지 `docker-compose pull && up -d` (3초 만에 배포 및 롤백 압도적으로 빠름)

### 기술
- **포트 충돌(Port Conflict) 및 Split Brain 현상 해결**
	- 스프링 부트 서버는 `3306` 포트로 접속해 테이블을 무사히 생성했으나, 워크벤치로 조회 시 "Table doesn't exist" 에러 발생.
	- 원인: 내 맥북 백그라운드에 Homebrew로 설치했던 `mysql@8.0`이 몰래 IPv4 3306 포트를 가로채고 있었고, 도커 컨테이너는 IPv6로 동작한 상황. (스프링 부트는 로컬 DB에, 워크벤치는 도커 빈 DB에 접속하는 이중 촌극 발생)
	- `lsof -i :3306` 및 `docker exec mysql mysql -e "SHOW TABLES;"` 명령어를 통해 진짜 실행 중인 프로세스를 추적하여 트러블슈팅 성공.
	- 해결: `brew services stop mysql@8.0` 및 `brew uninstall mysql`을 통해 완전히 로컬 MySQL 환경을 폐기하여 오직 도커만을 신뢰하는 깨끗한 로컬 개발 셋업 구축.
- **스프링 부트와 도커 컴포즈 연동 옵션 조정**
	- `spring-boot-docker-compose` 라이브러리의 과도한 친절함(커넥션 정보 자동 오버라이딩)을 방어.
	- 컨테이너에 `labels: org.springframework.boot.ignore: "true"`를 부여하여 스프링 부트가 생명주기(켜고 끄기)만 관리하고 DB 접속 설정(application-local.yaml의 타임존, DB 생성 조건 등)은 나의 세팅을 우선하도록 강제.

## 핵심 정리

- 데이터베이스 아키텍처의 정석: **영구 보관 및 데이터 무결성이 생명인 정형 데이터는 무거운 하드디스크 기반의 RDBMS(MySQL)**에, **휘발되어도 무방하며 초당 수천 번 읽혀 극한의 퍼포먼스가 필요한 데이터는 메모리(RAM) 기반의 In-Memory DB(Valkey)**로 완벽하게 역할을 분담하여 사용한다.
