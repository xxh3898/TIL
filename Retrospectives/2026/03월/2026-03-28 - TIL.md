# TIL - 2026-03-28


## 오늘 배운 것

### 개념
- **Stateless vs Stateful 인증**: JWT를 통해 서버가 세션 상태를 저장하지 않고 요청마다 전달받은 토큰만으로 인증을 수행하는 구조적 차이점 재확인
- **Refresh Token Lifecycle**: Access Token 만료 시 보안을 유지하며 사용자 경험을 개선하는 토큰 갱신 흐름 및 Redis의 역할 제고
- **Mermaid Gantt Chart**: 프로젝트 진행 상황을 가시화하기 위해 `done`, `active` 상태를 활용한 실시간 스케줄 관리 방법 익힘

### 기술
- **Spring Boot 3.4.1 (Spring Security 6.4) 필터 최적화**: 매 요청마다 DB를 조회하는 대신 토큰 클레임(Claim) 정보를 사용하여 인증 객체를 생성하는 최적화 기법 적용
- **JJWT 0.12.6 API**: 최신 버전의 JJWT 라이브러리를 사용한 토큰 생성(`builder()`), 파싱(`parser()`), 서명 검증(`verifyWith()`) 방식 숙달
- **Redis TTL (Time To Live)**: `StringRedisTemplate`을 활용해 만료 시간이 지나면 자동으로 데이터가 삭제되게끔 설정하여 효율적인 토큰 보관소 관리

---

## 코드 예제

```java
// Stateless 인증 성능 최적화 핵심 로직
public Authentication getAuthentication(String token) {
    Claims claims = extractAllClaims(token);
    String email = claims.getSubject();
    String role = claims.get("role", String.class);

    // DB 조회 없이 토큰 정보만으로 인증 객체(Principal) 생성
    List<GrantedAuthority> authorities = Collections.singletonList(new SimpleGrantedAuthority(role));
    User principal = new User(email, "", authorities);

    return new UsernamePasswordAuthenticationToken(principal, token, authorities);
}
```

---

## 핵심 정리

- **JWT의 장점을 살리려면 필터에서의 DB 조회를 최소화해야 한다.**
- **Redis는 Refresh Token 관리 시 성능(인메모리)과 보안(중앙 집중형 무효화) 모두를 만족시키는 최적의 저장소다.**
- **예외 상황(위변조, 경로 부재 등)에 대한 꼼꼼한 Filter-level 핸들링이 전체 시스템 안정성에 직결된다.**
