# TIL - 2026-03-30


## 오늘 배운 것

### 개념
- **User Enumeration Prevention**: 로그인 실패 시 "이메일이 틀렸습니다"와 "비밀번호가 틀렸습니다"를 구분하지 않고 동일한 401 에러를 반환하여, 공격자가 특정 이메일의 가입 여부를 알아내지 못하게 하는 보안 기법.
- **Refresh Token Rotation (RTR)**: Refresh Token을 1회용으로 사용하여 보안성을 높이는 전략. 특히 이미 사용된 토큰이 다시 들어올 경우 이를 '탈취'로 간주하고 해당 유저가 발급받은 모든 토큰을 무효화하는 **Family Revocation**의 중요성을 배움.
- **Gradle Task Input/Output Consistency**: Gradle 8.x 이상에서 특정 태스크의 결과물이 다른 태스크의 입력 소스 디렉토리에 직접 영향을 줄 때 발생하는 순환 의존성(Circular Dependency) 문제를 이해함.

### 기술
- **Redis 기반 Blacklisting**: 로그아웃 시 Access Token의 남은 만료 시간만큼 Redis에 `jti`를 키로 저장하여 필터 단계에서 재사용을 차단하는 방법.
- **Gradle Resources Exclusion**: `processResources` 태스크에서 특정 경로를 `exclude` 처리하여 빌드 시점에 생성되는 정적 자원(REST Docs)과의 충돌을 방지하는 기술.

## 코드 예제

### Gradle Circular Dependency 해결 (build.gradle)
```gradle
// REST Docs 생성 결과물을 resources 폴더로 복사할 때 발생하는 무한 루프 방지
tasks.named('processResources') {
    exclude('static/docs/**') // 빌드 타임에 생성되는 문서는 리소스 처리에서 제외
}

tasks.register('copyDocument', Copy) {
    dependsOn asciidoctor
    from file("build/docs/asciidoc")
    into file("src/main/resources/static/docs")
}
```

### Refresh Token 재사용 탐지 및 전면 무효화
```java
// 저장된 토큰과 요청된 토큰이 불일치할 경우 (이미 사용됨 = 탈취 의심)
if (!savedRefreshToken.getToken().equals(requestedToken)) {
    // 해당 유저의 모든 리프레시 토큰 삭제 (토큰 패밀리 전체 폐기)
    refreshTokenService.revokeAllByEmail(email);
    throw new CustomApiException("비정상적인 접근이 탐지되어 모든 세션이 강제 종료되었습니다.");
}
```

## 핵심 정리

- 보안은 "뚫리지 않는 문"을 만드는 것만큼 "이미 뚫렸을 때 피해를 최소화하는 로직(RTR, Revocation)"을 짜는 것이 결정적이다.
- Gradle 빌드 도구는 매우 강력하지만, 태스크 간의 입출력 관계가 명확하지 않으면 CI 환경에서 예기치 못한 실패를 야기할 수 있으므로 `exclude`나 `dependsOn`을 정교하게 설계해야 한다.
