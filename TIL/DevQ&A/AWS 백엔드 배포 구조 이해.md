## Topic
AWS 백엔드 배포 구조 이해

### 1. Why
- **실무**: 백엔드 서비스는 로컬에서 끝나지 않고 실제 서버에 배포되어야 한다. AWS 배포 구조를 이해해야 서버, DB, 파일 저장, 리버스 프록시, 환경 변수 관리가 어떻게 연결되는지 설명할 수 있다.
- **구조적 의미**: 배포 구조는 애플리케이션 코드 바깥의 실행 환경 설계다. `EC2`, `RDS`, `S3`, `Nginx`, 환경 변수는 각각 다른 역할을 가진 인프라 컴포넌트다.
- **면접 의도**: 면접에서는 각 서비스의 이름을 아는지보다, 실제 배포 흐름에서 무엇이 어떤 역할을 하는지 이해하는지 보려는 경우가 많다.

### 2. Core Concept
- **개념 정의**: `EC2`는 애플리케이션 서버를 실행하는 가상 머신이고, `RDS`는 관리형 관계형 데이터베이스다. `S3`는 파일과 정적 자산을 저장하는 오브젝트 스토리지이고, `Nginx`는 리버스 프록시, 정적 파일 서빙, SSL 종료, 로드 밸런싱 등에 사용된다.
- **동작 방식**: 일반적인 구조는 클라이언트 요청이 `Nginx`를 거쳐 `EC2`에서 실행 중인 Spring Boot 애플리케이션으로 들어오고, 애플리케이션은 `RDS`와 연결해 데이터를 읽고 쓴다. 파일 업로드나 정적 자원은 `S3`에 저장하고, 민감한 설정값은 코드에 넣지 않고 환경 변수나 비밀 관리 수단으로 주입한다.
- **장점 / 단점**: 역할이 분리된 구조는 운영과 확장에 유리하다. 하지만 네트워크, 보안 그룹, IAM 권한, 환경 변수 누락, 배포 순서 문제처럼 코드 밖에서 생기는 장애 포인트가 많아진다.
- **비교 포인트**: `EC2`는 서버 실행, `RDS`는 데이터 영속성, `S3`는 파일 저장, `Nginx`는 요청 진입과 트래픽 제어 역할을 맡는다. 모든 것을 한 서버에 몰아넣는 구조보다 운영은 복잡하지만 훨씬 현실적인 서비스 구조다.
- **사용 조건 / 주의점**: 환경 변수에는 DB 접속 정보, JWT secret, 외부 API key 같은 민감 정보를 두고 코드나 깃에 하드코딩하지 않아야 한다. 또 보안 그룹, IAM, 포트 개방 범위, 백업 정책까지 함께 설계해야 운영 리스크를 줄일 수 있다.

### 3. Interview Answer Version
- 30~40초 답변: AWS 백엔드 배포 구조를 단순하게 보면 `EC2`는 애플리케이션 서버, `RDS`는 데이터베이스, `S3`는 파일 저장소, `Nginx`는 요청을 받아 애플리케이션으로 전달하는 프록시 역할이라고 정리할 수 있습니다. 실제 배포 흐름은 보통 코드를 빌드해서 서버에 배포하고, `Nginx`가 외부 요청을 받고 내부의 Spring Boot 애플리케이션으로 넘기고, 애플리케이션은 `RDS`와 통신하고 파일은 `S3`에 저장하는 구조입니다. 이때 중요한 것은 환경 변수를 통해 민감한 설정을 분리하고, 인프라 역할을 명확히 나누는 것입니다.

### 4. Practical Tip
- 실제 적용 사례: EC2에는 애플리케이션만 실행하고, DB는 RDS로 분리하고, 이미지 파일은 S3로 분리하면 장애 범위를 나누기 쉽고 운영이 편해진다.
- 잘못 쓰면 생기는 문제: 모든 설정을 서버 안에 수동으로 넣거나 환경 변수를 코드에 하드코딩하면 서버 교체, 롤백, 보안 관리가 매우 어려워진다.
- 프로젝트 연결: 개인 프로젝트에서도 `Nginx -> Spring Boot on EC2 -> RDS`, 파일은 `S3`, 민감 정보는 환경 변수로 분리하는 구조를 이해하고 설명할 수 있으면 실무 감각을 보여주기 좋다.

### 5. 예상 꼬리 질문
1. `Nginx`를 두는 이유는 무엇인가요?
- 답변: `Nginx`는 외부 요청을 받아 SSL 종료, 정적 파일 처리, 리버스 프록시, 로드 밸런싱 같은 역할을 맡기기 좋습니다. 애플리케이션 서버를 직접 외부에 노출하는 것보다 운영상 더 유연하고 안전합니다.
2. `EC2` 대신 ECS나 Elastic Beanstalk를 쓰는 경우는 언제인가요?
- 답변: 컨테이너 기반 배포와 자동 스케일링, 운영 자동화를 더 강하게 가져가고 싶다면 ECS 같은 관리형 오케스트레이션이 유리할 수 있습니다. 반대로 학습용이나 제어가 중요한 작은 서비스는 EC2가 단순하고 이해하기 쉽습니다.
3. 환경 변수를 어디까지 두고, 무엇은 별도 비밀 관리로 분리해야 하나요?
- 답변: 일반 설정값은 환경 변수로도 충분하지만, DB 비밀번호나 API key처럼 민감하고 교체 주기가 있는 값은 AWS Secrets Manager나 Parameter Store 같은 별도 비밀 관리 도구가 더 안전합니다. 핵심은 코드와 이미지 안에 비밀을 굳히지 않는 것입니다.
