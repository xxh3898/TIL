### 1. 개념 (What is CI/CD?)

#### **CI (Continuous Integration) : 지속적 통합**
개발자가 작성한 코드를 중앙 저장소(GitHub 등)에 **자주** 통합하고, 그 과정에서 자동화된 **빌드**와 **테스트**가 수행되는 것을 의미합니다.
- **핵심**: "통합했을 때 코드가 깨지지 않는가?"를 검증합니다.
- **결과물**: 에러 없는 빌드 파일, 테스트 리포트.

#### **CD (Continuous Delivery / Deployment) : 지속적 제공 및 배포**
CI를 거친 코드를 실제 서버에 **자동으로 배포**하는 과정을 의미합니다.
- **Continuous Delivery**: 코드 수정이 완료되면 '배포 가능한 상태'를 유지하고, 실제 배포는 사람이 승인(버튼 클릭 등)하여 진행합니다.
- **Continuous Deployment**: 승인 과정 없이 테스트를 통과한 코드가 운영 서버까지 **자동**으로 배포됩니다.
- **핵심**: "사용자에게 얼마나 빠르고 안전하게 기능을 전달할 수 있는가?"

---

### 2. 왜 사용하는가? (The Reason)

사용하게 된 가장 큰 이유는 **'휴먼 에러 방지'**와 **'배포 공포증(Deployment Anxiety) 해소'**입니다.

1.  **환경의 일관성**: "내 컴퓨터에선 되는데 서버에선 안 돼요"라는 상황을 막기 위해, 아무것도 없는 '클린한 서버(CI 서버)'에서 테스트를 돌려 환경 의존성을 제거합니다.
2.  **빠른 피드백**: 코드를 올리자마자 1~5분 안에 "테스트가 깨졌습니다"라는 알림을 받으면, 기억이 생생할 때 바로 고칠 수 있습니다.
3.  **병합 지옥(Merge Hell) 탈출**: 코드를 수개월씩 쟁여놓다 한꺼번에 합치면 충돌이 너무 많습니다. CI는 매일 조금씩 합치는 문화를 유도합니다.

---

### 3. 장점과 단점

#### **장점 (Pros)**
-   **코드 품질 보장**: 자동 테스트 덕분에 기존 기능을 망가뜨리는(Regression) 실수를 사전에 차단합니다.
-   **배포 속도**: 수동으로 서버 접속(SSH)하고 빌드하고 재시작하는 모든 노가다를 자동화하므로 훨씬 빠릅니다.
-   **포트폴리오 신뢰도**: 앞서 말씀드린 대로 "이 프로젝트는 검증된 코드다"라는 보증수표가 됩니다.

#### **단점 (Cons)**
-   **초기 설정 비용**: 설정파일을 작성하고 서버를 연결하는 초기 노력이 필요합니다.
-   **리소스 비용**: 빌드를 돌리는 서버(GitHub Actions runner 등) 비용이 발생할 수 있습니다 (프리티어라면 상관없지만 기업 단위에선 큼).
-   **파이프라인 관리**: 툴 버전이 바뀌거나 설정이 꼬이면 배포 자체가 막히는 일이 생겨, 이 시스템 자체를 관리하는 공수가 듭니다.

---

### 4. 실제 프로젝트(Cubing Hub) 적용


1.  **CI 단계 (`./gradlew test`)**:
    -   GitHub Actions가 Ubuntu 서버를 빌려옵니다.
    -   푸시한 `backend` 코드를 가져와서 DB(Testcontainers)를 띄워 테스트를 돌립니다.
    -   **목적**: "푸시한 코드가 기존 기능을 망가뜨리지 않았나?" 확인.

2.  **CD 단계:
    -   CI가 성공하면 AWS EC2에 접속해 이미지를 새로 받아서 띄웁니다.
    -   **목적**: "유저들이 접속하는 실제 서버에 반영".

---

**요약하자면:**
CI는 **"합칠 때 안 깨지게 하는 것"**, CD는 **"배포를 손 안 대고 하는 것"**입니다. 이 두 개를 합쳐서 **'파이프라인(Pipeline)'**이라고 부릅니다.
