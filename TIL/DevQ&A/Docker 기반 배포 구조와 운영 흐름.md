## Topic
Docker 기반 배포 구조와 운영 흐름

### 1. Why
- **실무**: 로컬에서는 되는데 서버에서는 안 되는 문제를 줄이기 위해 Docker를 많이 쓴다. 빌드 결과물만 배포하는 것보다 실행 환경까지 패키징하면 운영 일관성이 좋아진다.
- **구조적 의미**: Docker는 애플리케이션과 실행 환경을 함께 묶는 배포 표준화 도구다. 즉 배포를 "파일 복사"가 아니라 "컨테이너 실행" 문제로 바꾼다.
- **면접 의도**: 면접에서는 Docker가 무엇인지, 이미지 / 컨테이너 / Dockerfile이 어떻게 연결되는지, 왜 필요한지, 실제 배포 흐름과 운영 환경 일관성을 설명할 수 있는지 본다.

### 2. Core Concept
- **개념 정의**: Docker는 애플리케이션을 컨테이너 단위로 실행하는 플랫폼이다. 이미지는 실행 가능한 애플리케이션 패키지이고, 컨테이너는 그 이미지를 실제로 실행한 인스턴스다. `Dockerfile`은 이미지를 어떻게 만들지 정의하는 스크립트다.
- **동작 방식**: 코드와 빌드 결과를 기준으로 `Dockerfile`에서 이미지를 만들고, 그 이미지를 레지스트리에 저장한 뒤 서버에서 pull 받아 컨테이너로 실행한다. 환경 변수와 포트, 볼륨, 네트워크 설정을 분리하면 같은 이미지를 여러 환경에서 재사용할 수 있다.
- **장점 / 단점**: Docker의 가장 큰 장점은 운영 환경 일관성과 배포 재현성이다. 반면 이미지 관리, 레이어 최적화, 볼륨/네트워크 설정, 로그 관리 같은 운영 복잡도가 추가된다.
- **비교 포인트**: 이미지가 "설치 파일"이라면 컨테이너는 "실행 중인 프로세스"에 가깝다. 단순 JAR 배포보다 Docker 배포는 환경 차이를 줄이기 쉽지만, Docker 자체를 이해해야 운영 문제를 해결할 수 있다.
- **사용 조건 / 주의점**: `Dockerfile`에는 불필요한 파일을 넣지 않고, 비밀값을 이미지에 굳히지 말아야 한다. 운영 환경 일관성을 얻으려면 로컬, 테스트, 운영에서 가능한 한 같은 이미지와 비슷한 실행 방식을 유지하는 것이 중요하다.

### 3. Interview Answer Version
- 30~40초 답변: Docker는 애플리케이션과 실행 환경을 함께 패키징해서 컨테이너로 실행하게 해 주는 도구입니다. `Dockerfile`로 이미지를 만들고, 그 이미지를 서버에서 실행하면 컨테이너가 됩니다. 이렇게 하면 로컬과 운영 서버의 자바 버전이나 OS 환경 차이로 생기는 문제를 크게 줄일 수 있어서 운영 환경 일관성에 유리합니다. 실제 배포는 보통 이미지를 빌드하고 레지스트리에 올린 뒤, 서버에서 pull 받아 새 컨테이너를 실행하는 흐름으로 진행됩니다.

### 4. Practical Tip
- 실제 적용 사례: Spring Boot 애플리케이션을 멀티 스테이지 빌드로 이미지화하고, 서버에서는 `docker pull` 후 기존 컨테이너를 내리고 새 컨테이너를 띄우는 방식이 기본적인 운영 흐름이다.
- 잘못 쓰면 생기는 문제: 이미지에 `.env`나 secret을 넣어 버리면 보안 문제가 생기고, 로컬과 운영에서 서로 다른 실행 방식을 쓰면 Docker를 써도 환경 차이 문제가 남는다.
- 프로젝트 연결: 개인 프로젝트에서도 Docker로 백엔드와 DB를 함께 띄워 보면 배포 구조, 포트 매핑, 환경 변수 주입, 로그 확인 흐름을 한 번에 이해하기 좋다.

### 5. 예상 꼬리 질문
1. 이미지와 컨테이너는 정확히 어떻게 다른가요?
- 답변: 이미지는 실행에 필요한 파일과 설정을 담은 불변 템플릿이고, 컨테이너는 그 이미지를 실제로 실행한 인스턴스입니다. 같은 이미지로 여러 컨테이너를 띄울 수 있다는 점이 핵심 차이입니다.
2. `Dockerfile` 최적화에서 가장 먼저 신경 쓸 포인트는 무엇인가요?
- 답변: 우선 이미지 크기를 줄이고 빌드 캐시가 잘 활용되도록 레이어를 설계하는 것이 중요합니다. 동시에 비밀값을 이미지에 넣지 않고 멀티 스테이지 빌드로 불필요한 산출물을 제거하는 습관이 필요합니다.
3. Docker를 써도 운영 환경 문제가 완전히 사라지지 않는 이유는 무엇인가요?
- 답변: Docker는 애플리케이션 실행 환경 차이를 줄여 주지만, 네트워크, 볼륨, OS 커널, 외부 인프라 설정 차이까지 전부 없애 주지는 않습니다. 결국 컨테이너 밖의 운영 환경 설계도 함께 맞아야 합니다.
