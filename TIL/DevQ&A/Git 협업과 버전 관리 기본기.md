## Topic
Git 협업과 버전 관리 기본기

### 1. Why
- **실무**: Git은 단순 백업 도구가 아니라 협업 기록과 변경 이력 관리 도구다. `commit / push / pull`, 브랜치 전략, 충돌 해결 방식을 이해해야 팀 단위 개발이 안정적으로 굴러간다.
- **구조적 의미**: 버전 관리는 코드 변경을 시간 축으로 관리하는 구조다. 누가 언제 왜 바꿨는지 추적 가능해야 품질과 협업 속도가 올라간다.
- **면접 의도**: 면접에서는 Git 명령어 암기보다 `merge / rebase`, `reset / revert`, 브랜치 전략, 좋은 커밋 메시지를 실무 맥락에서 구분해서 설명할 수 있는지 본다.

### 2. Core Concept
- **개념 정의**: `commit`은 변경 이력을 로컬 저장소에 기록하는 작업이고, `push`는 원격 저장소로 업로드, `pull`은 원격 변경을 가져와 현재 브랜치에 반영하는 작업이다. 브랜치는 독립적인 작업 흐름을 분리하는 수단이다.
- **동작 방식**: 협업에서는 보통 기능별 브랜치에서 작업하고, 리뷰 후 메인 브랜치로 통합한다. 통합 방식은 `merge`로 분기 흔적을 남길 수도 있고, `rebase`로 커밋 히스토리를 선형으로 정리할 수도 있다. `reset`은 커밋 이력을 되돌리는 데 강력하지만 위험할 수 있고, `revert`는 기존 이력을 보존한 채 반대 커밋을 추가해 안전하게 취소한다.
- **장점 / 단점**: Git은 강력한 이력 관리와 병렬 작업에 유리하지만, 명령 의미를 잘못 이해하면 이력 손실이나 충돌이 생길 수 있다. `merge`는 안전하고 이력 보존이 명확하지만 히스토리가 복잡해질 수 있고, `rebase`는 깔끔하지만 이미 공유된 이력을 바꾸면 위험하다.
- **비교 포인트**: 공유된 이력을 취소할 때는 보통 `revert`, 아직 공유되지 않은 로컬 정리에는 `reset`이 더 적합하다. 브랜치 전략은 Git Flow처럼 무겁게 갈 수도 있고, trunk-based처럼 단순하게 갈 수도 있지만 핵심은 팀 규모와 배포 빈도에 맞는 규칙을 만드는 것이다.
- **사용 조건 / 주의점**: 좋은 커밋 메시지는 무엇을 왜 바꿨는지 드러나야 한다. 한 커밋에 서로 다른 목적의 변경을 섞지 말고, `rebase`는 이미 팀과 공유된 브랜치에 무심코 사용하지 않는 것이 안전하다.

### 3. Interview Answer Version
- 30~40초 답변: Git 협업의 핵심은 변경 이력을 안전하게 쪼개고 공유하는 것이라고 생각합니다. `commit`으로 의미 있는 변경 단위를 만들고, `push / pull`로 원격과 동기화하며, 기능별 브랜치에서 작업한 뒤 리뷰 후 메인 브랜치로 통합하는 흐름이 기본입니다. 통합할 때는 히스토리 보존이 중요하면 `merge`, 선형 정리가 필요하면 `rebase`를 선택할 수 있고, 이미 공유된 이력을 취소할 때는 `reset`보다 `revert`가 안전합니다. 또 좋은 커밋 메시지는 나중에 이력을 읽을 때 바로 의도를 알 수 있게 쓰는 것이 중요합니다.

### 4. Practical Tip
- 실제 적용 사례: 기능 단위로 브랜치를 따고, 리뷰 가능한 크기로 커밋을 나눈 뒤 PR에서 `merge` 또는 스쿼시 머지를 사용하는 방식이 가장 일반적이다.
- 잘못 쓰면 생기는 문제: 여러 목적의 변경을 한 커밋에 섞거나, 공유된 브랜치에 `rebase`를 잘못 적용하면 충돌과 이력 혼란이 커진다.
- 프로젝트 연결: 개인 프로젝트라도 커밋 메시지를 일관되게 쓰고 기능 단위 브랜치를 분리해 두면 추후 회고, 포트폴리오 정리, 버그 추적이 훨씬 쉬워진다.

### 5. 예상 꼬리 질문
1. `merge`와 `rebase`는 실무에서 어떤 기준으로 선택하나요?
- 답변: 이미 여러 사람이 보고 있는 공유 브랜치라면 히스토리 보존이 명확한 `merge`가 더 안전합니다. 반면 로컬 작업 브랜치를 PR 전에 깔끔하게 정리할 때는 `rebase`가 유용합니다.
2. `reset`보다 `revert`가 더 안전한 이유는 무엇인가요?
- 답변: `revert`는 기존 이력을 지우지 않고 반대 변경을 새 커밋으로 남기기 때문에 협업 중인 원격 브랜치에서도 안전하게 쓸 수 있습니다. `reset`은 이력 자체를 움직일 수 있어서 공유된 브랜치에서는 위험합니다.
3. 좋은 커밋 메시지를 작성할 때 가장 중요한 기준은 무엇인가요?
- 답변: 나중에 커밋 로그만 봐도 무엇을 왜 바꿨는지 이해할 수 있어야 합니다. 그래서 결과보다 의도와 변경 범위를 드러내는 것이 중요하고, 서로 다른 목적의 변경을 한 커밋에 섞지 않는 것이 전제입니다.
