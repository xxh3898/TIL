## Topic
JPA 핵심 동작 원리

### 1. Why
- **실무**: JPA는 편해 보이지만 내부 동작을 모르고 쓰면 왜 쿼리가 나가는지, 왜 변경이 반영되는지, 왜 같은 객체를 다시 조회했는데 SQL이 안 나가는지 설명하지 못한다.
- **구조적 의미**: JPA는 단순 SQL 실행기가 아니라 ORM과 영속성 컨텍스트를 기반으로 객체 상태를 관리하는 프레임워크다. 그래서 내부 메커니즘을 이해해야 설계와 성능 판단이 가능하다.
- **면접 의도**: 면접에서는 ORM, 영속성 컨텍스트, 1차 캐시, dirty checking, `flush / commit`, entity lifecycle을 설명할 수 있는지 자주 묻는다.

### 2. Core Concept
- **개념 정의**: ORM은 객체와 관계형 DB 테이블을 매핑해 객체 중심으로 데이터를 다루도록 돕는 방식이다. JPA의 핵심은 영속성 컨텍스트라는 1차 메모리 공간에서 Entity 상태를 관리하는 데 있다.
- **동작 방식**: Entity는 `비영속 -> 영속 -> 준영속 -> 삭제` 같은 entity lifecycle을 가진다. 영속 상태가 되면 영속성 컨텍스트가 Entity를 관리하고, 같은 트랜잭션 안에서는 1차 캐시 덕분에 같은 PK 조회 시 DB 대신 메모리 객체를 재사용할 수 있다. 트랜잭션 종료 전 `flush` 시점에 변경 사항을 SQL로 반영하고, dirty checking을 통해 개발자가 `update` SQL을 직접 호출하지 않아도 바뀐 필드를 추적한다. `commit`은 DB 트랜잭션을 실제 확정하는 단계다.
- **장점 / 단점**: 장점은 객체 지향적으로 데이터를 다루고 반복 SQL을 줄일 수 있다는 점이다. 단점은 내부 상태 관리가 보이지 않아 학습 없이 쓰면 오히려 동작을 오해하기 쉽다는 점이다.
- **비교 포인트**: `flush`는 변경 내용을 DB에 반영하는 시점이고, `commit`은 트랜잭션을 확정하는 시점이다. 영속성 컨텍스트의 1차 캐시는 Redis 같은 애플리케이션 캐시와 목적이 다르고, 동일 트랜잭션 내부에서 Entity 일관성을 맞추기 위한 메커니즘이다.
- **사용 조건 / 주의점**: JPA를 제대로 쓰려면 영속 상태와 준영속 상태를 구분하고, Entity 변경이 언제 SQL로 반영되는지 이해해야 한다. 영속성 컨텍스트를 무시하고 무조건 SQL처럼 생각하면 예상과 다른 쿼리 실행 때문에 혼란이 생긴다.

### 3. Interview Answer Version
- 30~40초 답변: JPA의 핵심은 ORM과 영속성 컨텍스트라고 생각합니다. JPA는 객체와 테이블을 매핑해서 객체 중심으로 데이터를 다루게 해 주고, 영속성 컨텍스트가 Entity를 관리하면서 1차 캐시와 dirty checking을 제공합니다. 그래서 같은 트랜잭션 안에서는 같은 엔티티를 다시 조회해도 DB 대신 메모리 객체를 재사용할 수 있고, 영속 상태의 엔티티를 수정하면 `flush` 시점에 변경 사항이 SQL로 반영됩니다. 또 `commit`은 그 트랜잭션을 최종 확정하는 단계라서 `flush`와 구분해서 이해하는 게 중요합니다.

### 4. Practical Tip
- 실제 적용 사례: 게시글을 조회한 뒤 제목만 바꾸고 트랜잭션이 끝나면 별도의 `update` 호출 없이 반영되는 것이 바로 dirty checking 동작이다.
- 잘못 쓰면 생기는 문제: 영속성 컨텍스트를 이해하지 못하면 왜 같은 요청에서 SQL이 안 나가는지, 왜 수정했는데 저장 코드 없이 반영되는지, 왜 벌크 연산 후 값이 꼬이는지 설명하기 어려워진다.
- 프로젝트 연결: Spring Data JPA를 쓸 때도 내부적으로는 영속성 컨텍스트와 Entity 생명주기를 따르므로, Repository 메서드만 외우기보다 JPA 동작 원리를 알고 쓰는 것이 중요하다.

### 5. 예상 꼬리 질문
1. `flush`와 `commit`은 정확히 어떻게 다른가요?
- 답변: `flush`는 영속성 컨텍스트의 변경 내용을 DB에 SQL로 반영하는 동기화 단계입니다. `commit`은 그 트랜잭션 자체를 최종 확정하는 단계라, `flush`가 일어나도 `commit` 전에 롤백되면 반영이 취소될 수 있습니다.
2. 준영속 상태가 되면 어떤 문제가 생길 수 있나요?
- 답변: 준영속 상태의 Entity는 더 이상 변경 감지를 받지 않기 때문에 값을 바꿔도 자동 업데이트되지 않습니다. 또 지연 로딩 프록시를 초기화하지 못해 `LazyInitializationException` 같은 문제가 생길 수 있습니다.
3. 1차 캐시와 2차 캐시는 무엇이 다른가요?
- 답변: 1차 캐시는 영속성 컨텍스트 단위로 항상 존재하는 캐시이고, 같은 트랜잭션 안에서 Entity 동일성을 보장하는 역할이 큽니다. 2차 캐시는 애플리케이션 전역에서 공유하는 선택적 캐시로, 설정과 전략에 따라 별도로 사용합니다.
