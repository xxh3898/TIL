## Topic
String, 불변성, equals hashCode, 객체 비교 정리

### 1. Why
- **실무**: 문자열 비교, 객체 동등성 판단, `HashMap` key 사용은 자바에서 매우 자주 나오고, 여기서 실수하면 버그가 눈에 잘 안 보인다. 특히 \==와 `equals`, `hashCode` 계약은 기본기이면서도 실전 영향이 크다.
- **구조적 의미**: `String`과 불변성, 동등성 비교는 자바 객체 모델의 핵심이다. 객체를 "같다"고 볼 기준을 어떻게 정의하느냐가 컬렉션 동작과 도메인 설계에 직접 연결된다.
- **면접 의도**: 면접에서는 `String pool`, immutable 객체, \==와 `equals`, `hashCode` 관계를 함께 묶어 설명할 수 있는지 자주 확인한다.

### 2. Core Concept
- **개념 정의**: `String`은 대표적인 immutable 객체로, 한 번 생성되면 내부 값이 바뀌지 않는다. `String pool`은 동일한 문자열 리터럴을 재사용하기 위한 메모리 영역 개념이다. \==는 참조 비교이고, `equals`는 논리적 동등성 비교다. `hashCode`는 해시 기반 자료구조에서 빠른 탐색을 위한 정수 값이다.
- **동작 방식**: 문자열 리터럴은 `String pool`에 저장되어 같은 리터럴이면 같은 참조를 재사용할 수 있다. 하지만 `new String("abc")`처럼 만들면 별도 객체가 생길 수 있으므로 \==로 내용 비교를 하면 안 된다. 내용 비교는 `equals`를 사용해야 하고, `equals`를 오버라이드하면 같은 객체는 같은 `hashCode`를 가져야 한다.
- **장점 / 단점**: immutable 객체는 예기치 않은 변경이 없어 안전하고 캐싱, 공유, 스레드 안정성 측면에서 유리하다. 반면 변경이 잦은 상황에서는 새 객체 생성 비용이 누적될 수 있어 `StringBuilder` 같은 대안이 필요할 수 있다.
- **비교 포인트**: \==는 메모리 주소가 같은지, `equals`는 의미상 같은지를 비교한다. `String`은 내용 비교가 중요하므로 거의 항상 `equals`를 써야 한다. `hashCode`는 빠른 탐색을 위한 값이지 동등성 자체를 완전히 보장하는 값은 아니고, 최종 비교는 `equals`가 담당한다.
- **사용 조건 / 주의점**: 문자열 비교에서 \==를 쓰면 버그가 생기기 쉽다. 또 `equals`만 재정의하고 `hashCode`를 같이 재정의하지 않으면 해시 기반 컬렉션에서 예상과 다른 동작이 발생한다. 값 객체는 가능하면 immutable하게 설계하는 편이 안전하다.

### 3. Interview Answer Version
- 30~40초 답변: 자바에서 `String`은 대표적인 immutable 객체라 한 번 생성되면 값이 바뀌지 않고, 문자열 리터럴은 `String pool`에서 재사용될 수 있습니다. 객체 비교에서는 \==가 참조 비교이고 `equals`가 논리적 동등성 비교이기 때문에, 문자열 내용 비교는 보통 `equals`를 써야 합니다. 또 `equals`를 재정의하면 같은 객체는 같은 `hashCode`를 가져야 한다는 계약이 있어서, 둘은 항상 같이 고려해야 합니다. 이 개념이 `HashMap`, `HashSet` 같은 컬렉션 동작과도 직접 연결됩니다.

### 4. Practical Tip
- 실제 적용 사례: 회원 이메일, 주문 번호 같은 값 객체는 immutable하게 두고, 비교 기준이 명확한 도메인 객체는 `equals`와 `hashCode`를 함께 재정의하면 컬렉션 사용과 테스트가 안정적이다.
- 잘못 쓰면 생기는 문제: 문자열 비교에 \==를 쓰거나, `equals`만 오버라이드하고 `hashCode`를 재정의하지 않으면 중복 제거, 조회, 비교 로직이 예상과 다르게 동작한다.
- 프로젝트 연결: JPA Entity와 값 객체를 설계할 때도 어떤 필드를 기준으로 동등성을 볼지 신중해야 하고, 특히 해시 기반 컬렉션에 넣는 객체는 `equals/hashCode` 계약을 반드시 지켜야 한다.

### 5. 예상 꼬리 질문
1. `String`이 immutable하게 설계된 이유는 무엇인가요?
- 답변: 문자열은 매우 자주 공유되고 `String pool`, 해시 캐싱, 보안 파라미터 처리에 많이 쓰이기 때문에 바뀌지 않는 편이 훨씬 안전합니다. immutable하면 여러 곳에서 같은 객체를 공유해도 예기치 않은 변경이 없어서 성능과 안정성 모두에 도움이 됩니다.
2. `hashCode`가 같으면 두 객체는 반드시 같은가요?
- 답변: 아닙니다. 서로 다른 객체가 같은 해시 값을 가질 수 있는데 이를 해시 충돌이라고 합니다. 그래서 해시 기반 컬렉션도 `hashCode`로 후보를 좁힌 뒤 최종 비교는 `equals`로 합니다.
3. Entity에서 `equals`와 `hashCode`를 재정의할 때 주의할 점은 무엇인가요?
- 답변: JPA Entity는 식별자 생성 시점과 프록시 이슈가 있어서 단순히 모든 필드를 기준으로 비교하면 위험할 수 있습니다. 보통 변경되지 않는 비즈니스 키를 쓰거나, 식별자 기반 비교를 매우 신중하게 설계해야 합니다.
