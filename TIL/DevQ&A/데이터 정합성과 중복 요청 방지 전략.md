## Topic
데이터 정합성과 중복 요청 방지 전략

### 1. Why
- **실무**: 결제, 주문 생성, 좋아요, 게시글 등록처럼 같은 요청이 두 번 들어오면 데이터가 깨질 수 있는 기능은 많다. 네트워크 재시도, 사용자 중복 클릭, 분산 환경의 동시 요청 때문에 중복 요청 방지는 필수다.
- **구조적 의미**: 데이터 정합성은 단순히 DB에 저장되느냐가 아니라, 같은 의도가 여러 번 들어와도 결과가 의도대로 한 번만 반영되도록 시스템 경계를 설계하는 문제다.
- **면접 의도**: 면접에서는 멱등성과 중복 제출 방지를 말로만 아는지, 트랜잭션 경계와 분산 환경 주의점까지 연결해 설명할 수 있는지 확인한다.

### 2. Core Concept
- **개념 정의**: 데이터 정합성은 시스템 상태가 비즈니스 규칙에 맞게 유지되는 상태를 말한다. 멱등성은 같은 요청을 여러 번 보내도 결과가 한 번 수행한 것과 같아야 하는 성질이고, 중복 제출 방지는 실제로 같은 작업이 중복 실행되지 않도록 막는 전략이다.
- **동작 방식**: 보통 요청별 고유 키를 두거나, DB unique 제약조건, 상태 체크, 락, 트랜잭션 경계를 조합해 중복 반영을 막는다. 예를 들어 주문 생성 요청에 `idempotency key`를 두면 같은 키의 재요청은 기존 결과를 반환하도록 설계할 수 있다.
- **장점 / 단점**: 멱등성과 중복 방지 전략을 넣으면 데이터 신뢰도가 올라가고 재시도에 강해진다. 반면 구현이 복잡해지고, 키 관리나 락 전략을 잘못 설계하면 성능 저하나 오탐 차단이 생길 수 있다.
- **비교 포인트**: HTTP 메서드 차원의 멱등성은 프로토콜 의미이고, 비즈니스 차원의 멱등성은 실제 시스템 상태를 중복 없이 유지하는 설계다. 단순 애플리케이션 서버 메모리 체크는 단일 서버에서는 동작할 수 있지만 분산 환경에서는 신뢰하기 어렵다.
- **사용 조건 / 주의점**: 트랜잭션 경계 안에서 최종 정합성 검증이 이뤄져야 하고, 애플리케이션 레벨 체크만 믿지 말고 DB 제약조건이나 원자적 갱신도 함께 고려해야 한다. 분산 환경에서는 여러 인스턴스가 동시에 같은 요청을 처리할 수 있으므로 Redis 락, 메시지 중복 제거, unique index 같은 보강 장치가 필요하다.

### 3. Interview Answer Version
- 30~40초 답변: 데이터 정합성과 중복 요청 방지는 결국 같은 작업이 여러 번 들어와도 시스템 상태를 올바르게 유지하는 설계라고 생각합니다. 여기서 핵심은 멱등성, 트랜잭션 경계, 그리고 분산 환경 대응입니다. 사용자의 중복 클릭이나 네트워크 재시도는 흔하기 때문에 주문이나 결제 같은 기능에는 `idempotency key`, unique 제약조건, 상태 체크를 함께 두는 편이 안전합니다. 또 애플리케이션에서 한 번 검사했다고 끝나는 게 아니라, 트랜잭션 안에서 최종 검증하고 분산 환경이면 락이나 중복 제거 전략까지 같이 봐야 합니다.

### 4. Practical Tip
- 실제 적용 사례: 결제 승인이나 주문 생성 API는 요청마다 고유 키를 발급하고, 같은 키로 다시 오면 새로 생성하지 않고 기존 결과를 반환하는 방식이 많이 쓰인다.
- 잘못 쓰면 생기는 문제: "이미 주문이 없으면 생성" 같은 로직만 애플리케이션에서 검사하고 DB 제약조건이 없으면 동시 요청 두 개가 동시에 통과해 중복 데이터가 생길 수 있다.
- 프로젝트 연결: Spring + JPA 프로젝트에서는 `@Transactional` 경계 안에서 상태 검증을 수행하고, 필요한 경우 unique index, 낙관적 락, Redis 기반 중복 요청 키 저장을 함께 조합하는 것이 현실적이다.

### 5. 예상 꼬리 질문
1. HTTP의 멱등성과 비즈니스 멱등성은 무엇이 다른가요?
- 답변: HTTP 멱등성은 메서드 의미 차원에서 같은 요청을 여러 번 보내도 서버 상태가 같아야 한다는 프로토콜적 개념입니다. 비즈니스 멱등성은 결제나 주문처럼 실제 도메인 상태가 중복 반영되지 않도록 시스템이 보장해야 하는 더 실무적인 개념입니다.
2. 분산 환경에서 중복 요청 방지는 어떤 방식으로 구현할 수 있나요?
- 답변: `idempotency key` 저장, Redis 분산 락, DB unique 제약조건, 메시지 중복 제거 같은 전략을 조합하는 경우가 많습니다. 핵심은 애플리케이션 서버 여러 대가 동시에 같은 요청을 받아도 최종 반영은 한 번만 되도록 만드는 것입니다.
3. unique 제약조건만으로 중복 처리 방지가 충분한가요?
- 답변: unique 제약조건은 가장 강한 마지막 방어선으로 매우 중요하지만, 그것만으로 사용자 경험과 재시도 흐름까지 해결되지는 않습니다. 보통 애플리케이션 레벨의 idempotency 처리와 함께 써야 중복 실패를 더 자연스럽게 제어할 수 있습니다.
