## Topic
도메인 중심 패키지 구조와 프로젝트 구조 설계 기준

### 1. Why
- **실무**: 기능이 늘어나면 계층형 패키지는 관련 코드가 `controller`, `service`, `repository`에 흩어져 변경 범위를 따라가기 어려워진다. 도메인 중심 구조는 같은 기능의 코드를 한곳에 묶어 변경 영향도를 줄인다.
- **구조적 의미**: 패키지 구조는 단순 폴더 정리가 아니라 책임 경계 설계다. 어떤 기준으로 패키지를 나누느냐에 따라 결합도, 재사용성, 팀 협업 방식이 달라진다.
- **면접 의도**: 면접에서는 "왜 그런 구조를 선택했는가", "계층형 vs 도메인형을 언제 다르게 쓰는가", "유지보수성 관점에서 어떤 판단을 했는가"를 확인하려는 경우가 많다.

### 2. Core Concept
- **개념 정의**: 계층형 구조는 `controller`, `service`, `repository`처럼 기술 계층 기준으로 묶는 방식이고, 도메인형 구조는 `member`, `record`, `order`처럼 비즈니스 도메인 기준으로 묶은 뒤 그 안에 계층을 둔다.
- **동작 방식**: 계층형에서는 모든 요청이 전역 계층 패키지를 오가고, 도메인형에서는 특정 요청이 해당 도메인 패키지 안에서 `Controller -> Service -> Repository` 흐름으로 닫히는 경우가 많다. 패키지 분리 기준은 보통 도메인 경계, 변경 이유, 트랜잭션 경계, 팀 소유권을 본다.
- **장점 / 단점**: 계층형은 시작이 쉽고 구조가 단순하지만 기능이 많아질수록 관련 코드가 퍼진다. 도메인형은 응집도가 높고 유지보수성이 좋지만, 공통 영역 설계를 잘못하면 중복과 경계 혼란이 생길 수 있다.
- **비교 포인트**: 작은 CRUD 프로젝트나 학습 단계에서는 계층형도 충분하다. 반면 기능 수가 많고 도메인별 수정이 자주 일어나는 서비스는 도메인형이 유리하다. 실무에서는 최상위는 도메인형, 내부는 계층형으로 섞는 하이브리드 구조를 많이 쓴다.
- **사용 조건 / 주의점**: 무조건 세분화하지 말고 실제로 함께 변경되는 코드끼리 묶어야 한다. `common`, `util` 패키지를 과하게 키우면 다시 결합도가 올라가고, 도메인 간 순환 참조가 생기지 않도록 의존 방향을 통제해야 한다.

### 3. Interview Answer Version
- 30~40초 답변: 패키지 구조는 보기 좋게 정리하는 문제가 아니라 변경을 어디서 흡수할지 결정하는 구조 설계라고 생각합니다. 작은 프로젝트는 계층형이 단순해서 빠르게 시작할 수 있지만, 기능이 늘어나면 관련 코드가 여러 계층에 흩어져 추적이 어려워집니다. 그래서 실무에서는 보통 도메인 기준으로 상위 패키지를 나누고, 그 안에서 `Controller`, `Service`, `Repository`를 두는 방식이 유지보수에 더 유리합니다. 저는 패키지 분리 기준을 도메인 경계, 변경 이유, 트랜잭션 경계 기준으로 잡는 편입니다.

### 4. Practical Tip
- 실제 적용 사례: `member`, `record`, `scramble`처럼 핵심 비즈니스 단위별로 패키지를 만들고, 공통 예외 처리나 보안 설정만 별도 패키지로 두면 수정 범위를 빠르게 좁힐 수 있다.
- 잘못 쓰면 생기는 문제: 계층형 구조를 큰 프로젝트에 그대로 유지하면 기능 하나 수정할 때 여러 전역 패키지를 동시에 열어야 하고, 반대로 도메인형을 과하게 잘게 쪼개면 중복 DTO와 중복 서비스가 생긴다.
- 프로젝트 연결: Spring Boot 프로젝트에서는 도메인별 패키지 아래에 `controller`, `service`, `repository`, `dto`를 두는 구조가 API 변경 추적과 테스트 범위 설정에 가장 실용적이다.

### 5. 예상 꼬리 질문
1. 도메인형 패키지 구조를 쓸 때 `common` 패키지는 어디까지 허용해야 하나요?
- 답변: `common`에는 인증, 예외, 공통 응답처럼 여러 도메인이 정말 함께 쓰는 횡단 관심사만 두는 것이 좋습니다. 특정 도메인 규칙이나 비즈니스 로직이 들어가기 시작하면 사실상 경계가 무너진 것이므로 원래 도메인 패키지로 되돌리는 편이 맞습니다.
2. 계층형 구조에서 유지보수성이 급격히 떨어지는 시점은 언제라고 보나요?
- 답변: 기능 하나를 수정할 때마다 `controller`, `service`, `repository`, `dto`를 전역 패키지에서 계속 찾아가야 하는 순간부터 체감이 커집니다. 특히 도메인 수가 늘고 팀원이 여러 명일 때 변경 영향 추적이 어려워지면서 유지보수성이 빠르게 떨어집니다.
3. 패키지 구조와 DDD의 Bounded Context는 어떤 관계가 있나요?
- 답변: 패키지 구조가 곧 Bounded Context 그 자체는 아니지만, 작은 프로젝트에서는 컨텍스트 경계를 패키지로 표현하는 출발점이 될 수 있습니다. 다만 Bounded Context는 모델과 용어, 책임 범위까지 포함한 개념이라 단순 폴더 분리보다 더 큰 설계 단위입니다.
