## Topic
예외 처리와 API 에러 응답 설계

### 1. Why
- **실무**: 예외는 반드시 발생한다. 중요한 것은 예외를 숨기지 않고, 어디서 던지고 어떻게 공통 포맷으로 응답할지 정해 클라이언트가 일관되게 처리할 수 있도록 만드는 것이다.
- **구조적 의미**: 예외 처리는 오류 제어 흐름을 구조화하는 작업이다. 비즈니스 실패, 검증 실패, 시스템 실패를 같은 방식으로 다루면 책임이 섞이고 응답 계약이 불안정해진다.
- **면접 의도**: 면접에서는 예외를 어디서 던지는지, 커스텀 예외와 전역 예외 처리를 어떻게 설계하는지, 에러 코드와 응답 포맷을 클라이언트 협업 관점에서 설명할 수 있는지 본다.

### 2. Core Concept
- **개념 정의**: 예외는 정상 흐름을 벗어난 상황을 표현하는 메커니즘이다. 커스텀 예외는 도메인 의미를 담기 위해 직접 정의한 예외이고, 전역 예외 처리는 여러 예외를 공통 정책으로 응답하기 위한 방식이다.
- **동작 방식**: 보통 검증이나 비즈니스 규칙 위반은 `Service`나 도메인 로직에서 예외를 던지고, `ControllerAdvice` 같은 전역 예외 처리기에서 상태 코드, 에러 코드, 메시지, 세부 필드를 공통 응답 포맷으로 변환한다. 입력값 검증 실패, 인증 실패, 시스템 오류는 서로 다른 코드와 메시지 정책을 가진다.
- **장점 / 단점**: 예외를 계층별로 명확히 던지고 전역에서 수습하면 중복 `try-catch`가 줄고 응답 형식이 일관된다. 반면 예외 타입이 너무 많아지면 관리가 어려워지고, 모든 예외를 커스텀으로 감싸면 오히려 가독성이 떨어질 수 있다.
- **비교 포인트**: `Controller`는 HTTP 바인딩 오류를 처리하고, `Service`는 비즈니스 예외를 던지는 쪽이 자연스럽다. 단순 문자열 메시지보다 안정적인 에러 코드를 함께 두면 클라이언트가 분기 처리하기 쉽다. 전역 예외 처리 없이 컨트롤러마다 응답을 만들면 API 포맷이 쉽게 흔들린다.
- **사용 조건 / 주의점**: 예외 메시지는 사용자 노출용과 로그용을 구분하는 것이 좋다. 에러 코드는 사람이 읽기 쉬우면서도 안정적으로 유지되어야 하고, 응답 포맷은 성공 응답과 실패 응답 모두에서 협업 가능한 규칙을 가져야 한다.

### 3. Interview Answer Version
- 30~40초 답변: 저는 예외를 발생 지점과 응답 지점을 분리해서 설계하는 편입니다. 비즈니스 규칙 위반은 `Service` 계층에서 커스텀 예외로 던지고, `ControllerAdvice` 같은 전역 예외 처리기에서 상태 코드와 에러 코드, 메시지를 공통 포맷으로 변환합니다. 이렇게 해야 컨트롤러마다 `try-catch`를 반복하지 않아도 되고, 클라이언트도 일관된 에러 응답을 받을 수 있습니다. 특히 메시지보다 안정적인 에러 코드를 함께 주는 것이 협업 관점에서 중요하다고 생각합니다.

### 4. Practical Tip
- 실제 적용 사례: `MEMBER_NOT_FOUND`, `DUPLICATE_EMAIL`, `INVALID_TOKEN`처럼 의미 있는 에러 코드를 만들고, 응답을 `code`, `message`, `details`, `timestamp` 같은 구조로 통일하면 프론트엔드가 대응하기 쉽다.
- 잘못 쓰면 생기는 문제: `Service`에서 예외를 삼키거나, 컨트롤러마다 제각각 메시지를 만들어 응답하면 클라이언트가 상황별 처리를 할 수 없고 장애 분석도 어려워진다.
- 프로젝트 연결: Spring Boot에서는 `@RestControllerAdvice`로 전역 예외 처리를 두고, validation 예외와 비즈니스 예외를 분리해 상태 코드와 응답 포맷을 통일하는 방식이 가장 실용적이다.

### 5. 예상 꼬리 질문
1. 커스텀 예외를 너무 많이 만드는 것은 어떤 문제가 있나요?
- 답변: 예외 클래스 수만 많아지고 실제 구분 기준이 흐려지면 오히려 읽기 어려워집니다. 그래서 도메인별로 의미 있는 단위만 정의하고, 세부 분기는 에러 코드로 정리하는 편이 더 실용적입니다.
2. validation 예외와 비즈니스 예외는 왜 구분해야 하나요?
- 답변: validation 예외는 입력 형식과 필수값 문제이고, 비즈니스 예외는 도메인 규칙 위반이라 실패 원인이 다릅니다. 둘을 구분해야 상태 코드, 메시지, 클라이언트 대응 방식도 더 정확하게 설계할 수 있습니다.
3. 에러 코드와 메시지 중 무엇이 더 중요한가요?
- 답변: 시스템 협업 관점에서는 안정적인 에러 코드가 더 중요합니다. 메시지는 사람 친화적인 설명이고 변경될 수 있지만, 코드는 클라이언트 분기 처리와 로그 분석 기준이 되기 때문입니다.
