## Topic
추상 클래스, 인터페이스, 다형성의 실무 활용

### 1. Why
- **실무**: 코드가 커질수록 구현체를 교체하거나 기능을 확장할 수 있는 구조가 중요해진다. 이때 추상 클래스, 인터페이스, 다형성은 확장 가능한 설계를 만드는 기본 도구다.
- **구조적 의미**: 세 개념은 객체지향의 핵심 추상화 수단이다. 구현에 직접 의존하지 않고 역할에 의존하게 만들어 결합도를 낮춘다.
- **면접 의도**: 면접에서는 "정의"를 묻는 것보다, 추상 클래스 vs 인터페이스를 어떤 기준으로 구분하는지, 다형성을 실무에서 어떻게 쓰는지, 확장 구조를 어떻게 설계하는지 알고 있는지 확인한다.

### 2. Core Concept
- **개념 정의**: 추상 클래스는 공통 상태와 기본 구현을 일부 제공하면서 하위 클래스 확장을 전제로 하는 클래스다. 인터페이스는 구현보다 역할 계약을 정의하는 데 가깝고, 다형성은 같은 타입으로 여러 구현체를 다룰 수 있는 성질이다.
- **동작 방식**: 상위 타입에 의존하고 실제 실행 시점에는 구현체가 주입되거나 선택된다. 예를 들어 `PaymentProcessor` 인터페이스를 두고 카드, 포인트, 쿠폰 결제를 구현체로 나누면 호출 코드는 구현체를 몰라도 된다. 공통 필드와 보호 메서드가 꼭 필요하면 추상 클래스를 활용할 수 있다.
- **장점 / 단점**: 인터페이스는 결합도를 낮추고 교체와 테스트가 쉽다. 추상 클래스는 공통 로직 재사용에 유리하지만 상속 구조가 깊어지면 유연성이 떨어질 수 있다. 다형성은 확장에 강하지만 설계가 과하면 단순한 문제도 복잡해질 수 있다.
- **비교 포인트**: 추상 클래스는 "공통 기반 구현"에 가깝고, 인터페이스는 "역할 계약"에 가깝다. 공통 상태와 템플릿 메서드가 필요하면 추상 클래스가 적합하고, 여러 구현체 교체 가능성과 유연한 확장 구조 설계가 중요하면 인터페이스가 더 적합하다.
- **사용 조건 / 주의점**: 단순히 객체지향스럽게 보이려고 인터페이스를 남발할 필요는 없다. 실제로 교체 가능성이 있거나 외부 의존성을 분리해야 할 때 쓰는 것이 좋고, 상속보다 조합이 더 단순한 경우에는 굳이 추상 클래스를 만들지 않는 편이 낫다.

### 3. Interview Answer Version
- 30~40초 답변: 추상 클래스와 인터페이스는 둘 다 추상화 수단이지만 목적이 조금 다릅니다. 추상 클래스는 공통 상태나 기본 구현을 공유할 때 적합하고, 인터페이스는 역할 계약을 정의해서 구현체를 교체 가능하게 만들 때 더 적합합니다. 다형성은 이런 상위 타입에 의존해서 실제 구현체를 런타임에 바꿔 끼울 수 있게 해 주기 때문에, 확장 구조 설계의 핵심이라고 생각합니다. 실무에서는 단순 상속보다 인터페이스 + 조합을 우선 고려하고, 정말 공통 기반 구현이 필요할 때만 추상 클래스를 쓰는 편이 안전합니다.

### 4. Practical Tip
- 실제 적용 사례: 알림 발송 기능에서 `Notifier` 인터페이스를 두고 이메일, Slack, SMS 구현체를 분리하면 다형성으로 같은 흐름에서 여러 채널을 처리할 수 있다.
- 잘못 쓰면 생기는 문제: 교체 가능성도 없는데 인터페이스를 과하게 만들면 구조만 복잡해지고, 추상 클래스에 책임을 과하게 몰아 넣으면 하위 클래스가 강하게 결합된다.
- 프로젝트 연결: Spring에서는 생성자 주입과 전략 패턴을 함께 쓰기 쉬워서 인터페이스 기반 설계와 다형성 활용이 자연스럽다. 공통 처리 흐름이 명확할 때만 추상 클래스를 쓰는 편이 유지보수에 유리하다.

### 5. 예상 꼬리 질문
1. 인터페이스 대신 추상 클래스를 선택해야 하는 대표적인 상황은 언제인가요?
- 답변: 여러 구현체가 공통 상태와 기본 동작을 실제로 많이 공유하고, 템플릿 메서드처럼 흐름 일부를 상속으로 강제하고 싶을 때는 추상 클래스가 더 적합할 수 있습니다. 단순히 "혹시 나중에 공통 로직이 생길 것 같아서" 정도라면 인터페이스가 더 안전합니다.
2. 다형성을 남용하면 어떤 문제가 생기나요?
- 답변: 구현체가 너무 많아지고 호출 흐름이 여러 추상층 뒤로 숨으면 실제 동작 추적이 어려워집니다. 확장성을 얻으려다 오히려 읽기 어려운 구조가 될 수 있어서, 실제 교체 가능성과 변동성이 있을 때만 쓰는 것이 좋습니다.
3. 상속보다 조합을 더 선호하는 이유는 무엇인가요?
- 답변: 조합은 필요한 기능만 주입해서 쓸 수 있어 결합도가 낮고 변경에 더 유연합니다. 반면 상속은 부모 구조에 강하게 묶이기 때문에 설계가 틀어지면 하위 클래스 전체가 영향을 받기 쉽습니다.
