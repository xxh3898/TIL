## Topic
테스트 전략과 테스트 코드 작성 기준

### 1. Why
- **실무**: 테스트는 단순히 통과 개수를 늘리는 게 아니라, 변경해도 안심하고 배포할 수 있는 안전장치를 만드는 일이다. 그래서 단위 테스트 / 통합 테스트를 구분하고 계층별 우선순위를 정하는 것이 중요하다.
- **구조적 의미**: 좋은 테스트 전략은 "무엇을 어디까지 검증할 것인가"를 설계하는 문제다. 모든 테스트를 같은 방식으로 작성하면 속도와 신뢰도 균형을 잃기 쉽다.
- **면접 의도**: 면접에서는 `MockMvc`, `Mockito`, `Testcontainers`를 단순 도구 소개가 아니라, 어떤 계층을 어떻게 테스트할지와 연결해서 설명하는지 본다.

### 2. Core Concept
- **개념 정의**: 단위 테스트는 외부 의존성을 최대한 배제하고 특정 클래스나 메서드의 로직을 검증하는 테스트다. 통합 테스트는 여러 컴포넌트가 함께 붙었을 때 실제 협업 결과를 검증한다.
- **동작 방식**: 보통 서비스 계층의 순수 비즈니스 로직은 단위 테스트로 빠르게 검증하고, 웹 계층은 `MockMvc`로 요청/응답과 validation, 상태 코드를 확인한다. 의존성 대체가 필요하면 `Mockito`로 mock 객체를 만들고, 실제 DB나 인프라와 가까운 검증은 `Testcontainers`로 운영과 유사한 환경을 구성할 수 있다.
- **장점 / 단점**: 단위 테스트는 빠르고 실패 원인을 찾기 쉽지만 실제 환경 차이를 잡기 어렵다. 통합 테스트는 신뢰도가 높지만 느리고 셋업 비용이 크다. 그래서 둘을 적절히 나눠야 한다.
- **비교 포인트**: `MockMvc`는 컨트롤러 요청 흐름 검증에 적합하고, `Mockito`는 단위 테스트에서 외부 의존성 격리에 유용하다. `Testcontainers`는 인메모리 DB와 달리 실제 DB 환경에 더 가까운 테스트를 할 수 있지만 속도 비용이 있다. 테스트 우선순위는 실패 비용이 큰 핵심 비즈니스 로직과 외부 계약부터 잡는 것이 일반적이다.
- **사용 조건 / 주의점**: 모든 레이어를 똑같이 많이 테스트할 필요는 없다. 순수 계산 로직은 단위 테스트, API 계약과 예외 응답은 웹 테스트, DB 쿼리와 영속성 이슈는 통합 테스트로 나누는 것이 효율적이다. mock만 많은 테스트는 실제 동작 보장이 약할 수 있으니 균형이 중요하다.

### 3. Interview Answer Version
- 30~40초 답변: 테스트 전략은 모든 것을 한 종류의 테스트로 커버하는 게 아니라, 계층별로 적절한 도구를 선택하는 문제라고 생각합니다. 순수 비즈니스 로직은 단위 테스트로 빠르게 검증하고, 컨트롤러 요청과 응답은 `MockMvc`로 확인하고, 외부 의존성을 격리할 때는 `Mockito`를 사용합니다. 또 DB 쿼리나 실제 인프라와 가까운 검증은 `Testcontainers` 기반 통합 테스트가 더 신뢰도가 높습니다. 우선순위는 실패 비용이 큰 핵심 로직과 외부 계약부터 잡는 것이 좋다고 봅니다.

### 4. Practical Tip
- 실제 적용 사례: `Service`의 계산 로직은 단위 테스트, `Controller`의 validation과 상태 코드는 `MockMvc`, Repository와 실제 DB 호환성은 `Testcontainers` 기반 통합 테스트로 나누면 역할이 명확하다.
- 잘못 쓰면 생기는 문제: 모든 테스트를 mock 기반으로만 작성하면 실제 쿼리나 직렬화, 트랜잭션 문제를 놓칠 수 있고, 반대로 모든 테스트를 통합 테스트로만 작성하면 너무 느려져서 유지가 어렵다.
- 프로젝트 연결: Spring Boot 프로젝트에서는 핵심 비즈니스 규칙과 예외 응답을 우선 테스트하고, DB 의존성이 큰 기능은 점진적으로 `Testcontainers`를 도입하는 방식이 현실적이다.

### 5. 예상 꼬리 질문
1. 단위 테스트와 통합 테스트의 비율은 어떻게 가져가는 것이 좋나요?
- 답변: 정해진 황금비율보다는 프로젝트 성격이 더 중요하지만, 일반적으로는 빠른 피드백을 주는 단위 테스트를 더 많이 두고 핵심 흐름은 통합 테스트로 보강하는 편이 현실적입니다. 중요한 것은 개수보다 실패 비용이 큰 영역을 놓치지 않는 것입니다.
2. `MockMvc`와 실제 서버를 띄우는 테스트는 어떤 차이가 있나요?
- 답변: `MockMvc`는 실제 네트워크 서버를 띄우지 않고 Spring MVC 계층을 빠르게 검증할 수 있어 속도가 빠릅니다. 반면 실제 서버 테스트는 더 현실적이지만 느리고 셋업 비용이 더 큽니다.
3. `Testcontainers`를 도입하면 어떤 점이 좋아지고 어떤 비용이 생기나요?
- 답변: 실제 운영과 유사한 DB나 인프라 환경을 테스트에 가져올 수 있어 신뢰도가 높아집니다. 대신 컨테이너 기동 시간과 테스트 환경 준비 비용이 추가돼 전체 테스트 속도는 느려질 수 있습니다.
