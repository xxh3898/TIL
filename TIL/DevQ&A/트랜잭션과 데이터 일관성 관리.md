## Topic
트랜잭션과 데이터 일관성 관리

### 1. Why
- **실무**: 계좌 이체, 주문 생성, 재고 차감처럼 여러 변경이 하나의 작업으로 묶여야 하는 경우가 많다. 이때 트랜잭션 경계를 잘못 잡으면 일부만 반영돼 데이터 일관성이 깨진다.
- **구조적 의미**: 트랜잭션은 여러 DB 작업을 하나의 논리적 단위로 묶는 경계다. 즉 일관성 관리는 코드 스타일이 아니라 작업 단위 설계 문제다.
- **면접 의도**: 면접에서는 트랜잭션이 무엇인지, ACID, `commit / rollback`, Spring의 `@Transactional`, 전파 속성, 실무에서 경계를 어디에 잡는지 설명할 수 있는지 본다.

### 2. Core Concept
- **개념 정의**: 트랜잭션은 모두 성공하거나 모두 실패해야 하는 작업 단위다. ACID는 원자성, 일관성, 격리성, 지속성을 의미하며 트랜잭션이 지켜야 할 핵심 성질이다.
- **동작 방식**: 트랜잭션 안에서 여러 작업이 실행되다가 모두 성공하면 `commit`, 중간에 오류가 나면 `rollback`되어 이전 상태로 되돌아간다. Spring에서는 `@Transactional`을 통해 메서드 단위로 트랜잭션을 선언하고, 프록시 기반으로 시작과 종료를 관리한다. 전파 속성은 이미 진행 중인 트랜잭션이 있을 때 현재 메서드가 그 트랜잭션에 참여할지 새로 시작할지를 정하는 규칙이다.
- **장점 / 단점**: 트랜잭션은 데이터 일관성을 보장하는 가장 기본 수단이다. 반면 경계를 너무 크게 잡으면 락 유지 시간이 길어지고 성능이 떨어질 수 있으며, 너무 잘게 쪼개면 한 작업의 일관성을 보장하기 어려워진다.
- **비교 포인트**: `REQUIRED`는 기존 트랜잭션에 참여하거나 없으면 새로 만들고, `REQUIRES_NEW`는 항상 새 트랜잭션을 만든다. 핵심은 전파 속성을 외우는 것보다, 어떤 작업을 하나의 비즈니스 단위로 묶을지 먼저 판단하는 것이다.
- **사용 조건 / 주의점**: 실무에서 트랜잭션 경계는 보통 서비스 유스케이스 단위로 잡는다. 외부 API 호출까지 긴 트랜잭션 안에 묶으면 위험할 수 있고, 읽기 전용 조회는 `readOnly` 옵션을 고려할 수 있다. 또한 동일 클래스 내부 호출은 `@Transactional` 프록시가 적용되지 않는다는 점도 주의해야 한다.

### 3. Interview Answer Version
- 30~40초 답변: 트랜잭션은 여러 DB 작업을 하나의 작업 단위로 묶어 모두 성공하거나 모두 실패하게 만드는 장치입니다. 그래서 데이터 일관성을 지키는 핵심이고, ACID가 그 기본 성질입니다. Spring에서는 보통 서비스 계층 메서드에 `@Transactional`을 붙여 트랜잭션 경계를 선언하고, 성공하면 `commit`, 예외가 나면 `rollback`됩니다. 실무에서는 전파 속성을 외우는 것보다 어떤 비즈니스 작업을 하나의 경계로 묶을지 판단하는 게 더 중요하다고 생각합니다.

### 4. Practical Tip
- 실제 적용 사례: 주문 생성에서 주문 저장, 주문 항목 저장, 재고 차감이 하나의 작업이라면 같은 서비스 메서드 안에서 하나의 트랜잭션으로 묶는 편이 자연스럽다.
- 잘못 쓰면 생기는 문제: 트랜잭션 경계가 너무 넓으면 락이 길게 잡히고, 너무 좁으면 주문은 저장됐는데 재고 차감은 실패하는 식의 부분 성공 상태가 생길 수 있다.
- 프로젝트 연결: Spring Boot + JPA에서는 `@Transactional`을 보통 서비스 계층에 두고, 조회 전용 메서드는 `readOnly = true`를 검토하며, 외부 시스템 호출과 DB 트랜잭션을 한 덩어리로 오래 묶지 않도록 주의하는 편이 좋다.

### 5. 예상 꼬리 질문
1. `REQUIRED`와 `REQUIRES_NEW`는 언제 다르게 쓰나요?
- 답변: 외부 작업과 하나의 비즈니스 흐름으로 함께 묶어야 하면 `REQUIRED`가 일반적입니다. 반면 바깥 트랜잭션 실패와 무관하게 별도로 남겨야 하는 감사 로그 같은 경우는 `REQUIRES_NEW`를 고려할 수 있습니다.
2. `@Transactional`을 컨트롤러보다 서비스 계층에 두는 이유는 무엇인가요?
- 답변: 트랜잭션 경계는 HTTP 요청 단위보다 비즈니스 유스케이스 단위에 더 가깝기 때문입니다. 서비스 계층에 두면 웹 기술과 무관하게 같은 비즈니스 작업을 재사용하고 테스트하기도 쉬워집니다.
3. 분산 트랜잭션이 어려운 이유는 무엇인가요?
- 답변: 여러 DB나 외부 시스템을 동시에 하나의 원자적 작업으로 묶으려면 실패 복구와 일관성 조율이 매우 복잡해집니다. 네트워크 지연과 부분 실패가 생기기 때문에 실무에서는 보통 보상 트랜잭션이나 이벤트 기반 처리로 우회하는 경우가 많습니다.
