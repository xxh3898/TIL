## Topic
파일 업로드와 정적 리소스 처리 구조

### 1. Why
- **실무**: 프로필 이미지, 첨부 파일, 게시글 이미지 같은 기능은 거의 항상 필요하지만, 파일은 일반 JSON 데이터보다 용량과 보안 이슈가 크다. 그래서 업로드와 정적 리소스 제공 구조를 따로 설계해야 한다.
- **구조적 의미**: 파일 처리 구조는 바이너리 데이터 저장 위치, 메타데이터 관리 방식, 접근 제어와 검증 규칙을 포함하는 인프라 설계 문제다.
- **면접 의도**: 면접에서는 `multipart` 처리, `S3` 업로드, 파일 메타데이터 관리, 보안/검증 포인트를 실제 서비스 구조로 설명할 수 있는지 본다.

### 2. Core Concept
- **개념 정의**: 파일 업로드는 보통 `multipart/form-data` 요청으로 처리한다. 실제 파일 바이너리는 로컬 디스크보다 `S3` 같은 외부 스토리지에 저장하고, 애플리케이션 DB에는 파일 이름, URL, 용량, 콘텐츠 타입 같은 메타데이터를 저장하는 구조가 일반적이다.
- **동작 방식**: 클라이언트가 `multipart` 요청으로 파일을 보내면 서버는 크기와 확장자, MIME 타입을 검증한 뒤 저장소에 업로드한다. 업로드가 완료되면 S3 key, 접근 URL, 원본 파일명, 저장 파일명, 크기 같은 파일 메타데이터를 DB에 저장하고, 이후 조회 시 메타데이터를 기준으로 정적 리소스 URL을 내려준다.
- **장점 / 단점**: 외부 스토리지를 쓰면 서버 디스크 부담이 줄고 확장성과 백업이 쉬워진다. 반면 업로드 실패와 DB 저장 실패를 함께 고려해야 하고, 접근 권한과 보안 검증을 별도로 설계해야 한다.
- **비교 포인트**: 서버 로컬 저장은 단순하지만 다중 인스턴스 환경과 운영에 취약하다. S3 저장은 운영과 확장에 유리하지만 네트워크 비용과 권한 설정이 추가된다. 정적 리소스를 바로 공개할지, presigned URL로 제한할지도 요구사항에 따라 달라진다.
- **사용 조건 / 주의점**: 파일은 확장자만 믿지 말고 콘텐츠 타입과 크기 제한을 함께 검증해야 한다. 실행 파일 업로드, 경로 조작, 과대 용량 업로드 같은 보안 포인트를 반드시 막아야 하고, 메타데이터와 실제 파일 저장이 어긋나는 실패 케이스도 고려해야 한다.

### 3. Interview Answer Version
- 30~40초 답변: 파일 업로드는 일반적으로 `multipart/form-data`로 처리하고, 실제 파일은 서버 로컬보다 `S3` 같은 외부 스토리지에 저장하는 경우가 많습니다. 애플리케이션 DB에는 파일 URL, 저장 경로, 크기, 콘텐츠 타입 같은 메타데이터를 따로 관리해서 비즈니스 데이터와 연결합니다. 이때 중요한 것은 업로드 자체보다 검증과 보안입니다. 파일 크기, 확장자, MIME 타입을 함께 검증하고, 접근 권한과 실패 시 정합성 문제까지 고려해야 안정적인 구조가 됩니다.

### 4. Practical Tip
- 실제 적용 사례: 프로필 이미지 업로드에서는 서버가 파일을 받아 S3에 저장한 뒤, 사용자 테이블에는 이미지 URL이나 파일 ID만 연결하는 구조가 가장 단순하고 운영하기 좋다.
- 잘못 쓰면 생기는 문제: 파일을 애플리케이션 서버 로컬 디스크에만 저장하면 서버가 여러 대일 때 파일 일관성이 깨지고, 검증 없이 업로드를 허용하면 악성 파일 업로드나 저장소 고갈 문제가 생긴다.
- 프로젝트 연결: Spring Boot에서는 `MultipartFile`로 업로드를 받고, S3 SDK 또는 presigned URL 방식을 사용하며, DB에는 파일 메타데이터를 별도 테이블로 관리하는 구성이 가장 흔하다.

### 5. 예상 꼬리 질문
1. DB에 파일 자체를 넣는 방식과 S3에 넣는 방식은 어떻게 비교하나요?
- 답변: 작은 파일을 강한 트랜잭션 안에서 함께 관리해야 하는 특수 경우가 아니라면, 일반적으로는 S3 같은 외부 스토리지가 확장성과 운영 측면에서 더 유리합니다. DB에는 파일 자체보다 메타데이터만 저장하는 방식이 훨씬 흔합니다.
2. presigned URL 방식은 언제 유리한가요?
- 답변: 대용량 파일이나 업로드 트래픽이 많은 경우에는 클라이언트가 직접 S3에 업로드하게 해서 애플리케이션 서버 부담을 줄일 수 있습니다. 이때 presigned URL은 짧은 시간 동안 제한된 업로드 권한만 주기 때문에 보안과 성능 균형이 좋습니다.
3. 업로드 중 S3 저장은 성공했는데 DB 저장이 실패하면 어떻게 처리하나요?
- 답변: 가장 흔한 방식은 실패 시 S3의 orphan 파일을 즉시 삭제하는 보상 로직을 두는 것입니다. 즉시 정리가 어렵다면 주기적으로 미연결 파일을 청소하는 비동기 정리 작업도 함께 고려할 수 있습니다.
