## Topic
페이징, 정렬, 검색 API 설계 방법

### 1. Why
- **실무**: 목록 조회 API는 대부분 페이징, 정렬, 검색 조건이 붙는다. 처음엔 단순해 보여도 데이터가 늘어나면 성능과 사용성 차이가 크게 난다.
- **구조적 의미**: 목록 API 설계는 단순 조회가 아니라 데이터 양 통제, 정렬 기준 정의, 검색 조건 조합, 응답 메타데이터 설계를 포함한 계약 설계다.
- **면접 의도**: 면접에서는 `page / slice`, `offset` 기반 페이징, 검색 조건 조합, 성능 주의점을 이해하고 실제로 어떤 기준으로 설계하는지 확인한다.

### 2. Core Concept
- **개념 정의**: 페이징은 한 번에 필요한 데이터만 나눠 조회하는 방식이고, 정렬은 결과 순서를 정하는 기준이다. 검색 API는 여러 조건을 조합해 필요한 데이터만 조회하는 API다.
- **동작 방식**: 가장 흔한 방식은 `page`, `size`, `sort`를 query parameter로 받아 `offset` 기반 페이징을 수행하는 것이다. `Page`는 전체 개수까지 포함한 응답이고, `Slice`는 다음 페이지 존재 여부 정도만 제공해 count 쿼리 비용을 줄일 수 있다. 검색 조건 조합은 보통 null 조건 무시 방식이나 동적 쿼리 빌더를 사용한다.
- **장점 / 단점**: `offset` 기반은 구현이 단순하고 UI와 맞추기 쉽지만 뒤 페이지로 갈수록 성능이 떨어질 수 있다. `Page`는 풍부한 메타데이터를 주지만 count 비용이 있고, `Slice`는 가볍지만 전체 페이지 정보는 없다.
- **비교 포인트**: 전체 개수와 총 페이지가 꼭 필요하면 `Page`, 무한 스크롤처럼 다음 페이지 존재 여부만 중요하면 `Slice`가 적합하다. 검색 조건이 많아질수록 문자열 기반 쿼리보다 QueryDSL 같은 동적 쿼리 도구가 유지보수에 유리하다.
- **사용 조건 / 주의점**: 정렬 컬럼은 화이트리스트로 제한해 의도하지 않은 필드 정렬을 막는 것이 안전하다. `offset` 기반은 큰 데이터셋에서 비효율적일 수 있으므로 필요하면 cursor 기반으로 전환해야 한다. 검색 조건을 무작정 늘리면 인덱스 전략과 count 쿼리 비용이 함께 악화된다.

### 3. Interview Answer Version
- 30~40초 답변: 페이징, 정렬, 검색 API는 단순 목록 조회가 아니라 성능과 사용성을 같이 설계하는 문제라고 생각합니다. 일반적으로는 `page`, `size`, `sort`를 받아 `offset` 기반으로 시작하고, 전체 개수가 필요하면 `Page`, 무한 스크롤처럼 가볍게 가져오면 `Slice`를 선택합니다. 검색 조건이 많아지면 동적 쿼리로 조합하고, 정렬은 허용 컬럼만 받도록 제한하는 게 안전합니다. 또 데이터가 커지면 `offset` 기반은 느려질 수 있어서 cursor 방식 전환 가능성까지 염두에 두는 것이 좋습니다.

### 4. Practical Tip
- 실제 적용 사례: 게시글 목록 API에서 `keyword`, `category`, `sort`, `page`, `size`를 query parameter로 받고, 응답에는 리스트와 함께 현재 페이지 정보 또는 다음 페이지 존재 여부를 내려주는 방식이 일반적이다.
- 잘못 쓰면 생기는 문제: count 쿼리가 비싼데도 무조건 `Page`만 쓰거나, 검색 조건이 늘었는데 인덱스와 정렬 기준을 맞추지 않으면 목록 API가 가장 느린 API가 되기 쉽다.
- 프로젝트 연결: Spring Data JPA에서는 `Pageable`을 바로 받을 수 있고, 복잡한 조건 조합은 QueryDSL로 분리하면 API 파라미터와 쿼리 로직을 비교적 깔끔하게 유지할 수 있다.

### 5. 예상 꼬리 질문
1. `Page`와 `Slice`는 어떤 기준으로 선택하나요?
- 답변: 총 개수와 총 페이지 수가 꼭 필요하면 `Page`가 적합합니다. 반면 무한 스크롤처럼 다음 페이지 존재 여부만 알면 되는 경우는 count 쿼리를 줄일 수 있는 `Slice`가 더 효율적입니다.
2. `offset` 기반 페이징이 느려지는 이유는 무엇인가요?
- 답변: 페이지가 뒤로 갈수록 DB는 앞쪽 행들을 많이 건너뛰어야 해서 읽는 양이 점점 커지기 때문입니다. 즉 반환 행 수는 같아도 스캔 비용이 누적되어 성능이 떨어질 수 있습니다.
3. 검색 조건이 많아질 때 쿼리 유지보수는 어떻게 관리하나요?
- 답변: 검색 조건 DTO를 따로 두고 동적 쿼리 생성 로직을 Repository 커스텀 구현이나 QueryDSL로 분리하는 편이 좋습니다. 조건마다 `if`와 문자열 SQL을 늘리는 방식은 금방 유지보수가 어려워집니다.
