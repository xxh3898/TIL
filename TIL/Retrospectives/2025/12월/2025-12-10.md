## 📚 학습 내용
> 오늘 공부하면서 새롭게 배운 내용이나 정리한 개념

### 1. SQL 중심 개발의 한계와 JPA의 필요성
- **SQL 중심 개발의 문제점**:
    - 무한 반복되는 CRUD 쿼리 작성 (`INSERT`, `UPDATE`, `SELECT`...).
    - 객체 설계보다 테이블 설계가 우선시되는 주객전도 현상.
    - 필드 하나 추가 시 관련된 모든 SQL과 DTO를 수정해야 함.
- **객체와 RDB의 패러다임 불일치 4가지**:
    1.  **상속**: 객체는 상속이 있지만, RDB는 상속이 없다. (슈퍼타입-서브타입 관계로 유사 구현만 가능)
    2.  **연관관계**: 객체는 **참조(`member.getTeam()`)** 를 사용하지만, RDB는 **외래 키(`JOIN`)** 를 사용한다.
    3.  **데이터 타입**: 객체는 자유로운 타입(컬렉션 등)을 쓰지만, RDB는 기본 타입만 지원한다.
    4.  **식별 방법**: 객체는 참조값(주소)으로 식별하지만, RDB는 PK로 식별한다.

### 2. JPA (Java Persistence API)
- **개념**: 자바 진영의 ORM(Object-Relational Mapping) 기술 표준 인터페이스. (구현체: Hibernate 등)
- **사용 이유 (장점)**:
    - **생산성**: `em.persist(member)`, `em.find(memberId)` 등 메서드 하나로 SQL 실행.
    - **유지보수성**: 엔티티 필드만 수정하면 쿼리는 JPA가 알아서 처리.
    - **패러다임 불일치 해결**: 상속, 연관관계, 객체 그래프 탐색 문제 해결.
    - **성능 최적화**: 1차 캐시, 쓰기 지연, 지연 로딩(Lazy Loading) 지원.

### 3. JPA 테이블 설계 및 매핑 (실습 적용)
- **엔티티 매핑**: `@Entity` (필수), `@Table`(매핑할 테이블명 지정).
- **PK 매핑**: `@Id`, `@GeneratedValue(strategy = GenerationType.IDENTITY)` (MySQL Auto Increment).
- **연관관계 매핑**:
    - **@ManyToOne**: 다대일 관계 (게시글 `N` : 회원 `1`). **가장 중요!**
    - **@JoinColumn**: 외래 키(FK) 매핑 (`name = "member_email"`).
    - **@OneToMany**: 일대다 관계 (회원 `1` : 게시글 `N`). `mappedBy`로 주인이 아님을 명시.

## 🧑🏻‍💻 코드 예제
> 수업 내용과 실습 프로젝트(`board`)에 적용한 핵심 코드

**1. Member Entity (회원)**
*`@OneToMany`를 사용하여 회원이 작성한 게시글 목록을 조회할 수 있도록 양방향 매핑.*
```java
@Entity
@Getter 
@NoArgsConstructor(access = AccessLevel.PROTECTED) // JPA 스펙상 기본 생성자 필수
@AllArgsConstructor 
@Builder
@Table(name = "MEMBER")
public class Member {

    @Id // 직접 할당 방식 (이메일)
    @Column(length = 255)
    private String email;

    @Column(nullable = false)
    private String name;
    
    // 연관관계의 주인이 아님 (Board의 member 필드에 매핑됨)
    // 회원 삭제 시 작성한 게시글도 함께 삭제 (CascadeType.ALL)
    @OneToMany(mappedBy = "member", cascade = CascadeType.ALL)
    private List<Board> boards = new ArrayList<>();
}
```

**2. Board Entity (게시글)**
*`@ManyToOne`으로 연관관계의 주인을 설정하고 `LAZY` 로딩으로 최적화.*
```java
@Entity
@Getter 
@NoArgsConstructor(access = AccessLevel.PROTECTED)
@AllArgsConstructor 
@Builder
@Table(name = "BOARD")
public class Board {

    @Id 
    @GeneratedValue(strategy = GenerationType.IDENTITY) // IDENTITY 전략
    @Column(name = "board_id")
    private Long boardId;

    @Column(nullable = false)
    private String title;

    @Column(nullable = false, columnDefinition = "TEXT")
    private String contents;

    // N:1 관계 (게시글 N : 회원 1) - 연관관계의 주인
    // 지연 로딩(LAZY) 필수: 실제 객체를 조회할 때 쿼리 실행
    @ManyToOne(fetch = FetchType.LAZY)
    @JoinColumn(name = "member_email", nullable = false) // FK 컬럼명
    private Member member; 
}
```

## 💣 Trouble Shooting
> 발생한 문제와 해결책

- **문제**: `Board`를 조회할 때 연관된 `Member`까지 즉시 조회되어 불필요한 조인 쿼리가 발생함.
- **원인**: `@ManyToOne`의 기본 Fetch 전략이 `EAGER`(즉시 로딩)이기 때문에, 게시글 1개를 가져올 때 회원 정보도 무조건 함께 가져옴.
- **해결**: `fetch = FetchType.LAZY` (지연 로딩) 옵션을 추가하여, `board.getMember()`를 실제로 호출하는 시점에 쿼리가 나가도록 변경함.
