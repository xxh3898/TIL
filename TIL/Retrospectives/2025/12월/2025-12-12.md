## 📚 학습 내용
> 오늘 공부하면서 새롭게 배운 내용이나 정리한 개념

### 1. 연관관계 매핑의 필요성 (패러다임 불일치)
- **객체 지향 vs 관계형 DB**: 객체는 **참조(Reference)** 를 통해 다른 객체와 관계를 맺지만, DB는 **외래 키(FK)** 를 사용하여 테이블을 조인(Join)한다.
- **문제점**: 객체를 테이블에 맞추어 데이터 중심으로 설계(ID만 가지고 있는 형태)하면, 객체 간의 협력 관계를 만들 수 없고 끊임없이 DB를 다시 조회해야 하는 비효율이 발생한다.
- **해결**: ORM(JPA)을 사용하여 객체의 참조와 테이블의 외래 키를 매핑함으로써 패러다임 불일치를 해결한다.

### 2. 연관관계 매핑 시 필수 고려 3요소
1.  **다중성 (Multiplicity)**: 관계의 수 (N:1, 1:N, 1:1, N:M)
2.  **방향성 (Direction)**:
    -   **단방향**: 한쪽에서만 참조 (예: 게시글 -> 작성자)
    -   **양방향**: 양쪽 모두 참조 (예: 게시글 <-> 작성자). 사실상 서로 다른 단방향 관계 2개다.
3.  **연관관계의 주인 (Owner)**: 양방향 관계에서 **외래 키(FK)를 관리(등록, 수정)하는 주체**.

### 3. 연관관계의 주인과 mappedBy
- **규칙**: **외래 키가 있는 곳이 무조건 주인(Owner)** 이다.
    -   예: `Board`(N)와 `Member`(1) 관계에서 FK는 `BOARD` 테이블에 있으므로 `Board.writer`가 주인이다.
- **주인의 역할**: 외래 키 관리(INSERT, UPDATE).
- **비주인(Inverse)의 역할**: `mappedBy` 속성을 사용하여 주인을 지정. **읽기 전용(Read-only)**이며 DB에 영향을 주지 않음.

### 4. 관계 유형별 매핑 전략 및 실무 권장 사항
- **다대일 (N:1) `[@ManyToOne]`**
    -   **가장 중요하고 자주 사용됨.**
    -   반드시 **`fetch = FetchType.LAZY` (지연 로딩)** 로 설정하여 N+1 문제를 방지해야 함.
    -   `@JoinColumn(name = "FK_NAME")`으로 외래 키 매핑.
- **일대다 (1:N) `[@OneToMany]`**
    -   **권장하지 않음 (단방향 시).**
    -   FK가 상대방(N쪽)에 있어, 내 엔티티(1쪽)를 수정했는데 상대방 테이블에 UPDATE 쿼리가 나가는 비효율 발생.
    -   **해결**: **다대일(N:1) 양방향 매핑**을 사용하자. (읽기 전용으로 필요할 때만 추가)
- **일대일 (1:1) `[@OneToOne]`**
    -   외래 키를 **주 테이블**에 두는 것을 권장 (개발 편의성, NULL 허용 여부 확인 용이).
    -   대상 테이블에 FK를 두면 프록시 한계로 인해 무조건 즉시 로딩(EAGER)되는 단점이 있음.
- **다대다 (N:M) `[@ManyToMany]`**
    -   **실무 사용 금지.**
    -   연결 테이블에 단순히 ID만 들어가는 것이 아니라, 추가 정보(등록일, 수량 등)가 들어갈 수 없기 때문.
    -   **해결**: **연결 엔티티(중간 테이블)를 직접 만들고** `@ManyToOne` 두 개로 풀어내야 한다.

### 5. 기타 주요 어노테이션
- **`@JoinColumn`**: 외래 키 컬럼명 지정 (`name`, `nullable` 등 설정).
- **`@Enumerated`**: Enum 타입 매핑 시 반드시 **`EnumType.STRING`** 사용 (ORDINAL 사용 시 순서 변경되면 데이터 꼬임).

## 🧑🏻‍💻 코드 예제
> 수업 시간에 작성한 핵심 코드나 직접 변형한 코드

```java
// 1. 다대일(N:1) 양방향 매핑 - 게시글(Board)과 작성자(Member)

// [주인] Board 엔티티: 외래 키를 관리함
@Entity
public class Board {
    @Id @GeneratedValue
    private Long boardNo;

    // 실무 필수: 지연 로딩 설정
    @ManyToOne(fetch = FetchType.LAZY) 
    @JoinColumn(name = "BOARD_WRITER") // FK 컬럼명
    private Member writer;
    
    // 연관관계 편의 메서드 (양쪽 다 값을 세팅해주기 위함)
    public void setWriter(Member writer) {
        this.writer = writer;
        writer.getBoards().add(this);
    }
}

// [비주인] Member 엔티티: 읽기 전용
@Entity
public class Member {
    @Id
    private String userId;

    // mappedBy로 주인을 명시 (Board의 writer 필드)
    @OneToMany(mappedBy = "writer") 
    private List<Board> boards = new ArrayList<>();
}
```

```java
// 2. 다대다(N:M) 해결 패턴 - 중간 엔티티 사용

@Entity
public class BoardTag {
    @Id @GeneratedValue
    @Column(name = "BOARD_TAG_ID")
    private Long id;

    @ManyToOne
    @JoinColumn(name = "BOARD_NO")
    private Board board;

    @ManyToOne
    @JoinColumn(name = "TAG_ID")
    private Tag tag;
    
    private LocalDateTime createdAt; // 추가 정보 관리 가능
}
```

## 💣 Trouble Shooting
> 발생한 문제와 해결책

- **문제 1: 양방향 매핑 시 데이터가 저장되지 않음 (FK가 NULL)**
    - **원인**: 연관관계의 주인인 `Board`에 값을 넣지 않고, 주인이 아닌 `Member.boards.add()`에만 값을 넣었기 때문.
    - **해결**: 외래 키가 있는 **주인(Owner) 객체에 반드시 값을 설정**해야 DB에 반영된다. (`board.setWriter(member)`)

- **문제 2: `StackOverflowError` 발생 (무한 루프)**
    - **원인**: Lombok의 `@ToString`을 양쪽 엔티티에 모두 사용하면 서로를 계속 호출하다가 스택 오버플로우가 발생함. (JSON 변환 시에도 동일)
    - **해결**: `@ToString(exclude = "fieldName")`을 사용하여 연관관계 필드를 제외하거나, **엔티티를 직접 반환하지 않고 DTO를 사용하여 반환**한다.
