# 📅 2025-12-17 (수) 데일리 회고록
**🏷️ Tags:** #Spring #JPA #JPQL #NativeQuery #12월

---

## 📚 학습 내용
> 오늘 공부하면서 새롭게 배운 내용이나 정리한 개념

- **객체지향 쿼리 언어 (JPQL)**
    - JPA는 엔티티 객체를 중심으로 개발하므로, 검색(조회) 할 때도 테이블이 아닌 **엔티티 객체를 대상으로 검색**해야 한다.
    - SQL을 추상화했기 때문에 특정 데이터베이스 SQL에 의존하지 않는다.
    - **특징**:
        - 대소문자 구분: 엔티티와 속성은 대소문자를 구분한다 (예: `Member`, `username`). JPQL 키워드는 구분하지 않는다 (예: `SELECT`, `FROM`).
        - 별칭(Alias) 필수: `select m from Member m` 처럼 별칭을 반드시 지정해야 한다.

- **쿼리 API (TypedQuery vs Query)**
    - **`TypedQuery`**: 반환 타입이 명확할 때 사용한다. (제네릭 지원)
    - **`Query`**: 반환 타입이 명확하지 않거나 여러 타입이 섞여 있을 때 사용한다.

- **결과 조회 API**
    - `getResultList()`: 결과가 하나 이상일 때 리스트 반환. 결과가 없으면 빈 리스트 반환.
    - `getSingleResult()`: 결과가 정확히 하나일 때 단일 객체 반환.
        - **주의**: 결과가 없으면 `NoResultException`, 둘 이상이면 `NonUniqueResultException` 예외가 발생하므로 주의해야 한다.

- **프로젝션 (Projection)**
    - 조회할 대상을 지정하는 것 (엔티티, 임베디드 타입, 스칼라 타입 등).
    - **DTO 조회**: `SELECT new com.example.dto.UserDTO(m.username, m.age) FROM Member m` 처럼 패키지명을 포함한 전체 클래스명과 `new` 키워드를 사용해야 한다.

- **네이티브 쿼리 (Native Query)**
    - JPQL로 해결할 수 없는 특정 데이터베이스에 종속적인 기능(특정 힌트, 문법 등)을 사용할 때 SQL을 직접 작성하는 방식.

## 🧑🏻‍💻 코드 예제
> 수업 시간에 작성한 핵심 코드나 직접 변형한 코드

```java
// 1. 기본 JPQL 조회 및 파라미터 바인딩
String jpql = "select m from Member m where m.username = :username";
TypedQuery<Member> query = em.createQuery(jpql, Member.class);
query.setParameter("username", "user1"); // 이름 기준 파라미터 바인딩 (권장)
List<Member> resultList = query.getResultList();

// 2. DTO로 직접 조회 (New 명령어 사용)
// 패키지 명을 다 적어야 하는 점에 주의!
String dtoJpql = "select new com.example.dto.MemberDto(m.username, m.age) from Member m";
List<MemberDto> resultDTO = em.createQuery(dtoJpql, MemberDto.class).getResultList();

// 3. 네이티브 쿼리 사용
String sql = "SELECT ID, AGE, TEAM_ID, NAME FROM MEMBER WHERE NAME = 'kim'";
List<Object[]> resultNative = em.createNativeQuery(sql).getResultList();
```

## 💣 Trouble Shooting
> 발생한 문제와 해결책

- **문제**: `getSingleResult()`를 사용해 로그인 로직을 구현했는데, ID가 틀려서 결과가 없으니 `NoResultException`이 터지면서 서버 에러가 발생함.
- **해결**: `getSingleResult()`는 결과가 0개이거나 2개 이상이면 무조건 예외를 던진다. 결과가 없을 가능성이 있는 로직에서는 `getResultList()`를 사용하여 리스트 사이즈를 체크하거나, Spring Data JPA의 `Optional` 반환을 활용하는 것이 안전하다.

## 🌟 잘한 점
> 오늘 하루 스스로 칭찬할 만한 점

- SQL과 비슷해 보이지만 엔티티를 대상으로 한다는 JPQL의 핵심 차이를 이해하고, 파라미터 바인딩 시 위치 기반(`?1`)보다 이름 기반(`:name`)이 유지보수에 좋다는 점을 확실히 짚고 넘어갔다.

---

## ⚠️ 아쉬운 점
> 부족했던 점

- JPQL을 문자열로 작성하다 보니 오타가 나면 실행 시점(Runtime)에 에러를 발견하게 된다. 동적 쿼리를 작성할 때도 문자열 더하기 방식은 너무 복잡하고 위험해 보인다.
