# 📅 2025-12-23 (화) 데일리 회고록
**🏷️ Tags:** #Spring #JPA #SpringDataJPA #QueryMethod #Paging #12월

## 📚 학습 내용
> 오늘 공부하면서 새롭게 배운 내용이나 정리한 개념

- **Spring Data JPA 소개 및 장점**
    - 기존의 `EntityManager`를 직접 사용하여 `persist`, `find` 등을 호출하던 반복적이고 지루한 CRUD 코드를 획기적으로 줄여준다.
    - `JpaRepository<Entity, ID>` 인터페이스를 상속받기만 하면 기본적인 CRUD 메서드가 자동으로 구현된다.

- **쿼리 메소드 (Query Method)**
    - 메서드 이름만으로 쿼리를 생성하는 강력한 기능.
    - 예: `findByUsername(String username)` -> `SELECT ... WHERE username = ?`

- **@Query 어노테이션**
    - 메서드 이름으로 쿼리를 표현하기 힘들거나 너무 길어질 때, JPQL을 직접 작성하여 바인딩할 수 있다.
    - 오타 발생 시 컴파일 시점이 아닌 로딩 시점에 에러를 잡아준다(Named Query의 장점).
    - 파라미터 바인딩 시 `@Param("name")`을 명시적으로 사용해야 한다.

- **페이징과 정렬 (Paging & Sorting)**
    - `Pageable` 인터페이스를 파라미터로 넘겨서 손쉽게 페이징 처리를 할 수 있다.
    - 반환 타입을 `Page<T>`로 받으면 데이터뿐만 아니라 전체 페이지 수, 전체 데이터 개수(count 쿼리 포함) 등의 메타데이터도 함께 받을 수 있다.

- **네이티브 쿼리 (Native Query)**
    - JPQL로 해결이 안 되는 복잡한 쿼리나 DB 종속적인 기능 사용 시 `nativeQuery = true` 옵션을 주어 SQL을 직접 작성할 수 있다.
    - DTO 매핑이 까다로운데, `Interface Projection`을 활용하여 해결할 수 있다.

## 🧑🏻‍💻 코드 예제
> 수업 시간에 작성한 핵심 코드나 직접 변형한 코드

```java
// 1. 기본 Repository 정의 및 쿼리 메소드
public interface MemberRepository extends JpaRepository<Member, Long> {
    
    // 이름으로 조회 (쿼리 메소드)
    List<Member> findByUsername(String username);
    
    // 나이가 특정 값 이상인 회원 조회
    List<Member> findByAgeGreaterThan(int age);
}
```

```java
// 2. @Query를 이용한 JPQL 정의 및 페이징
public interface BoardRepository extends JpaRepository<Board, Long> {

    // 복잡한 조건은 JPQL로 직접 작성 (@Param 필수)
    @Query("SELECT b FROM Board b WHERE b.title LIKE %:title% AND b.status = 'Y'")
    Page<Board> findActiveBoards(@Param("title") String title, Pageable pageable);
}
```

```java
// 3. 서비스 계층에서 페이징 사용 예시
public void getBoardList() {
    // 0페이지부터 10개씩, boardNo 기준으로 내림차순 정렬
    PageRequest pageRequest = PageRequest.of(0, 10, Sort.by("boardNo").descending());
    
    Page<Board> result = boardRepository.findActiveBoards("공지", pageRequest);
    
    System.out.println("전체 페이지 수: " + result.getTotalPages());
    System.out.println("조회된 데이터: " + result.getContent());
}
```

## 💣 Trouble Shooting
> 발생한 문제와 해결책

- **문제**: `@Query`를 사용하여 파라미터를 바인딩할 때, 이름 기반 바인딩(`:name`)을 사용했는데 실행 시 파라미터를 찾을 수 없다는 에러가 발생함.
- **해결**: Spring Data JPA에서 `@Query` 사용 시 파라미터 앞에는 반드시 `@Param("name")` 어노테이션을 붙여줘야 JPQL의 파라미터 변수와 매칭된다는 점을 확인하고 수정함.

## 🌟 잘한 점
> 오늘 하루 스스로 칭찬할 만한 점

- 기존에 `EntityManager`로 길게 작성하던 DAO 코드를 `Repository` 인터페이스 하나로 대체하면서 코드 생산성이 얼마나 높아지는지 직접 체감했다.
- 페이징 처리를 위해 `limit`, `offset`을 직접 계산하던 번거로움에서 벗어나 `Pageable` 객체 하나로 해결하는 방법을 익혔다.

---

## ⚠️ 아쉬운 점
> 부족했던 점

- 네이티브 쿼리(`nativeQuery = true`)를 사용할 때 결과 매핑(DTO 변환)이 생각보다 까다로웠다. `Interface Projection` 방식이 권장되지 않는다고 하는데, QueryDSL 같은 대안을 아직 배우지 못해 아쉽다.

---

## 🤔 내일 시도할 점
> 내일 시도하고 싶은 부분

- 실무에서 동적 쿼리를 처리할 때 `Specification`이나 `QueryDSL`을 많이 쓴다고 들었는데, Spring Data JPA만으로 동적 쿼리를 깔끔하게 처리하는 한계가 어디까지인지 테스트해보고 싶다.
