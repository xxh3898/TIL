# TIL - 2026-03-29


## 오늘 배운 것

### 개념
- **Conventional Commits**: `feat`와 `build` 타입의 차이를 학습함. `build.gradle`과 같은 빌드 시스템 수정은 `build` 타입이 더 적절함.
- **API 문서 관리 전략**: 자동 생성된 HTML 문서는 Git 충돌 방지와 원본(adoc, 테스트 코드) 보호를 위해 `.gitignore` 처리하는 것이 실무 표준임을 배움.
- **Gradle Build Lifecycle**: `./gradlew build` 작업은 단순히 코드를 빌드하는 것을 넘어, 자원 복사(`processResources`)부터 전체 테스트(`test`) 및 최종 패키징(`bootJar`)까지의 파이프라인을 의미함을 깨달음. 단편적인 테스트 코드 실행만으로는 잡아내지 못하는 배포 단계의 충돌 에러를 사전에 차단하는 핵심 관문임.

### 기술
- **Spring Rest Docs 빌드 자동화**: `asciidoctor` 태스크 결과물을 배포용 `app.jar`에 포함시키도록 설정함. 이때 `bootJar`가 중간 복사 태스크(`copyDocument`)를 우회하고 직접 `asciidoctor`에 의존하도록 설정하여 Gradle 8+의 엄격한 Task Input/Output 충돌 검증 에러를 완벽히 해결함.
- **Docker Compose 실전 연동**: Prometheus와 Grafana를 포함한 모니터링 스택을 애플리케이션과 연동하여 로컬 환경에서 검증함.

## 코드 예제

### Gradle Build Task (Document Copy & CI Fix)
```gradle
// 로컬 개발용 수동 복사 태스크 (CI 빌드와 격리)
tasks.register('copyDocument', Copy) {
    dependsOn asciidoctor
    from file("build/docs/asciidoc")
    into file("src/main/resources/static/docs")
}

bootJar {
    dependsOn asciidoctor // copyDocument에 의존하지 않음으로써 processResources와 충돌 방지
    from ("${asciidoctor.outputDir}") {
        into 'static/docs'
    }
}
```

## 핵심 정리

- API 명세서의 "Source of Truth"는 테스트 코드와 ADOC 파일이며, HTML은 결과물일 뿐이다.
- 실무 환경에서는 배포 시점에 파이프라인(CI/CD)에서 문서를 생성하여 배포본에 포함시킨다.
- PR 커밋/푸시 전, 인텔리제이 테스트(단편적 실행)에만 의존하지 않고 로컬 터미널에서 반드시 `./gradlew build`를 실행하여 CI 파이프라인(배포 단계)의 잠재적 환경 에러를 사전에 차단하는 습관을 가져야 한다.
