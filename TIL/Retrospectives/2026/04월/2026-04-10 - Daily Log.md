## 오늘 한 일

- `GET /api/rankings` V1 조사를 하면서 `records` 전체 기록 기준과 `user_pbs` 기준 랭킹의 의미 차이를 정리했다. 일정 문서의 "병목 유도 V1" 의도와 실제 도메인 모델의 `user_pbs` 존재를 같이 읽어, 구현 전에 의미를 먼저 확정해야 한다는 결론을 남겼다.
- 랭킹 API 응답 후보 필드와 인증 정책도 같이 검토했다. `rank`, `nickname`, `eventType`, `timeMs` 수준의 최소 응답으로 시작하는 안, `/api/rankings`를 별도 컨트롤러로 두는 안, 공개 조회로 열지 여부를 구현 전 결정 항목으로 분리했다.
- Day 11 게시판 API V1 리서치에서 일정 문서의 `CRUD` 표현을 그대로 믿지 않고, 실제 체크리스트 기준으로는 `POST /api/posts`, `GET /api/posts` 수준이 승인 범위에 가깝다고 정리했다.
- 게시글 목록 조회는 `keyword + author` 조합, QueryDSL 커스텀 리포지토리, 작성자 식별은 `nickname`, 생성 요청은 현재 로그인 사용자 이메일 기반 `UserRepository.findByEmail(...)` 흐름을 재사용하는 쪽으로 가닥을 잡았다.
- 게시판 쪽에서는 `viewCount`가 존재하지만 상세 조회가 Day 11 범위와 다를 수 있다는 점, 검색 성능 최적화나 페이징/정렬 공통화는 아직 과한 범위라는 점도 같이 적어 두었다.
- `docs/Development Log`를 어떤 식으로 쌓을지 위한 조사와 계획도 같이 남겼다. 단순 작업 일지보다 구현 이유, 다음 액션, 문서화 포인트를 따라갈 수 있는 기록 구조가 필요하다고 봤다.
- 도메인-레이어 리팩토링 조사에서는 `auth`, `post`, `record`, `user` 패키지 안에 controller/service/repository/entity/dto가 섞여 있는 구조를 실제 호출 흐름 기준으로 다시 읽었다.
- 리팩토링 조사 과정에서 이미 QueryDSL projection + join으로 N+1을 피한 부분과, 상세 조회나 예외 응답처럼 실제로 문제가 남아 있는 부분을 분리했다. 구조가 지저분하다는 이유만으로 전부 문제로 취급하지 않기로 했다.
- `AGENTS Contributor Guide` 초안과 전역 하네스 쪽 AI 산출물 위치 규칙도 함께 정리했다. 연구/계획 문서가 `docs/ai`나 `tmp/ai` 루트에 평면적으로 쌓이지 않고 작업 폴더 단위로 남아야 이후 계보 추적이 쉬워진다는 방향을 잡았다.

## 배운 점 요약

- 일정 문서의 문장만으로 구현에 들어가면 범위가 쉽게 커진다. 체크리스트, 현재 코드, 엔티티 모델을 같이 읽어 "오늘 닫아야 하는 계약"만 추려야 한다.
- 랭킹처럼 해석이 갈리는 기능은 성능보다 의미 정의가 먼저다. `records` 기준인지 `user_pbs` 기준인지 닫히지 않으면 이후 Redis 설계나 문서화도 흔들린다.
- 도메인-레이어 리팩토링은 "패키지가 예쁘지 않다"보다 "호출 흐름과 책임이 어디서 섞이는가"를 먼저 봐야 한다.
- 작업 산출물도 source of truth다. 연구/계획 문서 위치를 작업 폴더 기준으로 정리해 두면 후속 검토와 회고 비용이 크게 줄어든다.

## 문제 / 막힌 점

- 랭킹이 `solve leaderboard`인지 `user leaderboard`인지 고민했다.
- `/api/rankings`를 공개 조회로 열지, 응답 필드를 어디까지 노출할지 문서만으로는 확정되지 않았다.
- Day 11 게시판 요구사항의 `CRUD` 표현과 실제 승인 범위 사이 간극이 커서, 구현 전에 범위를 다시 줄여 읽어야 했다.
- AI 산출물 폴더 구조는 `yyyymmdd-작업명`까지만 정해졌고, 동일 날짜/동일 작업명 충돌 처리 규칙은 후속 정리가 필요했다.
