- 일정 문서에 `CRUD`처럼 넓게 적힌 요구를 실제 하루 작업 범위로 어떻게 줄여야 하는가
- 도메인 패키지 안에 controller/service/repository/entity/dto가 섞여 있을 때, 리팩토링 우선순위를 어디서부터 잡아야 하는가

## 개념 정리

- 요구사항 해석은 문서 한 장으로 끝나지 않는다. 일정 문서, 현재 코드, 엔티티 모델, 테스트 패턴을 함께 읽어 "지금 구현해야 하는 계약"과 "나중에 확장할 계약"을 분리해야 한다.
- 랭킹 기능은 조회 쿼리보다 의미 정의가 먼저다. `records`를 그대로 정렬하면 solve leaderboard가 되고, `user_pbs`를 읽으면 user leaderboard가 된다. 두 모델은 응답 의미, Redis 적재 방식, UI 기대값까지 모두 다르게 만든다.
- 범위 축소는 기능 삭제가 아니라 승인 범위를 명확히 하는 일이다. Day 11 문서에 `CRUD`가 적혀 있어도 실제 체크리스트가 `POST`, `GET /api/posts`라면 그날 닫아야 하는 범위는 생성과 목록 조회 쪽에 더 가깝다.
- 패키지 리팩토링은 모양보다 호출 흐름 기준으로 봐야 한다. 이미 projection + join으로 잘 분리된 조회를 괜히 뜯기보다, 상세 조회 LAZY 로딩, 응답 포맷, 예외 책임처럼 실제 결합이 센 지점을 먼저 손봐야 한다.

## 실전 적용

- 랭킹 API 조사에서는 일정 문서의 병목 유도 의도를 존중하면 `records`가 자연스럽고, 도메인 의미를 우선하면 `user_pbs`가 자연스럽다는 점을 동시에 적어 두었다. 구현 전에 의미 정의를 사람 결정으로 올리지 않으면 이후 문서와 코드가 같이 흔들릴 수 있다고 봤다.
- 게시판 API 조사에서는 `keyword + author` 검색, `nickname` 공개 식별자, QueryDSL 커스텀 리포지토리, 로그인 사용자 이메일 기반 작성자 조회라는 현재 저장소 패턴을 그대로 재사용하는 쪽으로 정리했다.
- 도메인-레이어 리팩토링 조사에서는 `auth`, `post`, `record`, `user` 전체를 다시 쓰려 하지 않고, 실제로 책임이 섞이는 `AuthService`, 상세 조회, 예외 응답 경계 중심으로 문제를 추렸다.
- AI 산출물 위치 규칙 쪽에서는 연구/계획 문서를 `docs/ai` 또는 `tmp/ai` 루트에 평면적으로 두지 말고 작업 폴더 아래에 묶어 두는 기준이 나중에 source tracking에 유리하다는 점을 정리했다.
