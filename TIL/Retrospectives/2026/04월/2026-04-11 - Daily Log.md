## 오늘 한 일

- `frontend/src/App.css`, `frontend/public/icons.svg`, `GlobalExceptionHandler`의 미사용 파라미터를 정리하는 dead code cleanup을 수행했다. 스캐폴드 잔여물과 읽히지 않는 파라미터만 제거하고, 향후 바로 재사용 가능성이 있는 코드는 일부러 남겨 두었다.
- dead code 정리 뒤에는 `cd backend && ./gradlew test`, `cd frontend && npm run lint`, 전역 `rg` 검색으로 참조가 끊기지 않았는지 확인했다. 코드 삭제 작업에서도 확인 가능한 검증은 바로 붙여 두는 편이 안전했다.
- Day 12 클라이언트 연동 조사에서는 현재 프런트가 `refresh_token`을 직접 읽는 구조가 아니라, 브라우저가 `HttpOnly` 쿠키를 `/api/auth/**`에 자동 전송하는 전제를 따라야 한다는 점을 정리했다.
- `localhost:5173`와 `localhost:8080`이 서로 다른 port를 써도 same-site라는 점, 다만 same-origin은 아니므로 `credentials: 'include'`와 서버 쪽 credential 허용이 둘 다 필요하다는 점을 명확히 적어 두었다.
- 기록 저장 API 쪽에서는 `POST /api/records`에 `eventType`, `timeMs`, `penalty`, `scramble`가 모두 필요하고, 현재 실제 스크램블 지원은 `WCA_333` 중심이라는 점을 다시 확인했다. 프런트가 placeholder scramble을 정상값처럼 저장하지 않게 API 레벨에서 미지원 종목을 막아야 한다고 봤다.
- 화면 중심 `feature spec` 문서 리서치도 같이 진행했다. 실제 프런트 구현은 `/auth`, `/timer` 정도인데, 백엔드 API는 랭킹/게시판까지 일부 열려 있어서 "구현된 화면"과 "설계 중인 화면"을 문서에서 분리해야 한다는 기준을 세웠다.
- `Stitch UI` 프롬프트, UI 정리, 한국어 카피 전환, 헤더/푸터 공통화 같은 프런트 정리 작업도 조사 문서로 남겼다. 이 단계에서는 실제 코드 변경보다 어떤 서비스 톤과 정보 구조가 필요한지 보는 쪽에 가까웠다.
- 전역 하네스 쪽에서는 승인 게이트를 더 엄격히 만드는 조사도 진행했다. 계획 피드백 대화와 실제 구현 승인 문장을 섞어 해석하지 않도록, `"이제 ..."` 형식 승인만 실행으로 보는 기준을 문서화하는 방향을 잡았다.

## 배운 점 요약

- 로컬 개발에서 `same-site`와 `same-origin`을 구분해 두지 않으면 인증 쿠키 동작을 잘못 해석하기 쉽다.
- 프런트 연동 문서는 현재 API 계약과 실제 지원 종목 범위를 같이 써야 한다. 그렇지 않으면 UI가 placeholder나 미지원 기능을 정상 기능처럼 소비하게 된다.
- dead code cleanup은 "지금 안 쓰는가"뿐 아니라 "다음 작업에서 바로 쓸 가능성이 있는가"를 함께 봐야 한다.
- 실행 규칙 문서는 예시보다 트리거 조건이 중요하다. 피드백과 실행 승인이 섞이지 않게 하려면 문장 형식까지 분명해야 한다.

## 문제 / 막힌 점

- `refresh_token` 쿠키의 localhost 동작은 문서상으로는 설명 가능했지만, 실제 브라우저별 수동 검증은 아직 남아 있었다.
- 스크램블 API는 실질적으로 `WCA_333` 중심인데 프런트 이벤트 옵션은 더 많아, UI와 계약 사이에 방어 로직이 필요했다.
- `feature spec` 문서를 쓸 때 이미 구현된 API와 아직 비어 있는 홈/학습/대시보드 영역을 어떻게 구분할지가 과제였다.
