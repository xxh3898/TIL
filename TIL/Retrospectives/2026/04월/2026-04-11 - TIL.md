# 2026-04-11 - TIL

## 오늘 다룬 질문

- `localhost` 개발 환경에서 `same-site`와 `same-origin` 차이가 인증 쿠키 전송에 어떤 영향을 주는가
- 프런트 연동 문서를 쓸 때, 이미 구현된 API와 아직 없는 화면 계약을 어떻게 구분해야 하는가
- 승인 게이트를 엄격히 만들려면 어떤 표현만 실행 승인으로 인정해야 하는가

## 개념 정리

- `same-site`와 `same-origin`은 다르다. `localhost:5173`와 `localhost:8080`은 포트가 달라도 same-site일 수 있지만, 여전히 서로 다른 origin이므로 `fetch`에는 `credentials: 'include'`가 필요하고 서버도 credential 허용이어야 한다.
- `HttpOnly` refresh cookie 구조에서는 프런트가 토큰을 읽는 게 아니라 브라우저에 전송을 맡겨야 한다. 프런트가 직접 관리할 수 있는 것은 access token과 인증 상태 복구 흐름이다.
- 화면 명세 문서는 "현재 구현된 것"과 "곧 구현할 것"을 섞어 적지 않는 편이 좋다. API가 없는 홈/학습/대시보드 영역은 화면/정보 구조 명세로 적고, 계약이 닫힌 인증/타이머/랭킹/게시판은 API 기준으로 적어야 혼선이 줄어든다.
- 승인 게이트는 문맥 추론보다 형식이 안전하다. 계획 피드백 대화에서 나온 `"해줘"` 류 표현을 실행으로 오해하지 않게 하려면, 실제 실행을 여는 문장 형식을 명시적으로 좁혀 두는 편이 낫다.

## 실전 적용

- Day 12 클라이언트 연동 조사에서는 `refresh_token`을 JS가 읽는 구조로 가정하지 않고, `/api/auth/refresh` 요청에 브라우저가 쿠키를 실어 보내는 구조로 정리했다.
- API base URL은 `127.0.0.1`이 아니라 `localhost`로 통일해야 same-site 전제가 깨지지 않는다는 점도 함께 메모했다.
- `feature spec` 문서 조사에서는 실제 프런트가 `/auth`, `/timer`만 실연동 상태라는 사실과, 홈/학습/랭킹/게시판은 구현 범위와 계약 수준이 각각 다르다는 점을 구분해 적는 기준을 세웠다.
- 전역 하네스 승인 게이트 조사에서는 `"이제"`로 시작하는 명시적 실행 승인만 실제 구현/진행으로 해석하는 방향이 안전하다고 정리했다.

## 핵심 요약

- 프런트 연동에서 가장 먼저 닫아야 하는 것은 "토큰을 어디에 저장할까"보다 "브라우저가 어떤 토큰을 어떤 조건에서 보내는가"다. 그 전제가 닫혀야 API 문서, 상태 관리, 오류 처리까지 일관되게 설계할 수 있다.
