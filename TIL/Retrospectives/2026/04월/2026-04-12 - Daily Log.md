- [x] 서비스형 UI 방향과 Day 13 이후 스케줄을 현실적인 기준으로 다시 잡기

## 오늘 한 일

- PRD, ERD, 목업, 현재 코드/프런트를 한 번에 대조하는 `mockup-doc-sync-review`를 수행했다. 큰 축은 맞지만 이후 구현 단계에서 반드시 충돌할 항목을 먼저 식별하려는 목적이었다.
- 이 리뷰에서 PRD에 댓글 기능이 빠져 있다는 점을 분명히 확인했다. ERD와 목업은 댓글을 이미 제품 범위로 보고 있었기 때문에, PRD가 그대로면 API 백로그와 우선순위 논의에서 댓글이 누락될 수 있었다.
- 랭킹도 문서와 구현의 해석이 어긋나 있었다. 목업과 ERD는 사용자당 1개의 PB row를 기대하게 만드는 반면, 당시 백엔드는 raw `record`를 빠른 순으로 정렬하는 구조였다. 이 차이를 문서 차원에서 먼저 닫아 두기로 했다.
- 커뮤니티 목록 요구사항도 다시 정리했다. 목업과 PRD는 카테고리 필터, 닉네임/제목 검색, 페이지네이션을 전제로 하고 있었지만, 현재 API는 `keyword`, `author` 중심이라 그 차이를 구현 전 backlog로 노출시켰다.
- 홈/마이페이지, 피드백, 오늘의 스크램블에 대해서도 "UI가 이미 말하고 있는 제품 의미"와 "현재 백엔드에 실제로 있는 공개 API"를 따로 적었다. 홈/마이페이지용 조회 계약이 아직 비어 있고, 피드백은 DB 저장보다 메일 전달 해석이 더 맞는다는 사용자 결정도 반영됐다.
- `AGENTS.md` 재검토에서는 문서가 실제 저장소보다 뒤처진 부분을 정리했다. 프런트는 더 이상 Vite 초기 스캐폴드가 아니고, `bootRun`은 테스트와 REST Docs 생성을 동반하며, `./gradlew test`는 Docker/Testcontainers 전제를 갖는다는 점을 명확히 잡아냈다.
- 커밋 가이드도 실제 히스토리와 맞지 않는 부분이 보여, 단순 "짧은 명령형"보다 `feat:`, `docs:` 같은 접두사와 한글 제목/본문 관행을 더 사실적으로 설명할 필요가 있다고 봤다.
- 서비스형 UI 전환 조사에서는 현재 프런트를 단순 기능 데모가 아니라 하나의 서비스 경험으로 보이게 만드는 방향을 검토했다. 코드 변경보다 어떤 정보 구조와 톤으로 정리할지가 더 중요한 단계였다.
- Day 13 이후 스케줄 재정렬과 `질문:` prefix 규칙도 함께 정리했다. 질문형 입력은 순수 Q&A로 처리하고, 구현/문서 수정으로 바로 넘어가지 않게 하는 기준이 생기면서 후속 흐름도 더 안정적으로 잡혔다.

## 배운 점 요약

- 문서 충돌은 구현 중간에 찾기보다, PRD/ERD/목업/코드를 같이 읽는 리뷰에서 먼저 잡는 편이 훨씬 싸다.
- 제품 문서의 불일치는 보통 댓글, 랭킹 기준, 페이지네이션 수치, 피드백 처리 채널 같은 "작아 보이지만 API 설계에 바로 영향을 주는 항목"에서 먼저 드러난다.
- 실행 가이드 문서는 실제 비용과 환경 전제를 숨기면 안 된다. `bootRun`의 숨은 테스트 비용이나 Docker 전제를 누락하면 다음 작업자가 잘못된 기대를 갖게 된다.
- 서비스형 UI 방향 같은 일회성 디자인 결정은 그 자체로 TIL보다는 Daily Log 쪽이 더 어울리고, TIL은 그 뒤에 남는 일반 원칙만 가져가는 편이 맞다.

## 문제 / 막힌 점

- 랭킹을 사용자당 PB 기준으로 확정하면 현재 API 구현과 이후 Redis 적재 방식도 같이 바뀌어야 해서, 단순 문구 수정 이상의 영향이 있었다.
- 홈/마이페이지는 제품 문서와 데이터 모델은 설명 가능했지만, 실제 조회 API가 비어 있어 구현 순서 재조정이 필요했다.
- `AGENTS.md`는 저장소 현재 상태를 따라가지 못하면 오히려 잘못된 작업 경로를 안내하게 되므로, 문서와 코드 드리프트를 계속 주의해야 했다.
