## 오늘 한 일

- Day 14 인증 실연동을 실제 코드로 닫았다. 백엔드에는 `GET /api/me`를 추가하고, refresh cookie `Secure` 속성을 환경별로 분기하도록 바꿨으며, `application-local.yaml`, `application-prod.yaml`, `application-test.yaml`에도 관련 설정을 넣었다.
- 프런트 쪽에서는 `AuthContext`에 현재 사용자 컨텍스트와 `/api/me` 폴백을 넣고, `apiClient`에 Authorization 자동 부착, 단일 in-flight refresh, `401 -> refresh -> retry` 흐름을 추가했다.
- `LoginPage`, `SignupPage`, `MyPage`를 실제 인증 API와 붙이고, 보호 라우트, guest-only 라우트, 로그인 후 복귀 흐름까지 같이 정리했다. `CommunityDetailPage`에는 비로그인 댓글 CTA도 적용했다.
- 인증 연동 변경 후 `cd backend && ./gradlew test`, `cd frontend && npm run lint`, `cd frontend && npm run build`를 통과시켰고, 회원가입 -> 로그인 -> 보호 route 복귀 -> refresh retry -> 로그아웃까지 수동 검증도 확인했다.
- 프로젝트 중간 점검도 같이 진행했다. 문서 7종과 실제 코드/테스트/보안 설정을 대조하면서 `PLUS_TWO` 페널티가 PB/랭킹에 반영되지 않는 문제, 사용자 대표 기록이 아니라 raw solve 로그를 그대로 노출하는 랭킹, UI 요구와 API 계약의 어긋남을 핵심 리스크로 올렸다.
- 중간 점검에서는 하드코딩된 비밀값, `localStorage` access token 구조, 예외 응답 평탄화 같은 보안/운영 리스크도 함께 정리했다. 기능이 돌아가는 것과 포트폴리오 기준에서 신뢰성이 충분한 것은 다른 문제라는 점을 다시 확인했다.
- 테스트 쪽에서는 `AGENTS` 테스트 규칙을 추가하고, 테스트 메서드 네이밍을 `should_{expectedBehavior}_when_{condition}` 형태로 맞추는 리팩토링을 진행했다.
- 백엔드 테스트 구조 리뷰에서는 현재 상태를 실무 기준 `불합격`으로 판단했다. REST Docs 테스트가 기능 검증을 과도하게 떠안고 있고, 공통 통합 테스트 베이스가 잘못된 격리 가정을 제공하며, 핵심 서비스 로직의 빠른 단위 피드백 루프가 부족하다고 정리했다.
- 그래서 문서화 테스트는 계약/문서 생성 책임만 남기고, 서비스 단위 테스트와 QueryDSL repository 테스트를 분리하는 방향을 기준으로 삼았다. 실패 응답 문서화도 REST Docs에 추가해 성공 케이스만 문서화되는 상태를 줄였다.
- 전역/로컬 하네스 쪽에서는 전역 AGENTS 병합, 테스트 정책 추가, 문서/테스트 규칙 정리 작업도 같이 진행했다. 작업 절차, 테스트 규칙, 기록 규칙을 글로만 중복 설명하지 않고 실제 운영 기준에 맞게 맞추는 흐름이었다.

## 배운 점 요약

- SPA 인증은 로그인 성공만으로 닫히지 않는다. `/api/me` 부트스트랩, refresh retry, 보호/guest 라우트, 실패 시 auth state 정리까지 있어야 실제 사용 흐름이 안정된다.
- REST Docs 테스트가 기능 검증까지 모두 책임지게 두면 테스트는 통과해도 구조는 건강하지 않을 수 있다. 문서화 책임과 기능 검증 책임을 분리해야 디버깅 비용이 줄어든다.
- 프로젝트 중간 점검은 "무엇이 구현됐는가"보다 "무엇이 사용자/면접관 입장에서 위험하게 보이는가"를 기준으로 해야 우선순위가 분명해진다.
- 테스트, 문서, 하네스 규칙은 기능 구현이 어느 정도 올라온 뒤에야 필요해지는 부록이 아니라, 회귀와 문서 드리프트를 막는 핵심 안전장치다.

## 문제 / 막힌 점

- `/api/me`는 아직 `role`, `mainEvent`를 주지 않아 admin 판별과 상세 마이페이지 UX가 제한적이었다.
- `CommunityDetailPage`의 owner 판별은 nickname 기준 임시 UX 수준이라 권한 모델로는 약했다.
- 랭킹/PB 정합성 문제, access token 저장 방식, 하드코딩된 비밀값 문제는 여전히 남아 있어 Day 15 이후 우선순위가 다시 조정될 수밖에 없었다.
