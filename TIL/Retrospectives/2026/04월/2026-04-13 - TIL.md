# 2026-04-13 - TIL

## 오늘 다룬 질문

- SPA 인증 실연동에서 로그인 성공 이후 어떤 복구/재시도 구조까지 있어야 안정적이라고 볼 수 있는가
- REST Docs 테스트, 서비스 단위 테스트, QueryDSL 통합 테스트를 어떤 기준으로 나눠야 하는가
- 프로젝트 중간 점검에서 기능 목록보다 먼저 봐야 하는 리스크 축은 무엇인가

## 개념 정리

- SPA 인증은 `login -> token 저장`으로 끝나지 않는다. 새로고침 시 `/api/me`로 현재 사용자 상태를 복구하고, access token 만료 시 refresh를 단일 in-flight로 재시도하며, 실패 시 auth state를 즉시 정리하는 흐름이 있어야 사용자 경험이 무너지지 않는다.
- 문서화 테스트는 문서화 책임만 남겨야 한다. 요청/응답 계약, validation, security wiring, 대표 happy path는 REST Docs에 두고, 서비스 성공/실패 분기와 QueryDSL 쿼리 엣지 케이스는 별도 테스트로 내려야 실패 원인을 더 빠르게 좁힐 수 있다.
- 중간 점검은 구현률 체크가 아니라 감점 요소 탐지다. 도메인 정합성(`PLUS_TWO`, 랭킹 의미), API/UI 계약 어긋남, secret 관리, 토큰 저장 방식, 예외 응답 일관성처럼 면접이나 코드 리뷰에서 바로 질문받을 부분을 먼저 정리해야 한다.
- 테스트 규칙과 하네스 규칙은 "나중에 문서화"가 아니라 반복되는 실수와 리뷰 코멘트를 구조적으로 줄이는 장치다. 네이밍 규칙, 역할 분리, 검증 기준을 명문화해 두면 다음 변경의 기준선이 생긴다.

## 실전 적용

- 인증 실연동에서는 `/api/me`를 추가하고 `AuthContext`, `apiClient`, 보호/guest 라우트를 연결해 새로고침과 만료 토큰 상황에서도 흐름이 끊기지 않도록 했다.
- 백엔드 테스트 리뷰에서는 REST Docs 테스트가 repository side-effect까지 떠안는 구조를 문제로 보고, 문서화 테스트와 기능 검증 테스트의 책임 분리를 기준으로 다시 설계했다.
- 프로젝트 중간 점검에서는 `PLUS_TWO` 미반영, solve leaderboard와 user leaderboard 혼선, 하드코딩된 secret, `localStorage` 토큰 노출면, 예외 응답 평탄화 같은 항목을 핵심 리스크로 올렸다.
- 테스트 네이밍 리팩토링과 AGENTS 테스트 규칙 추가는 단순 문체 정리가 아니라, 이후 테스트 품질을 리뷰할 공통 기준을 만들기 위한 작업이었다.
