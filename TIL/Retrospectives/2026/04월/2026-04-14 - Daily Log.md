## 오늘 한 일

- `/docs` 전체 동기화와 `docs-folder-sync` 작업을 통해 공개 문서와 내부 일정 문서가 현재 코드 상태를 더 잘 반영하도록 다시 맞췄다. 문서가 과거 상태를 계속 설명하지 않게 만드는 쪽에 집중했다.
- Day 15 일정과 `dev-log`에는 `JaCoCo` 도입 항목을 반영했다. 공개 일정에는 한 줄 수준으로, 내부 일정에는 `jacocoTestReport`, 커버리지 기준선, CI 연계 검토까지 넣어 외부/내부 문서의 역할 차이를 분리했다.
- 현재 개발 단계 리뷰도 같이 진행했다. `cd backend && ./gradlew test`, `./gradlew build -x test`, `cd frontend && npm run lint`, `npm run test -- --run`, `npm run build`까지 모두 통과한 상태에서 프로젝트를 다시 읽었다.
- 그 결과 인증과 타이머는 실연동으로 닫혔지만, 랭킹/커뮤니티/홈/마이페이지/피드백은 아직 mock 또는 부분 연동 상태라는 점, 그리고 `PLUS_TWO`/랭킹 정합성이 가장 먼저 닫혀야 한다는 점을 다시 정리했다.
- `AGENTS` 문서 확장 반영과 프로젝트 하네스 백엔드 품질 규칙 병합도 진행했다. 테스트 작성, 검증 범위, 보안/민감정보, dead code, 문서 동기화 같은 항목을 로컬 규칙으로 더 선명하게 만들었다.
- 전역 하네스 쪽에서는 코드 품질 guardrail과 커밋 단위 작업 규칙도 같이 정리했다. 커밋 메시지, 커밋 전 승인, 구현-검증-커밋 단위 분리 같은 실무형 기준이 점점 구체해졌다.
- docs facts-first 정리와 현재 개발 단계 리뷰 링크 추가도 같이 했다. 감상보다 확인한 사실과 남은 리스크를 먼저 남기는 쪽으로 문서 톤을 바꾸는 작업이었다.
- 포트폴리오 문서 초안 작성, Day 15 체크리스트 마감 계획, 랭킹 PB 기준 전환 계획, 일요일 일정 압축, 문서 커밋 분리 계획도 함께 다뤘다. 구현과 문서, 일정이 따로 놀지 않게 묶는 날이었다.

## 배운 점 요약

- 문서 동기화는 기능 설명만 맞추는 작업이 아니라, "지금 무엇이 닫혔고 무엇이 아직 안 닫혔는가"를 정확하게 쓰는 작업이다.
- 빌드, 테스트, 린트가 모두 녹색이어도 핵심 도메인 규칙이 틀리면 프로젝트 단계 판단은 달라질 수 있다.
- JaCoCo 같은 품질 도구는 실제 설정 전에 먼저 문서와 일정에 올라가야 팀의 기대치가 생긴다.
- 품질 규칙과 커밋 규칙을 하네스에 올리면 반복되는 설명을 줄이고, 후속 작업의 최소 기준선을 만들 수 있다.

## 문제 / 막힌 점

- `JaCoCo`는 문서와 일정에는 반영됐지만, 실제 coverage gate를 둘지 리포트 생성 수준에서 멈출지는 아직 미확정이었다.
- 랭킹/PB 정합성 문제는 여전히 다음 구현 우선순위의 최상단에 남아 있었다.
- 문서와 코드가 맞아도 이후에 다시 드리프트가 날 수 있어, facts-first 정렬을 계속 유지해야 했다.
