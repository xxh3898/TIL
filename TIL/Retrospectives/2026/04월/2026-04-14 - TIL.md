# 2026-04-14 - TIL

## 오늘 다룬 질문

- 구현이 빠르게 변하는 프로젝트에서 문서를 어떻게 facts-first로 동기화할 것인가
- 품질 규칙과 커밋 규칙을 언제 하네스 수준으로 끌어올리는 것이 좋은가
- 테스트/빌드가 모두 통과해도 프로젝트 단계를 "아직 중후반"으로 판단하는 근거는 무엇인가

## 개념 정리

- 문서 동기화는 서술 보강보다 상태 진단에 가깝다. 공개 문서에는 외부에 보여 줄 사실만 남기고, 내부 일정과 로그에는 현재 상태, 리스크, 체크리스트를 더 자세히 남기는 식으로 역할을 나눠야 한다.
- facts-first 문서 정리는 "이미 구현된 것처럼 보이게 꾸미지 않는 것"이 핵심이다. 구현이 덜 닫힌 화면이나 규칙은 그대로 미완 상태로 적어야 이후 우선순위 판단이 흔들리지 않는다.
- 품질 규칙은 리뷰에서 같은 지적이 반복될 때 하네스에 올리는 편이 좋다. 테스트 작성 의무, dead code 정리, 민감정보 처리, 검증 범위, 커밋 승인 규칙처럼 계속 되풀이되는 기준은 문서보다 실행 규칙으로 만드는 편이 유지비가 낮다.
- 프로젝트 단계 판단은 통과한 검증 수보다 "남아 있는 핵심 도메인 리스크"가 더 좌우한다. 랭킹/PB 정합성처럼 제품 핵심이 아직 안 닫힌 항목이 있으면, 인증과 타이머가 실연동이어도 전체 프로젝트는 아직 마감 단계가 아니다.

## 실전 적용

- Day 15 일정과 `dev-log`에 `JaCoCo`를 반영하면서 공개 일정에는 한 줄, 내부 일정에는 체크리스트를 두는 식으로 외부/내부 문서의 밀도를 다르게 가져갔다.
- 현재 개발 단계 리뷰에서는 백엔드/프런트 검증을 모두 통과시킨 뒤에도, 랭킹 정합성과 MVP 화면 실연동 미완성 때문에 "운영 직전"이 아니라 "중후반 개발 단계"로 판단했다.
- 프로젝트 하네스 백엔드 품질 규칙과 전역 코드 품질/커밋 규칙을 강화해, 이후 변경이 테스트, 보안, 문서 동기화 기준을 자연스럽게 따라가도록 정리했다.
- docs facts-first 정리와 리뷰 링크 추가는 "감상형 개발일지"보다 "다음 사람이 바로 근거를 찾을 수 있는 로그"를 남기기 위한 선택이었다.

## 핵심 요약

- 문서와 하네스는 구현 뒤에 따라오는 부수물이 아니라, 현재 상태를 과장 없이 고정하고 다음 우선순위를 흔들리지 않게 만드는 장치다. 그래서 docs sync, quality rule, stage review는 한 묶음으로 보는 편이 맞다.
