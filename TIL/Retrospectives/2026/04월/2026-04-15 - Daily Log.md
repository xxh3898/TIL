# 2026-04-15 - Daily Log

## 오늘 목표 (Top 3)

- [x] `cubing-hub`의 CI와 Day 16 핵심 기능을 실제 연동 기준으로 진전시키기
- [x] PKM 볼트의 루트 구조, 템플릿, 대시보드, 공개 범위를 실제 사용 가능한 수준으로 잡기
- [x] Obsidian 설정과 기존 공개 TIL 자산을 새 PKM 흐름에 맞게 정렬하기

## 오늘 한 일

- `cubing-hub` CI는 단일 `ci.yml`에서 `backend-ci.yml`, `frontend-ci.yml` 두 개로 분리했다. 변경 범위와 상관없이 모든 job이 매번 도는 구조를 피하고, `concurrency`, `timeout-minutes`, 실패 분석용 artifact 경로를 추가해 운영 가드도 함께 넣었다.
- frontend workflow에는 `.ci-reports/` 기반 failure reports를 넣고, lint 결과와 vitest junit report, build log를 남기도록 정리했다. backend는 기존 검증 흐름을 유지하면서 concurrency와 timeout을 추가하는 방향으로 정비했다.
- 이 변경은 로컬 검증으로도 확인했다. `cd frontend && npm run lint`, `npm test -- --run ...`, `npm run build`, `cd backend && ./gradlew test jacocoTestReport`, `./gradlew build -x test`를 통과시켰고, 생성되는 report 파일도 실제로 확인했다.
- Day 16 핵심 기능 구현도 같이 진행했다. 커뮤니티 목록 API 검색/페이지네이션 계약 확장, `CommunityPage` 실연동, `/api/me.role` 확장, 커뮤니티 상세/작성/삭제 실연동, 댓글 API와 `CommunityDetailPage` 연동, public `GET /api/home`와 `HomePage` 실연동, 로그인 전용 `POST /api/feedbacks`와 `FeedbackPage` 실연동까지 묶어서 닫았다.
- Day 16 기능은 백엔드 build, 프런트 테스트 44개, lint, production build, generated REST Docs 반영, 브라우저 수동 흐름까지 검증했다. 커뮤니티 목록/상세/작성/삭제, 댓글 작성/삭제, guest/auth 홈 분기, 피드백 제출 UX, 일반 사용자/관리자 권한 분기를 직접 확인한 점이 컸다.
- PKM 쪽에서는 루트 폴더 구조를 실제 사용 기준으로 잡았다. `Inbox`, `Projects`, `Areas`, `Resources`, `TIL`, `Templates`, `Archive`, `Assets` 구조를 만들고 `TIL/Project Log` 대신 `Projects/<project>/Log`로 역할을 나누는 기준을 정했다.
- `Templates/Daily Log.md`, `Templates/TIL.md`, `Templates/DevQ&A.md`, `Templates/Dashboard.md`를 만들고, `Dashboard.md`, `Areas/PKM.md`까지 함께 구성했다. 일일 기록과 재사용 지식을 분리하는 최소 schema가 여기서 잡혔다.
- 공개 Git 준비도 진행했다. 공개 allowlist 기반 `.gitignore`를 만들고, 공개 범위를 `README.md`, `TIL/**`, `Templates/**` 정도로 줄인 뒤 nested repo 없이 PKM 루트를 공개 Git 기준으로 삼는 방향을 정리했다.
- 기존 공개 TIL 자산도 일부 옮겼다. 예전 `TIL` 저장소의 `README.md`, `TIL/DevQ&A/**`, `TIL/Retrospectives/**`를 현재 PKM 구조로 가져와 공개 자산과 새 구조가 한곳에서 이어지게 만들었다.
- Obsidian 설정 정렬에서는 `Templater`, `QuickAdd` 경로를 `Templates`로 고정하고, `New Daily Pair`, `New DevQ&A`, `Open Dashboard` choice를 추가했다. 특히 `New Daily Pair`는 같은 날짜의 `Daily Log`와 `TIL`이 이미 있으면 다시 만들지 않고 여는 방향으로 구성했다.

## 배운 점 요약

- CI는 "한 번에 다 돌리는 것"보다 "변경 범위와 실패 분석 경로가 분명한 것"이 실전 운영에 더 유리하다.
- 핵심 기능 구현은 API 추가보다 검증 범위를 어디까지 묶어 닫느냐가 더 중요하다. Day 16은 자동 검증과 수동 흐름을 같이 돌려서 신뢰도가 높았다.
- PKM 구조는 콘텐츠 이관 전에 schema와 분류 기준을 먼저 잡아야 이후 마이그레이션이 덜 흔들린다.
- 일일 노트 자동화는 생성보다 멱등성이 중요하다. 이미 있는 파일을 덮어쓰지 않고 여는 동작이 실제 사용성에 훨씬 직접적이다.

## 문제 / 막힌 점

- Git 이력 승계와 실제 `git init`/이전 레포 이전은 아직 하지 않아 공개 저장소 전환은 준비 단계에 머물렀다.
- Obsidian 설정은 JSON 기준으로 정렬했지만, 실제 앱에서 `New Daily Pair`나 `Open Dashboard`가 의도대로 동작하는지 수동 검증은 남아 있었다.
- 기존 `README.md`는 아직 예전 구조 설명을 많이 담고 있어, 공개 저장소 소개 문서는 후속 재작성 필요가 있었다.
