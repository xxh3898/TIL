# 2026-04-15 - TIL

## 오늘 다룬 질문

- 언제 단일 CI workflow를 분리하고 실패 분석 artifact까지 같이 설계해야 하는가
- 대량 마이그레이션 전에 PKM 구조와 노트 schema를 먼저 고정해야 하는 이유는 무엇인가
- 일일 노트 자동화에서 "새로 만들기"보다 "기존 파일 열기"가 왜 더 중요한가

## 개념 정리

- CI workflow 분리는 단순 취향 문제가 아니라 검증 비용과 피드백 속도의 문제다. backend/frontend 변경이 섞이지 않는 구조라면 workflow를 나누고, 각자 `concurrency`, `timeout`, failure artifact를 두는 편이 운영 효율이 좋다.
- failure artifact는 실패 원인을 로컬에서 재현하기 전에 빠르게 읽게 해 준다. lint 결과 텍스트, JUnit XML, build log처럼 실패 시 바로 읽을 수 있는 산출물을 남겨 두면 CI가 단순 통과/실패 신호를 넘어서 진단 도구가 된다.
- PKM이나 지식베이스 마이그레이션은 콘텐츠보다 schema가 먼저다. 폴더 구조, note_type, 템플릿 섹션, 공개/비공개 범위를 먼저 고정하지 않으면 이관할수록 구조가 더 어지러워진다.
- 일일 노트 자동화는 멱등성이 핵심이다. 같은 날짜의 `Daily Log`와 `TIL`을 다시 만드는 자동화보다, 이미 있으면 열고 없으면 만드는 자동화가 실제 사용 흐름과 충돌이 적다.

## 실전 적용

- `cubing-hub`에서는 단일 `ci.yml`을 `backend-ci.yml`, `frontend-ci.yml`로 분리하고, frontend 쪽에 `.ci-reports/` 기반 failure report 수집 구조를 넣었다.
- Day 16 핵심 기능 구현은 API 확장과 프런트 실연동을 함께 묶고, backend build, frontend test/lint/build, 브라우저 수동 흐름까지 연결해 "기능 + 검증" 단위로 닫았다.
- PKM 볼트에서는 루트 폴더 구조와 `Daily Log`/`TIL`/`DevQ&A`/`Dashboard` 템플릿을 먼저 만든 뒤, 그 다음에 기존 `TIL` 공개 콘텐츠와 Obsidian 설정을 정렬했다.
- `QuickAdd`의 `New Daily Pair`는 `fileExistsMode: "Nothing"` 기준으로 구성해, 같은 날짜 노트가 이미 있으면 재생성하지 않고 여는 쪽을 우선하게 만들었다.
