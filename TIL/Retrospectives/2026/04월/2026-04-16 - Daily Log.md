## 배운 점 요약

- 대규모 PKM 이관은 "구조 먼저, 콘텐츠 다음, 링크/README/대시보드 정리 그다음, source cleanup 마지막" 순서로 가야 덜 흔들린다.
- 모든 자료를 한 번에 옮기기보다 `원문 보존형`, `선택 이관형`, `요약 보존형`으로 나누는 편이 source of truth를 덜 오염시킨다.
- 공개 문서와 내부 문서를 함께 옮길 때는 링크 정책, `.internal.md` 같은 가시적 표식, broken link 전수 검사가 같이 가야 한다.
- AI 산출물은 생성 루트와 보관 루트를 분리해야 관리가 된다. 원본 `/Users/chiho/AI/**`, 수집함 `AI/Inbox/**`, 최종 보관소 `AI/<project>/**`는 같은 폴더가 아니어야 한다.
- 하네스 규칙은 전역과 로컬의 책임을 나눠야 유지비가 낮다. 전역은 약한 공통 규칙, 로컬은 강한 저장소 규칙으로 두는 편이 덜 충돌한다.
- `research-*`, `plan-*`가 나온 작업은 `review-*`까지 문서로 닫아야 나중에 검증 결과와 리스크가 사라지지 않는다.
- 대시보드와 템플릿은 실제 데이터와 쿼리를 기준으로 한 번 더 검증해야 한다. 파일만 있다고 운영 흐름이 자동으로 맞아지지는 않는다.
- 자동화 프롬프트도 실행 순서만 적는 것으로는 부족하고, `가장 가까운 AGENTS.md`, 기존 노트/템플릿, 중복 확인 순서를 직접 명시해야 출력 품질이 덜 흔들린다.
- 항상 루트에서 세션을 시작하는 워크스페이스라면, 강한 분기 규칙은 읽힘이 보장되는 루트 하네스에 두고 하위 하네스는 특화 규칙만 맡기는 편이 안전하다.
- 시간 기반 자동화만 고집하기보다 `daily` / `weekly` / `all` 같은 manual runner 묶음을 두면 중복 실행과 실패 지점을 더 쉽게 통제할 수 있다.
