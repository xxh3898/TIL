# 2026-04-16 - TIL

## 오늘 다룬 질문

- 대규모 PKM/문서 이관을 할 때 source of truth, 이관 순서, 후속 cleanup을 어떤 단계로 나눠야 하는가
- 모든 문서를 무조건 옮기지 않고 `원문 보존`, `선택 이관`, `요약 보존`으로 나누는 기준은 무엇인가
- AI 산출물의 원본, 수집함, 최종 보관소를 자동화와 함께 운영하려면 어떤 구조가 필요한가
- 로컬 하네스와 전역 하네스는 어떤 책임 분리로 설계해야 장기 유지보수가 쉬운가
- 자동화 프롬프트는 어떤 읽기 순서와 중복 체크 규칙을 직접 품고 있어야 결과 품질이 안정되는가
- 항상 루트에서 시작하는 PKM에서는 어떤 규칙을 루트 하네스에 두고, 어떤 규칙만 하위 하네스나 템플릿으로 내려야 하는가

## 개념 정리

- 대규모 지식베이스 이관은 "구조 설계 -> 콘텐츠 이관 -> 링크/템플릿/대시보드 정비 -> source of truth 재정렬 -> 레거시 cleanup" 순서가 안전하다. 처음부터 모든 걸 한 번에 옮기면 어떤 경로가 현재 기준인지 다시 설명하기 어려워진다.
- 이관 전략은 하나가 아니다. 원문을 그대로 다시 볼 가치가 높으면 `원문 보존형`, 현재 규칙에 흡수됐지만 계보는 남겨야 하면 `요약 보존형`, 현재 구조에 실제로 편입해야 하면 `선택 이관형`으로 나누는 편이 과적재를 막는다.
- AI 산출물 관리는 생성 위치와 보관 위치를 섞지 않을 때 안정적이다. 원본 source of truth는 `/Users/chiho/AI/<project>/...`에 두고, 자동화는 먼저 `PKM/AI/Inbox/<project>/...`에 수집하며, 사람이 장기 가치가 있다고 판단한 문서만 `PKM/AI/<project>/...` 최종 보관소로 승격하는 3단계가 적절하다.
- 공개 문서와 내부 문서가 섞인 프로젝트 이관에서는 링크 정책이 핵심이다. 공개 범위는 상대 Markdown 링크, 내부 전용 연결은 `Internal Links`, 내부 문서는 `.internal.md` 같은 명시적 표식으로 분리하는 편이 안전하다.
- 하네스는 전역이 약한 공통 규칙을 맡고, 로컬이 강한 저장소 규칙을 맡을 때 역할이 덜 겹친다. 여기에 `research -> plan -> review` 문서 닫힘 조건까지 있으면 나중에 왜 그렇게 바뀌었는지 다시 추적하기 쉬워진다.
- 자동화 프롬프트는 "무엇을 할까"만 적는 것이 아니라 "무엇을 먼저 읽고 무엇과 중복을 비교할까"까지 포함해야 안정적이다. 특히 `가장 가까운 AGENTS.md -> 기존 노트/템플릿 -> 같은 날짜 회고/기존 목적지 노트 중복 확인 -> append/merge만 허용` 순서를 프롬프트에 써 두면 결과가 덜 흔들린다.
- 세션이 항상 특정 루트에서 시작되는 워크스페이스라면, 강한 라우팅 규칙은 읽힘이 보장되는 루트 하네스에 두는 편이 안전하다. 중간 하위 경로의 `AGENTS.md`는 읽히지 않을 수도 있으므로, prefix 같은 분기 규칙보다 프로젝트 전용 세부 규칙에 집중시키는 편이 낫다.
- 시간 기반 자동화와 manual runner는 대체 관계만이 아니다. 먼저 프롬프트와 역할 경계를 정리한 뒤, `daily` / `weekly` / `all`처럼 순서가 보이는 수동 실행 묶음을 두면 중복 실행, 실패 지점, inbox 소음을 더 쉽게 관찰할 수 있다.
- AI 산출물 원본 루트도 얇은 로컬 하네스를 두면 도움이 된다. `/Users/chiho/AI`처럼 코드 저장소가 아닌 경로에서는 source of truth, `AI/Inbox` 승격, `원본 경로:` 메모 같은 운영 규칙만 짧게 고정하는 편이 적절하다.

## 실전 적용

- 기존 Obsidian 볼트에서는 `JobPrep`, `독서 기록장`, `Project_Log`를 한 기준으로 다 옮기지 않고, `Areas/Career`, `Resources/Books`, `Archive/Projects/**/Log`로 목적지 자체를 다르게 잡았다. 문서 종류마다 최종 서식과 이용 맥락이 다르기 때문이었다.
- `CodingTests`는 `Study-Notes`만 떼어오지 않고 nested repo 전체를 가져온 뒤, 그 위에서 frontmatter, 문제 링크, wikilink, 대시보드, 정보 블록 중복 정리를 덧붙였다. 구조 보존과 PKM 정규화의 순서를 나눠 처리한 셈이다.
- `Cubing Hub` 문서 이관에서는 `docs/ai/**`를 `*.internal.md`로 바꿔 공개 문서와 내부 문서를 분리했고, 깨진 코드 링크는 그대로 놔두지 않고 plain text 코드 참조로 바꿨다. 링크가 깨진 채 남지 않게 하는 것이 더 중요했다.
- AI 산출물 정리에서는 `/Users/chiho` 전체에서 후보 732개를 모아 실제 AI 산출물 716개로 정제하고, 활성 421개에 처분 상태를 붙였다. 그 뒤 `/Users/chiho/AI/pkm/**` 절대경로 정렬, `AI/Inbox` 자동화 분리, 레거시 루트 삭제 순으로 진행해 source of truth를 하나로 줄였다.
- 하네스 정렬에서는 PKM 로컬 하네스는 강제 규칙만 남기고, 전역 하네스는 PKM 경로 조건부 규칙과 `review-*` 문서 의무 규칙을 두는 식으로 책임을 나눴다. `Daily Log`와 `TIL`의 역할 차이도 이 흐름 안에서 다시 명시했다.
- 11개 자동화 새 프롬프트 초안에서는 각 자동화가 먼저 가장 가까운 `AGENTS.md`를 읽고, 기존 목적지 노트와 recent inbox item을 중복 체크하도록 공통 규칙을 넣었다. 이 덕분에 `감사`, `실제 수집`, `승격 검토`, `Daily Log/TIL 반영` 역할을 더 선명하게 나눌 수 있었다.
- `codex exec`로 git repo 바깥에서도 비대화형 실행이 가능한 점을 확인해, `daily runner` / `weekly runner` / `all runner` 형태의 manual runner가 기술적으로 가능하다는 근거를 확보했다.
- PKM 하네스는 `질문:` / `면접:` / `서류:` prefix 기반 메인 라우터를 루트 `AGENTS.md`에 두고, 면접 대비용 템플릿과 허브 노트를 별도로 두는 구조로 재정리했다. 읽힘 보장이 약한 중간 `AGENTS.md`를 없애고, `Projects/Portfolio/AGENTS.md`는 실제 HTML 편집 규칙 전용으로 축소했다.
- `/Users/chiho/AI/AGENTS.md`에는 `project/date/task` 구조, legacy copy-only 이관 시 `원본 경로:` 메모, PKM `AI/Inbox`와 최종 보관소의 역할 분리를 문서화해 원본 루트 운영 규칙을 얇게 고정했다.

## 핵심 요약

- 이관 작업의 핵심은 "얼마나 많이 옮겼는가"가 아니라 "무엇이 원본이고, 무엇이 수집본이며, 무엇이 장기 보관본인가"를 끝까지 설명할 수 있게 만드는 것이다. 그 기준이 닫혀야 cleanup과 자동화도 안전해진다.
- 같은 원리는 자동화와 하네스에도 적용된다. 어떤 규칙이 어디서 먼저 읽히고, 어떤 시점에 중복을 제외하며, 어느 경로를 source of truth로 둘지 닫혀 있어야 운영이 흔들리지 않는다.
