# 2026-04-17 - TIL

## 오늘 다룬 질문

- optional auth endpoint와 보호 endpoint는 같은 인증 실패라도 왜 같은 응답 전략으로 묶으면 안 되는가
- `토큰은 남아 있지만 사용자 엔티티를 찾지 못하는 경우`를 어디에서 `401`로 끊고, 어디에서 guest fallback으로 흡수해야 하는가
- 백엔드 예외 경계와 프런트 상태 처리 정책은 어떤 식으로 같이 맞물려야 하는가

## 개념 정리

- 먼저 endpoint 계약을 나눠야 한다. 보호 endpoint는 인증된 사용자 컨텍스트가 전제이므로 사용자 조회 실패를 인증 실패로 보고 fail-closed 해야 한다. 이 경우 `401 Unauthorized`가 맞다.
- 반대로 optional auth endpoint는 guest도 정상 사용자다. 이 계약에서는 인증 컨텍스트 복원이 실패하더라도 guest 응답으로 내려갈 수 있으면 그렇게 하는 편이 맞다. 여기서 중요한 것은 "인증이 없으면 실패"가 아니라 "인증이 있으면 추가 정보를 붙인다"는 모델이다.
- 같은 예외 원인을 만났다고 해서 모든 endpoint가 같은 응답을 내야 하는 것은 아니다. 핵심은 예외 종류보다 endpoint가 외부에 약속한 동작이다.
- optional auth fallback은 아무 예외나 삼키는 뜻이 아니다. 사용자 조회 실패처럼 인증 컨텍스트 복원 단계의 예외만 제한적으로 흡수하고, 실제 도메인 오류나 시스템 오류는 그대로 드러내야 한다.
- 프런트도 같은 기준을 따라야 한다. 보호 화면은 `401`을 세션 정리나 재로그인 흐름으로 연결하고, optional auth 화면은 guest 상태 렌더링을 우선하되 진짜 오류일 때만 에러 UI를 띄워야 한다.
- 테스트 전략도 달라진다. 보호 endpoint는 `401`과 권한/validation 실패를 분리해 고정해야 하고, optional auth endpoint는 guest fallback이 언제 동작하고 언제 예외를 다시 던지는지 분기 테스트로 닫아야 한다.

## 실전 적용

- 이번 작업에서는 `PostService`, `CommentService`, `RecordService`에서 사용자 조회 실패를 `401 Unauthorized`와 `사용자를 찾을 수 없습니다.`로 통일했다. 커뮤니티, 댓글, 기록 관리가 모두 보호 endpoint이기 때문이다.
- `/api/home`는 optional auth endpoint라서 같은 뿌리의 사용자 조회 실패라도 guest home fallback으로 정리했다. 홈은 비로그인 사용자에게도 정상 응답을 주는 계약이기 때문이다.
- 프런트에서는 `MyPage`에 재시도 버튼을 추가해 보호 화면의 실패 복구 경로를 드러냈고, `FeedbackPage`는 제출 중 폼 잠금과 입력 변경 시 메시지 초기화로 상태 일관성을 맞췄다. 백엔드의 예외 경계와 프런트의 사용자 경험을 같은 방향으로 조정한 셈이다.
- 문서/검증 측면에서도 분리를 유지했다. Day 17 로그에는 보호 endpoint 안정화와 optional auth fallback을 분리해서 기록했고, 관련 테스트도 홈 쪽과 보호 API 쪽을 별도로 다시 실행했다.

## 핵심 요약

- 인증 예외를 어떻게 처리할지는 예외 타입보다 endpoint 계약이 먼저 결정한다. 보호 endpoint는 fail-closed, optional auth endpoint는 제한된 범위의 graceful degradation이 맞다.
- 그래서 같은 `사용자 없음` 상황도 커뮤니티/댓글/마이페이지에서는 `401`, 홈에서는 guest fallback으로 갈릴 수 있다. 이 차이를 명시적으로 설계하고 테스트해야 프런트 상태 처리도 덜 흔들린다.
