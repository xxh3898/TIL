## 오늘 한 일

- 전역 하네스 쪽에서는 세션 종료 규칙을 다시 정리했다. 종료 신호를 `세션종료:`로 명확히 좁히고, 세션을 닫기 전에는 현재 작업 폴더의 `research-*`, `plan-*`, `review-*` 문서를 최신 상태로 맞추도록 정리했다. 경미한 작업이라도 세션 종료 시점에는 `review-*`를 남겨야 한다는 흐름까지 같이 닫았다.
- PKM 링크 정리에서는 공개 범위 상대 Markdown 링크에서 `&`를 `%26`으로 남긴 경로와 수동 편집 중 섞인 비정상 `%` 오타를 정리했다. `Projects/Cubing Hub` 문서와 회고 노트 일부에서 `Authentication & Authorization Design`, `Deployment & Infrastructure Design`, `DevQ&A` 쪽 경로가 규칙과 실제 파일명에 맞게 다시 정렬됐다.
- 그다음에는 볼트 전체 링크를 한 번 감사했다. 내부 Markdown 링크, 위키 링크, 외부 URL를 분리해 보면서 실제 결함과 placeholder, 템플릿 생성 위치 기준 상대 링크, 실행 환경 의존 외부 URL을 따로 떼어 봤다. 덕분에 바로 고쳐야 할 것과 지금은 건드리면 안 되는 것을 같은 목록에서 분리할 수 있었다.
- 마지막으로 감사 결과 중 실제 수정 범위를 좁혀 `[[Dashboard]]` 모호성 17건만 explicit Markdown 상대 링크로 바꿨다. `Dashboard.md`와 `Templates/Dashboard.md`가 같은 basename을 공유하고 있었는데, 루트 `Dashboard.md`를 가리키는 상대 링크로 바꿔 정적 도구와 Obsidian 밖 해석에서도 덜 흔들리게 만들었다.

## 배운 점 요약

- PKM 공개 문서 링크는 공백만 `%20`으로 인코딩하고, 파일명 안의 `&`는 raw 문자 그대로 두는 편이 규칙과 실제 파일명 해석이 가장 일관된다.
- basename이 겹치는 노트가 생기면 짧은 위키 링크보다 explicit Markdown 상대 링크가 더 안전하다. 특히 템플릿과 실노트가 같은 이름을 공유하면 정적 감사 도구에서 곧바로 모호성이 생긴다.
- 볼트 전체 링크 감사는 결과를 바로 일괄 수정하기보다 `실제 결함 / placeholder / 템플릿 상대 링크 / 환경 의존 외부 URL`로 먼저 나눠야 의미가 있다. 분류 없이 한 번에 손대면 false positive까지 같이 고치게 된다.
- 세션 종료 같은 운영 규칙도 자연어 표현보다 충돌이 적은 명시 토큰과 문서 마감 순서를 같이 정의해야 이후 자동화와 회고가 덜 흔들린다.

## 문제 / 막힌 점

- 외부 URL 검증은 TLS 체인, anti-bot, timeout 영향을 많이 받아 `404`와 `환경/시점 이슈`를 완전히 분리하기 어려웠다.
- 템플릿 문서의 상대 링크는 템플릿 파일 위치에서 보면 깨져 보일 수 있어, 생성 위치 기준 placeholder인지 실제 결함인지 따로 판단해야 했다.
- `Dashboard.md` basename 충돌 자체는 남아 있어 앞으로 새 `[[Dashboard]]`가 다시 들어오면 같은 모호성이 재발할 수 있다.
