# 2026-04-20 - TIL

## 오늘 다룬 질문

- 저장소 계층 refactor 전후 성능 비교를 하려면 baseline 계약을 어디까지 먼저 고정해야 하는가
- 성능 benchmark를 일반 기능 CI와 같은 workflow에 섞지 말아야 하는 기준은 무엇인가
- 일정 재베이스라인과 성능 기준선 고정을 같은 날 처리할 때 무엇을 먼저 닫아야 하는가

## 개념 정리

- 성능 리팩토링 전후 비교에서 가장 먼저 고정해야 하는 것은 "무엇을 비교하는가"다. 조회 API, query parameter, page/size, 실패 허용 기준이 흔들리면 Day 19와 Day 20 숫자는 같은 차트를 써도 같은 실험이 아니다.
- 그다음은 데이터셋이다. 같은 seed를 다시 적재할 수 없는 benchmark는 재실행 때마다 후보 수와 정렬 비용이 달라져 비교 신뢰성이 떨어진다. 그래서 cleanup 스크립트보다 `DB reset -> same seed` 절차를 먼저 닫는 편이 안전할 때가 있다.
- 결과 형식도 실험 전에 고정해야 한다. `summary.json`, 사람이 읽는 Markdown/HTML 비교 문서, 같은 Grafana 대시보드 레이아웃처럼 "어떤 파일과 화면으로 남길지"가 먼저 닫혀 있어야 refactor 뒤에도 같은 형식으로 이어 붙일 수 있다.
- benchmark workflow는 일반 기능 CI와 목적이 다르다. 기능 CI는 빠른 회귀 검출이 핵심이지만 benchmark는 동일 조건 재현과 artifact 보관이 더 중요하다. 그래서 `workflow_dispatch` 기반 별도 workflow로 분리하는 편이 기준선 관리에 유리하다.
- 일정 재베이스라인도 같은 원리다. 아직 실행하지 않은 결과를 문서에 적는 대신, 공개 일정에는 실제 날짜와 마일스톤만, 내부 일정에는 압축 이유와 실행 순서를 나누어 남겨야 다음 날 결과 반영이 덜 흔들린다.

## 실전 적용

- 이번 Cubing Hub 작업에서는 기준 요청을 `GET /api/rankings?eventType=WCA_333&page=1&size=25`로 고정해 저장소 계층 차이만 비교하도록 범위를 좁혔다.
- baseline 데이터도 `users`, `records`, `user_pbs` 각 `300,000`건으로 고정하고, Day 20에도 같은 SQL을 다시 적재할 수 있게 단일 seed 자산으로 묶었다.
- 결과 보관 형식은 `summary.json`, 비교용 Markdown/HTML, 같은 Grafana 캡처 흐름으로 정해 Day 19 기준선과 Day 20 결과를 같은 레이아웃에서 비교할 수 있게 했다.
- 일반 `backend/frontend CI`에 benchmark를 섞지 않고 `workflow_dispatch` 기반 보조 workflow로 분리해, 기능 회귀 검증과 성능 기준선 보관의 목적을 갈라 두었다.
- 일정 문서도 같은 흐름으로 재정렬했다. 공개 일정은 `2026-04-20 baseline`, `2026-04-21 refactor + verify + deploy` 수준으로 유지하고, 내부 일정과 허브 문서에는 더 세밀한 실행 순서를 남겼다.

## 핵심 요약

- 성능 refactor 전후 비교는 refactor 자체보다 먼저 실험 계약을 고정해야 한다. 시나리오, seed, 결과 형식, 시각화 레이아웃이 먼저 닫혀야 다음 날 숫자가 설명 가능해진다.
- benchmark는 기능 CI의 한 단계가 아니라 별도의 재현 자산으로 다루는 편이 낫다. 같은 조건 재실행과 artifact 보관이 핵심이기 때문이다.
