## 오늘 한 일

- `GET /api/rankings` 기본 조회를 Redis ZSET read model 경로로 전환했다. `nickname` 검색과 ready marker 부재 시에는 기존 MySQL `user_pbs` 경로를 유지하는 hybrid 방식으로 닫았고, PB 변경 시 Redis 증분 동기화와 startup 재구축 경로까지 묶어 검증했다. 같은 seed와 같은 `k6` 기준으로 `avg 7245.23 ms -> 21.10 ms`, `p95 12429.58 ms -> 36.94 ms`까지 내려가는 비교 결과를 남겼다.
- `backend/Dockerfile`, `docker-compose.prod.yml`, `nginx.conf`, `application-prod.yaml`을 추가해 EC2 배포 artifact를 정리했다. EC2 Docker/Compose, 인증서, RDS 연결, 실제 DB 이름 `cubinghub`까지 확인했지만, `xxh3898/cubing-hub-backend:latest`에 `linux/amd64` manifest가 없어 `docker compose pull` 단계에서 실제 기동은 막혔다.

## 배운 점 요약

- read model 전환은 "무조건 Redis로 바꾸기"보다 source of truth를 MySQL에 남기고, hot path만 Redis로 우회하며 fallback 조건을 명시하는 편이 롤백과 검증이 단순하다.
- steady-state latency가 좋아도 rollout 비용은 따로 본다. 이번에는 local `300,000` PB 기준 startup 재구축이 약 9분 걸려 운영 트리거 정책을 별도로 다뤄야 했다.
- Apple Silicon 로컬에서 만든 이미지를 EC2 `amd64`에 배포할 때는 `docker compose config`가 통과해도 충분하지 않다. registry manifest에 `linux/amd64` 또는 multi-arch 지원이 없으면 실제 pull 단계에서 바로 막힌다.

## 문제 / 막힌 점

- startup 재구축이 약 9분 걸려 운영 환경에서 `rebuild-on-startup=true`를 그대로 쓰기 어렵다.
- `nickname` 검색은 여전히 MySQL fallback이라 검색 트래픽이 커지면 별도 병목이 남을 수 있다.
- EC2 배포는 Docker 이미지 아키텍처 mismatch 때문에 컨테이너 기동과 `actuator/health`, 로그인 흐름 smoke test까지 이어지지 못했다.
