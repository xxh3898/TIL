# 2026-04-21 - TIL

## 오늘 다룬 질문

- MySQL source of truth를 유지한 채 Redis read model을 어떻게 안전하게 붙여야 하는가
- Apple Silicon 로컬 빌드 이미지를 `amd64` EC2에 배포할 때 release 계약으로 무엇을 먼저 보장해야 하는가

## 개념 정리

- read model 전환의 핵심은 저장 위치를 바꾸는 것이 아니라 책임을 나누는 것이다. 영속 기준과 정합성 판단은 계속 MySQL `records` / `user_pbs`가 맡고, Redis는 조회 hot path를 빠르게 읽기 위한 파생 저장소로 두는 편이 안전하다.
- 이때 전환 경계도 좁게 잡아야 한다. 이번처럼 기본 랭킹 조회만 Redis로 옮기고 `nickname` 검색이나 ready marker 부재 같은 조건에서는 MySQL fallback을 유지하면 계약 변경 없이 성능 개선과 롤백 경로를 동시에 확보할 수 있다.
- read model은 steady-state latency만 보고 끝내면 안 된다. 증분 동기화 훅, 전체 backfill, startup 재구축 시간, ready marker 같은 운영 메커니즘까지 닫혀 있어야 실제 서비스에서 쓸 수 있다.
- 배포도 같은 원리다. 로컬에서 `docker build`가 성공했다는 사실과 EC2에서 `docker compose pull`이 성공한다는 사실은 다르다. Apple Silicon 환경은 기본적으로 `arm64` 이미지를 만들 수 있으므로, `amd64` EC2에 배포할 때는 registry manifest에 `linux/amd64` 또는 multi-arch 지원이 release 계약으로 포함돼 있어야 한다.
- 그래서 production deploy 체크리스트에는 compose syntax, env 파일, 인증서뿐 아니라 "대상 서버 아키텍처가 registry manifest에 포함돼 있는가"를 먼저 넣는 편이 낫다.

## 실전 적용

- Cubing Hub 랭킹 API에서는 `GET /api/rankings` 기본 조회를 Redis ZSET read model로 옮기고, 검색과 비정상 상태는 MySQL fallback으로 남겨 hybrid rollout을 택했다.
- 같은 seed와 같은 `k6` 시나리오 기준으로 `avg 7245.23 ms -> 21.10 ms`, `p95 12429.58 ms -> 36.94 ms`를 확인했지만, startup 전체 재구축은 local `300,000` PB 기준 약 9분이 걸려 운영 트리거를 따로 결정해야 했다.
- AWS 배포 준비에서는 EC2 Docker/Compose, 인증서, RDS 연결, 실제 DB 이름까지 확인했어도 `xxh3898/cubing-hub-backend:latest` manifest에 `linux/amd64`가 없어서 실제 `pull`과 `up`이 막혔다. 설정 파일보다 이미지 배포 계약이 먼저라는 점이 드러났다.

## 핵심 요약

- read model rollout은 "빠르게 읽힌다"보다 "어디가 source of truth인지, 언제 fallback 하는지, 재구축 비용이 얼마인지"를 먼저 닫아야 안전하다.
- EC2 배포에서 이미지 아키텍처 manifest는 선택 정보가 아니라 실행 계약이다. `amd64`가 빠진 이미지는 compose 파일이 맞아도 배포가 실패한다.
