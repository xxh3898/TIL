## 오늘 한 일

- 운영 CORS를 `www`와 apex 둘 다 허용하도록 바로잡고, 1차 배포 결과와 first deploy/redeploy 체크리스트, backend/frontend CD workflow를 문서와 저장소에 반영했다. 동시에 SSH 기반 backend 배포를 SSM Run Command로 바꾸기 위한 AWS 전제조건도 조사해 `22` 포트 의존을 줄이는 방향을 정리했다.
- 홈 `todayScramble`을 KST 일자 고정으로 바꾸고, grouped 페이지네이션과 상단 문구를 다듬고, 타이머 VisualCube 미리보기와 Ao5/Ao12/기록 추세 그래프를 추가했다. `index.html` `lang="ko"`와 `notranslate` 힌트도 넣었고, `csTimer` 기능 스캔으로 다음 제품 후보를 `inspection`, `manual entry`, `session/record stats` 쪽으로 좁혔다.
- `refresh_token` 변조가 login 자체를 막는 운영 이슈를 복구하기 위해 `POST /api/session/clear-refresh-cookie` public endpoint와 frontend best-effort 정리 흐름을 넣었다. 여기에 bootstrap refresh single-flight, local `/actuator/prometheus` permit flag, 검색 debounce까지 묶어 반복 요청과 불필요 재조회를 줄였다.
- 마이페이지 summary를 전체 `records` fetch 대신 aggregate query로 바꾸고, `GET /api/users/me/profile`를 `10 users x 10,000 records` 조건에서 baseline/current로 재측정했다. 결과는 `avg 451.04ms -> 77.86ms`, `p95 790.68ms -> 137.39ms`까지 내려갔고, 같은 benchmark를 Grafana 대시보드와 포트폴리오 문서까지 연결했다.
- 피드백 저장과 Discord 운영 알림 상태를 분리해 추적하는 구조를 넣었다. `notification_status`, 시도 횟수, 재시도 API, 프런트 상태 배너까지 추가해서 저장 성공과 알림 실패를 분리해 다룰 수 있게 했다.

## 배운 점 요약

- 운영 이슈는 경계 조건 하나가 핵심일 때가 많다. 이번엔 `https://cubing-hub.com` CORS origin 누락과 bad `refresh_token` path 결합이 증상을 만들었고, 둘 다 가장 좁은 경계 수정으로 풀렸다.
- 성능 개선을 말하려면 구현보다 비교 계약이 먼저다. baseline worktree, 동일 seed, 동일 `k6` 시나리오, 같은 Grafana dashboard/run label을 맞춰 두니 aggregate query 전환 효과를 재사용 가능한 자산으로 남길 수 있었다.
- 외부 채널 연동은 "비즈니스 성공"과 "알림 성공"을 분리해 다루는 편이 안전하다. 피드백 저장 성공과 Discord 실패를 따로 보여주니 재시도 UX와 운영 판단이 단순해졌다.

## 문제 / 막힌 점

- backend SSM 전환은 아직 조사 단계다. EC2 SSM agent/IAM role, GitHub Actions `ssm:SendCommand` 권한, health check 기준을 실제로 확인해야 한다.
- 자동 배포 workflow 파일은 추가했지만 GitHub Variables, 서버 `.env`, 인증서 갱신 자동화는 아직 비어 있다.
- auth recovery는 상당수 bad cookie 시나리오를 흡수하지만, 실제 운영 브라우저에서 malformed cookie 새로고침/재로그인 흐름 수동 검증이 남아 있다.
- Discord webhook과 production schema 반영, VisualCube 외부 응답 품질, `Recharts` chunk 경고는 운영 검증이 더 필요하다.
