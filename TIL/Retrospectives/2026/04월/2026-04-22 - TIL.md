# 2026-04-22 - TIL

## 오늘 다룬 질문

- malformed `refresh_token`가 login 자체를 막을 때 복구 endpoint는 왜 `cookie path` 밖에 있어야 하는가
- aggregate query 최적화 전후 비교를 왜 같은 seed, 같은 `k6`, 같은 Grafana 대시보드에서 고정해야 하는가

## 개념 정리

- 브라우저가 특정 path에 cookie를 계속 붙이는 구조에서는, 문제 cookie를 지우는 endpoint가 같은 path 아래 있으면 복구 요청 자체가 같은 bad cookie에 다시 막힐 수 있다. 그래서 clear endpoint는 원인 path 밖의 public 경로로 분리하는 편이 안전하다.
- client recovery도 전부 뭉뚱그리면 안 된다. guest 첫 진입처럼 정상적인 `missing cookie 400`까지 복구 루틴에 넣으면 불필요한 호출이 늘고 원인 분리가 흐려진다. malformed, `401`, network-error처럼 실제 오염 가능성이 큰 경우만 best-effort cleanup 대상으로 좁히는 편이 낫다.
- 성능 최적화 전후 비교는 "현재 코드가 빠르다"가 아니라 "baseline/current가 같은 계약 위에서 측정됐다"는 점이 핵심이다. 이번처럼 Git 이력에서 baseline worktree를 분리하고, 같은 schema reset, 같은 seed, 같은 `GET /api/users/me/profile` 시나리오를 쓰면 aggregate query 전환 효과를 신뢰할 수 있다.
- 수치 보관도 반복 비교 가능해야 한다. `summary.json`, `comparison.md/html`, Prometheus remote write, Grafana dashboard, `run` label(`mypage-10k-baseline`, `mypage-10k-current`)을 같이 고정하면 다음 리팩터링에서도 같은 패널과 자산으로 다시 비교할 수 있다.
- 이 방식은 성능 설명 자산으로도 유리하다. 구현 설명, 재현 절차, 시계열 캡처가 한 세트로 남아야 포트폴리오나 면접에서 "왜 이 수치를 믿어도 되는지"를 방어하기 쉽다.

## 실전 적용

- Cubing Hub auth 복구에서는 `POST /api/session/clear-refresh-cookie`를 `/api/auth` 밖에 두고, frontend는 bootstrap/login 실패 중 malformed, `401`, network-error 계열일 때만 best-effort로 cookie clear를 시도하게 했다.
- MyPage summary 최적화는 전체 `records` fetch 대신 aggregate query로 바꾸고, `10 users x 10,000 records` 기준 `avg 451.04ms -> 77.86ms`, `p95 790.68ms -> 137.39ms`를 확인했다.
- 같은 benchmark를 Grafana에 연결해 baseline/current를 같은 dashboard에서 비교하고, 포트폴리오 문서에도 같은 수치를 그대로 재사용했다.

## 핵심 요약

- bad auth cookie 복구는 "지우는 API가 어디에 있느냐"가 핵심이다. 원인 cookie path 밖의 public endpoint와 선택적 client recovery가 함께 있어야 login 막힘을 줄일 수 있다.
- 성능 수치는 같은 seed, 같은 시나리오, 같은 관측 자산 위에서만 설명 가치가 생긴다. baseline worktree와 Grafana run label을 고정해 두면 전후 비교가 재현 가능한 문서 자산이 된다.
