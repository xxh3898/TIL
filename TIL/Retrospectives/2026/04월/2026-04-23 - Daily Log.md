## 오늘 한 일

- `POST /api/auth/email-verification/request`, `POST /api/auth/email-verification/confirm`, signup `verified marker` 소비 흐름을 추가해 회원가입을 이메일 인증 2단계로 바꿨다. 같은 묶음에서 비밀번호 재설정 request/confirm, 마이페이지 `계정 관리` 모달의 프로필 수정/비밀번호 변경, 커뮤니티 게시글 수정 진입, 학습 화면 상단 회전기호 VisualCube 가이드도 함께 닫았다.
- `react-toastify` 전역 toast, 모바일 stacked card/table 레이아웃, 타이머 `touch`/`pen`/`Space` 공통 상태 전이, `running` 중 `blur` 유지 로직을 넣어 프런트 상호작용을 다듬었다. 랭킹 닉네임 검색은 검색 결과 안에서 순위를 다시 매기지 않고 전체 랭킹 기준 실제 순위를 유지하게 했고, 게스트 타이머 저장/자동 초기화/다음 스크램블 흐름도 같이 정리했다.
- 게시글 이미지 첨부와 `post_attachments`, S3 기반 업로드 추상화, 계정당 1회 조회수 집계, 공개 Q&A와 관리자 피드백/메모 화면까지 커뮤니티/운영 기능을 크게 확장했다. 이후 production에서 드러난 `multipart` 헤더 오염, `POST_IMAGES_*` compose 미전달, 게시글 상세 조회수 side effect, 업로드 body size 불일치를 순서대로 수정해 실제 경계 계약을 맞췄다.
- 배포 전 점검을 다시 돌리며 JDBC TLS 강제, Flyway baseline 도입, frontend production env fail-fast, Redis rebuild workflow 분리, EC2 deploy prune/pull 범위 축소, query index/MyPage 중복 호출/route lazy loading/Discord payload privacy/mainEvent validation 같은 배포 차단 및 후속 품질 항목을 커밋 단위로 정리했다. 관련 공식 문서와 세션 산출물도 현재 구현 상태에 맞게 동기화했다.
- PKM 쪽에서는 `AI/Inbox` copy-only 수집과 최종 보관소 승격을 실제로 반영했다. `cubing-hub` 2026-04-22 산출물 3개를 `AI/Inbox`로 복사하고, `global-harness`/`pkm` 장기 자산 9개를 최종 보관소로 승격해 원본 `/Users/chiho/AI/**`, 수집함, 최종 보관소 흐름을 실제 운영 상태에 맞췄다.

## 배운 점 요약

- 현재 auth 구조에서는 `PENDING` 사용자 상태를 도입하는 것보다 `Redis code + cooldown + verified marker`로 signup 전 인증을 강제하는 편이 영향 범위를 더 작게 유지한다.
- 모바일 호환성은 CSS만의 문제가 아니라 입력 상태 머신 문제이기도 하다. 키보드, `touch`, `pen`, `blur`가 같은 전이를 공유해야 회귀가 줄어든다.
- 이미지 업로드는 저장소 구현보다 경계 계약 정렬이 더 중요했다. `multipart` 헤더, runtime env 전달, reverse proxy/body size, 조회수 같은 부수효과 격리를 함께 맞춰야 production에서 안정적으로 동작한다.
- 배포 readiness review는 차단 이슈와 후속 품질 개선을 분리하고 커밋 단위로 닫아야 rollback과 운영 확인이 쉬워진다.

## 문제 / 막힌 점

- SMTP 실제 자격 증명을 연결한 회원가입/비밀번호 재설정 실메일 발송과 운영 end-to-end 수동 검증은 아직 남아 있다.
- 게시글 이미지 업로드는 `multipart` 헤더, compose env, Nginx/backend body size처럼 계층 간 계약이 조금만 어긋나도 바로 깨졌다. 현재 수정으로 맞췄지만 운영 브라우저 기준 smoke는 더 필요하다.
- EC2 루트 디스크 압박은 prune와 pull 범위 축소로 완화했지만, 서버 용량 자체 리스크는 남아 있다.
- `README.md` 포함 문서 전면 closeout은 계획 문서까지는 정리됐지만 최종 마감 반영 여부를 다시 확인해야 한다.
