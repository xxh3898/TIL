# 2026-04-23 - TIL

## 오늘 다룬 질문

- 게시글 이미지 업로드를 운영에 올릴 때 왜 스토리지 로직보다 먼저 경계 계약을 맞춰야 하는가
- `multipart` 업로드는 왜 프런트 헤더, runtime env, reverse proxy/body size, 부수효과 transaction을 한 세트로 봐야 하는가

## 개념 정리

- 파일 업로드 기능은 대개 "S3에 잘 저장되느냐"보다 "경계 계층이 같은 계약을 보고 있느냐"에서 더 자주 깨진다. 브라우저 요청 serializer, 프런트 공통 interceptor, 서버 multipart parser, 런타임 env 주입, reverse proxy body limit, 부수효과 write path가 서로 어긋나면 비즈니스 로직이 맞아도 production에서 바로 실패한다.
- 특히 `multipart/form-data`는 공통 `application/json` 헤더와 충돌하기 쉽다. `FormData` 요청일 때는 브라우저가 boundary를 포함한 `Content-Type`를 직접 만들게 두는 편이 안전하다. 클라이언트가 JSON 헤더를 억지로 유지하면 서버는 파일 part나 request DTO를 비정상적으로 읽을 수 있다.
- 운영 env 전달도 업로드 계약의 일부다. 애플리케이션이 `POST_IMAGES_*` 같은 설정을 읽도록 되어 있어도, 실제 compose나 배포 스크립트가 컨테이너에 넘기지 않으면 runtime에서는 기능이 비활성화된 것과 같다.
- 업로드 제한은 한 계층만 키워서는 해결되지 않는다. reverse proxy의 `client_max_body_size`, 애플리케이션의 `max-file-size`, `max-request-size`가 같은 전제를 공유해야 한다. 파일 하나는 `10MB`인데 요청 전체는 더 클 수 있으므로 single file과 total request를 분리해 생각해야 한다.
- 읽기 경로에 붙는 부수효과는 best-effort로 격리하는 편이 안전하다. 게시글 상세 응답의 핵심은 "본문을 읽을 수 있는가"이고, 조회수 기록은 그보다 덜 핵심이다. 조회수 write 실패가 상세 조회 `500`으로 이어지면 사용자는 읽기 자체를 잃는다.

## 실전 적용

- 이번 Cubing Hub 작업에서는 먼저 `post_attachments` 도메인과 S3 업로드 추상화를 붙여 게시글 이미지 첨부를 구현했다. 그런데 production에서는 구현 코드보다 경계 계약이 먼저 문제를 일으켰다.
- 첫 번째 문제는 프런트 전역 `application/json` 헤더가 `FormData` 요청에도 남아 있던 것이다. 이를 제거하자 backend가 multipart `request` part와 파일 part를 정상적으로 읽기 시작했다.
- 두 번째 문제는 `application-prod.yaml`이 기대하는 `POST_IMAGES_*` 값이 `docker-compose.prod.yml`에서 컨테이너로 전달되지 않던 점이었다. compose env wiring을 맞추자 EC2 `.env` 설정이 실제 runtime으로 이어졌다.
- 세 번째 문제는 업로드 크기 제한 불일치였다. `nginx client_max_body_size`, backend `spring.servlet.multipart.max-file-size`, `max-request-size`를 같은 기준으로 정렬해 프록시와 앱이 서로 다른 이유로 요청을 끊지 않게 했다.
- 마지막으로 게시글 상세 응답에 붙은 조회수 증가 로직은 best-effort로 내려 분리했다. 덕분에 view 기록 실패가 있어도 게시글 본문과 첨부 이미지는 계속 읽을 수 있게 됐다.

## 핵심 요약

- 이미지 업로드는 "파일 저장 기능"이 아니라 end-to-end 경계 계약이다. `multipart` 헤더, env wiring, reverse proxy/body size, 부수효과 격리를 같은 체크리스트로 다뤄야 production에서 덜 깨진다.
- 읽기 핵심 경로에 붙는 부수효과는 실패를 격리해야 한다. 조회수 같은 부가 write가 깨져도 상세 조회 자체는 살아 있어야 한다.
