## 오늘 한 일

- Cubing Hub 문서를 전반적으로 다시 맞췄다. `README.md`, `docs/Feature Backlog.md`, `docs/dev-log.md`, `docs/portfolio.internal.md`에 운영 URL, 구현 완료 backlog, Day 로그 범위, S3/SMTP/운영 adapter 설명 자산을 반영했다. 이후 개발일지를 커밋 시각 기준으로 다시 대조하고 `Day 01`부터 `Day 27`까지 흐름을 재배치해 `Day 13A`, `Day 18A` 같은 보조 번호가 남지 않게 정리했다.
- 공개 문서와 내부 로그의 한국어 문체를 한 번 더 다듬었다. `fallback`, `source of truth`, `current-state`, `manual-only`처럼 문장 안에 어색하게 섞인 영어식 표현은 한국어 설명으로 바꾸고, API 경로와 명령어, 환경 변수 같은 기술 식별자는 원문으로 유지했다.
- 프런트 사용자 문구와 백엔드 validation/예외 문구를 사용자 기준의 자연스러운 표현으로 정리했다. 같은 흐름에서 피드백 화면의 Discord 상태/재시도 UI를 제거하고, 일반 사용자용 `POST /api/feedbacks` 응답을 `data.id` 중심으로 줄였으며, public `notification-retry` endpoint도 제거했다. 관련 프런트 테스트, 백엔드 테스트, REST Docs, 공식 문서까지 함께 맞췄다.
- backend와 frontend의 테스트 사각지대를 크게 줄였다. SMTP/S3 production bean wiring, unavailable adapter, Discord notifier, ranking Redis, repository 구현, 관리자/Q&A 화면, auth/timer/api client 유틸까지 테스트를 보강했고, backend JaCoCo instruction/branch 100%, frontend Vitest coverage 100%를 확인했다.
- 최종 품질 검증 상태를 `README.md`, `Project Overview`, `Project Schedule`, `Deployment & Infrastructure Design`, `dev-log`, `Day 29`, 내부 일정/포트폴리오 문서에 반영했다. CI가 강제하는 범위와 로컬 최종 coverage 검증 범위를 구분해, 문서가 실제 보장보다 더 크게 말하지 않도록 조정했다.
- README 촬영용 로컬 데모 데이터를 별도 SQL로 만들고 로컬 DB/Redis에 적용했다. 운영 데이터 대신 로컬 데모 계정, 랭킹, 커뮤니티, Q&A, 관리자 메모 데이터를 넣고 주요 API 응답을 확인했다. 한글 깨짐은 `utf8mb4` 재적용으로 바로잡았다.
- PKM에서는 Cubing Hub 프로젝트를 `Projects/Cubing Hub`에서 `Archive/Projects/Cubing Hub`로 옮겼다. 최신 `Docs/**`와 `README.md`는 보존하고, 레거시 `Log/**`, 중복 `Troubleshooting/**`는 제거했으며, 볼트 전체 Cubing Hub 링크를 archive 경로로 맞췄다. 추가로 장기 참조 가치가 있는 배포 readiness와 업로드 body limit 산출물 2건을 `AI/Inbox`에 copy-only로 수집했다.

## 배운 점 요약

- 프로젝트 마감 문서는 기능 목록보다 "무엇을 자동 검증했고, 무엇은 수동 smoke로 확인했으며, 무엇은 아직 자동화하지 않았는가"를 분리해야 신뢰할 수 있다.
- public API는 프런트가 쓰지 않는 운영 내부 상태까지 노출하면 이후 구현을 바꾸기 어려워진다. 일반 사용자 응답과 관리자/운영 추적 응답은 처음부터 다르게 설계하는 편이 안전하다.
- coverage 100%는 운영 안정성 100%가 아니다. SMTP/S3처럼 외부 서비스와 자격 증명에 의존하는 경로는 adapter/config 테스트와 실제 smoke를 구분해야 한다.
- PKM archive 정리는 파일 이동만으로 끝나지 않는다. 허브 dataview 경로, 오래된 중복 복사본, 회고/Troubleshooting 링크까지 같이 정리해야 검색 기준이 흔들리지 않는다.

## 문제 / 막힌 점

- 실제 SMTP/S3/browser smoke는 사용자 수동 확인 기준으로 통과했지만, Codex가 직접 live 환경을 실행한 것은 아니었다.
- frontend coverage 100%를 위해 추가한 helper export와 `v8 ignore next`는 현재 UI 흐름 전제에 의존하므로, 화면 구조가 바뀌면 다시 점검해야 한다.
- README 촬영용 데모 seed는 로컬 DB/Redis 상태에 의존한다. `ddl-auto=create`로 백엔드를 재시작하면 다시 적용해야 할 수 있다.
- `mockCommunityPosts` legacy fixture 정리는 후속 후보로 남았다.
