## 오늘 한 일

- `Cubing Hub `README.md`에 대표 화면 7장과 기술 스택 badge/버전 표기를 정리했다. `JUnit 5/MockMvc` 검증 문장까지 보강해 공개 README 첫인상과 사실 관계를 함께 맞췄다.
- `PKM의 Cubing Hub를 `Archive/Projects/Cubing Hub` 기준으로 재정렬하고, 최신 `README.md`, `Docs/**`, README 이미지 7개를 archive 보관본에 다시 동기화했다.
- `포트폴리오 HTML/README, 커리어 허브, GitHub 프로필 README를 한 번에 정리했다. Cubing Hub 설명을 최신 1인 풀스택 기준으로 교체하고, 프로젝트 순서를 `CalmDesk → Cubing Hub → MediFlow`로 맞추고, 기술 스택 taxonomy를 `Core Backend`, `AI / External`, `Observability`, `Testing / Quality` 같은 공개 문맥에 맞게 재분류했다.
- iPhone 12 Safari 계열 모바일 레이아웃 깨짐을 CSS source가 아니라 빌드 출력 문제로 좁혔다. `frontend/vite.config.js`에 `build.cssTarget`을 추가해 `@media (max-width: 720px)`가 구형 브라우저에서 무시될 수 있는 `@media (width<=720px)`로 축약되지 않게 했고, `npm run build`, `npm run lint`, `npm run test -- --run` 407개 테스트까지 통과시켰다.

## 배운 점 요약

- 공개 README와 포트폴리오는 "전부 나열"보다 "전역 핵심 스택"과 "프로젝트별 운영/AI/외부 연동 경험"을 분리할 때 과장 없이도 더 넓은 경험을 전달할 수 있다.
- 모바일 회귀는 source CSS만 보면 놓치기 쉽다. Vite/Lightning CSS가 target에 따라 media query 문법을 재작성하므로, 배포된 CSS 산출물 자체를 확인해야 원인을 정확히 잡을 수 있다.
- PKM archive 동기화는 본문만 맞추면 끝나지 않는다. README가 참조하는 이미지 자산과 `Docs`/`docs` 대소문자 차이까지 함께 봐야 보관본 링크가 안정적이다.
