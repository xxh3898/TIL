# 2026-04-25 - TIL

## 오늘 다룬 질문

- Vite/Lightning CSS는 왜 source에 없는 `@media (width<=720px)` 같은 문법을 빌드 결과에 만들 수 있는가
- 모바일 반응형 회귀를 디버깅할 때 왜 source CSS보다 빌드 산출물의 media query 문법을 먼저 확인해야 하는가

## 개념 정리

- 프런트엔드 빌드는 source 파일을 그대로 배포하지 않는다. 번들러와 CSS optimizer는 지원 브라우저 기준에 맞춰 문법을 재작성하거나 축약한다. 그래서 source에는 `@media (max-width: 720px)`가 있어도 빌드 결과에서는 `@media (width<=720px)` 같은 Media Queries Level 4 range syntax로 바뀔 수 있다.
- 문제는 이 축약 문법이 "더 최신"이라고 해서 항상 안전한 것은 아니라는 점이다. 구형 iOS Safari, 오래된 Android Chrome, 일부 Samsung Internet 계열은 range syntax 지원이 불완전할 수 있어서, source CSS가 맞아도 배포본에서는 반응형 전환이 통째로 무시될 수 있다.
- JavaScript target과 CSS target은 별개로 봐야 한다. JS 트랜스파일이 충분하더라도 CSS optimizer가 더 공격적으로 최신 문법을 내보내면 브라우저 호환성 회귀가 남을 수 있다.
- 그래서 모바일 회귀를 볼 때는 JSX나 컴포넌트 CSS를 바로 뜯어고치기 전에, 실제 배포된 CSS 파일에서 media query가 어떤 문법으로 나갔는지 먼저 확인하는 편이 더 빠르고 정확하다.

## 실전 적용

- Cubing Hub 커뮤니티 목록은 source CSS 기준으로 이미 `@media (max-width: 720px)`에서 카드형 레이아웃으로 전환되도록 작성돼 있었다.
- 실제 문제는 소스가 아니라 빌드 산출물이었다. 배포 CSS가 `@media (width<=720px)` 형태로 축약되면서 친구의 iPhone 12 Safari 계열 환경에서 media query가 적용되지 않는 가능성이 드러났다.
- 해결은 레이아웃 재작성보다 빌드 출력 보수화가 더 작았다. `frontend/vite.config.js`에 `build.cssTarget`을 추가해 `safari16.3`, `ios16.3`, `chrome94` 수준에서 안전한 문법을 유지하게 했다.
- 이후 `npm run build`로 빌드 CSS를 다시 확인해 `@media (width<=...)`가 사라졌고, 기존 `@media (max-width: 720px)`와 카드형 규칙이 그대로 남아 있음을 확인했다. 같은 작업에서 `npm run lint`, `npm run test -- --run`도 통과했다.

## 핵심 요약

- 모바일 반응형 회귀는 source CSS보다 빌드된 CSS 문법이 원인일 수 있다. Vite/Lightning CSS가 지원 target에 따라 media query를 최신 문법으로 재작성하기 때문이다.
- 오래된 Safari/Android까지 고려해야 하면 컴포넌트 CSS를 다시 쓰기 전에 `build.cssTarget`과 실제 산출물의 media query 문법부터 확인하는 편이 가장 작은 수정으로 끝난다.
