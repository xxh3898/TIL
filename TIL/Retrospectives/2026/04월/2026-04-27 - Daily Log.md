# 2026-04-27 - Daily Log

- [x] Cubing Hub 시간 저장/표시 계약을 UTC instant + KST UI 기준으로 정리하고 화면 형식을 한국어 표현으로 맞추기
- [x] PKM AI 보관 흐름에서 final 승격, 데일리 로그 자동화, Inbox 중복 수집 여부를 정리하기

## 오늘 한 일

- `/learning`에 `초보자` 탭을 추가하고, 8단계 목록에서 단계 상세로 들어가는 구조를 만들었다. 1단계 안내형 정리, 2단계/5단계 그림 보정, 6단계 공식 재검증, `다음 단계` 이동 버튼 추가처럼 사용자 QA를 여러 차례 반영하면서 케이스 그림과 설명을 다듬고 테스트도 함께 보강했다.
- 백엔드 시간 필드와 응답 DTO를 `Instant` 기준으로 정리하고 `Clock.systemUTC()`와 `hibernate.jdbc.time_zone=UTC`를 적용했다. 프런트는 공통 `dateTime` 유틸로 KST 표시를 모으고 관련 API/DB/화면 문서까지 함께 동기화했다.
- 과거 `answered_at`류 값을 바로 보정하지 않고, 실제 저장 의미가 UTC instant인지 KST 벽시계인지 먼저 확인해야 한다는 기준과 확인 SQL 초안을 정리했다.
- 날짜+시간 표기를 `2026년 4월 27일 오후 7시 44분` 형식으로 통일했다. 정각은 `0분`을 생략하고, Q&A/관리자/커뮤니티/마이페이지 테스트와 문서도 같이 맞췄다.

## 배운 점 요약

- 시간 데이터는 DB/API에서 UTC instant 계약을 먼저 고정하고, 화면에서만 `Asia/Seoul`로 변환해야 서버 timezone과 브라우저 timezone에 흔들리지 않는다.
- 과거 시간값 backfill은 현재 코드 계약이 맞아도 별도 판단 문제다. 행 단위로 실제 저장 의미를 확인하지 않으면 `-9시간` 같은 일괄 보정이 정상 데이터까지 깨뜨릴 수 있다.
- 초보자용 시각화 작업은 “같은 공식이면 같은 그림”으로 끝나지 않는다. 케이스별 시작 상태와 완료 상태를 분리하고, 그림 `src` 중복까지 테스트로 확인해야 따라 하기형 학습 UI가 덜 헷갈린다.

## 문제 / 막힌 점

- `beginner-method-learning`은 사용자 QA가 여러 번 이어져 최종 화면 밀도와 그림 정확도는 높아졌지만, 단계별 시각화가 실제 모바일 화면에서 충분히 직관적인지는 추가 수동 확인이 더 필요하다.
- `time-contract`는 코드와 문서까지 정리했지만, 과거 운영 데이터가 UTC 의미인지 KST 의미인지 아직 확인하지 않아 `time-backfill` 판단이 남아 있다.
- `kst-display-format`은 테스트와 빌드는 통과했지만, 좁은 모바일 화면에서 긴 한국어 시간 문자열이 어떻게 보이는지는 별도 스모크가 필요하다.
