# 2026-04-27 - TIL

## 오늘 다룬 질문

- DB/API/UI 시간 표현을 언제 UTC instant로 고정하고, 언제 KST로 변환해야 하는가
- 기존 `LocalDateTime` 성격의 데이터는 왜 코드 계약 변경과 같은 흐름에서 일괄 backfill 하면 안 되는가

## 개념 정리

- 시간 처리에서 먼저 고정해야 하는 것은 보기 좋은 형식이 아니라 의미 계약이다. 저장 시각과 API 응답은 timezone에 덜 흔들리는 UTC instant로 맞추고, 사람에게 보이는 마지막 단계에서만 서비스 기준 timezone으로 변환하는 편이 안전하다.
- 이 계약을 코드에 심으려면 “현재 시각 생성”과 “DB/JPA 직렬화”도 같은 기준을 써야 한다. `Clock`을 주입해 UTC 기준 현재 시각을 만들고, Hibernate JDBC timezone도 UTC로 고정해야 저장과 직렬화가 서로 어긋나지 않는다.
- 화면 표기 형식은 그다음 문제다. `오후 7시 44분` 같은 한국어 표현은 이미 의미가 정리된 instant를 `Asia/Seoul`로 변환한 뒤에만 붙여야 한다. 의미 계약과 표시 형식을 한 번에 섞으면 테스트 범위와 디버깅 범위가 같이 커진다.
- 과거 데이터 보정은 코드 계약 변경과 별도 마이그레이션 문제다. 기존 행이 UTC 의미인지 KST 벽시계 의미인지 확인하지 않고 `-9시간` 같은 일괄 보정을 적용하면 정상 데이터까지 망가질 수 있다. 그래서 배포 시각, 샘플 행, 관련 컬럼 묶음을 먼저 확인해 보정 범위를 좁혀야 한다.

## 실전 적용

- Cubing Hub에서는 백엔드 감사 필드와 도메인 시간 필드를 `Instant`로 정리하고, `Clock.systemUTC()`와 `hibernate.jdbc.time_zone=UTC`를 적용해 저장/응답 계약을 먼저 맞췄다.
- 프런트는 시간 표시를 공통 `dateTime` 유틸로 모아 KST 변환과 한국어 오전/오후 포맷을 한곳에서 처리했다. 이렇게 나누니 Q&A, 관리자, 커뮤니티, 마이페이지 테스트 기대값도 공통 기준으로 갱신할 수 있었다.
- 같은 날 과거 데이터 보정은 바로 실행하지 않았다. `feedbacks.answered_at` 사례를 기준으로 “이 값이 실제 UTC 시각인지, 고치기 전 KST 입력인지”를 먼저 판별해야 한다고 정리했고, 실제 보정은 행 단위 또는 기간 단위 확인 뒤로 미뤘다.

## 핵심 요약

- 시간 처리에서는 저장/API 의미 계약을 UTC instant로 먼저 고정하고, 화면에서만 서비스 timezone으로 변환하는 편이 가장 안전하다.
- 기존 데이터 backfill은 별도 판단 문제다. legacy 행의 실제 의미를 증명하기 전에는 일괄 offset 보정을 하면 안 된다.
