# 2026-04-28 - TIL

## 오늘 다룬 질문

- 기능 회귀 없이 서비스 전반 프런트 리디자인을 진행할 때, 공통 shell과 페이지별 범위를 어떤 순서로 나눠야 하는가
- 시각 리디자인 도중 언제 API 계약을 넓히고, 언제 클라이언트 안에서만 해결해야 하는가

## 개념 정리

- 큰 프런트 리디자인은 페이지를 한꺼번에 건드리기보다 공통 shell, 디자인 token, 반응형 내비게이션 같은 바닥 구조를 먼저 고정한 뒤, 페이지군을 사용 빈도와 데이터 위험도로 나눠 순차 적용하는 편이 안전하다.
- 기본 원칙은 “보이는 것만 바꾸고 기능 계약은 그대로 둔다”다. 라우팅, 인증 흐름, 폼 label, 버튼 semantics, 데이터 저장 로직을 가능한 한 유지해야 회귀 범위를 줄일 수 있다.
- 다만 화면이 꼭 알아야 하지만 프런트에서 정확히 계산할 수 없는 값은 API를 좁게 확장하는 편이 낫다. 예를 들어 랭킹의 내 순위처럼 사용자별 데이터 맥락이 들어가는 값은 클라이언트 추정보다 서버 source of truth가 더 안전하다.
- 시각 변경이 넓을수록 검증도 두 층으로 나눠야 한다. 페이지별 핵심 테스트와 브라우저 DOM 확인으로 즉시 깨질 수 있는 부분을 막고, 마지막에는 전체 lint/test/build로 공통 shell 변경의 연쇄 영향을 다시 본다.

## 실전 적용

- Cubing Hub에서는 먼저 홈/타이머에서 공통 app shell, token, 모바일 하단 tab을 고정하고, 다음 단계에서 랭킹/커뮤니티/마이페이지, 그다음 인증, 학습, Q&A/피드백/관리자 화면으로 범위를 나눴다.
- 랭킹 화면은 시각 변경만으로는 해결되지 않아 `myRanking` 필드를 API에 추가했다. 현재 페이지에 보이는 랭킹 slice만으로는 로그인 사용자의 정확한 순위를 안정적으로 계산할 수 없었기 때문이다.
- 반대로 인증, 학습, 피드백, 관리자 화면은 API와 권한 경계를 바꾸지 않고 레이아웃과 문구, 상태 블록, CSS 구조만 정리하는 쪽을 택했다. 이 덕분에 기능 리스크를 크게 늘리지 않고 화면 톤을 맞출 수 있었다.
- 각 단계마다 관련 테스트와 `Screen Specification` 같은 문서를 같이 갱신해, 실제 코드 상태와 설명 문서가 어긋나지 않게 유지했다.

## 핵심 요약

- 서비스 전체 리디자인은 공통 shell과 token을 먼저 정리하고, 페이지군을 단계별로 밀되 기능 계약은 최대한 유지하는 방식이 가장 안전하다.
- 프런트가 신뢰할 수 없는 사용자별 계산은 억지로 클라이언트에서 만들지 말고, 필요한 만큼만 API를 확장하는 편이 낫다.
